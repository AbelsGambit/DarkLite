  1384→### WriterOptions design
  1385→
  1386→- **`dryRun`** (default `false`): when true, the writer walks the deps map,
  1387→  allocates IDs in memory, and produces an `ImportResult` showing what
  1388→  WOULD be written — but does NOT touch the filesystem or any pack files.
  1389→  Every `WrittenEntry.path` is set to the sentinel string `'<dryrun>'`.
  1390→  Useful for previewing an import before committing it.
  1391→
  1392→- **`namePrefix`** (default `'osrs_'`): prepended to every sanitized
  1393→  debugname. Combined with the sanitized OSRS name/ID to form the final
  1394→  debugname (e.g. `osrs_tormented_demon`, `osrs_model_45000`).
  1395→
  1396→- **`groupFile`** (default `'osrs_imports'`): subfolder under
  1397→  `content/models/` where binary assets (.ob2, .anim, .base) are written,
  1398→  AND the basename of the `.npc` / `.seq` config files. Multiple NPCs
  1399→  imported in one batch share the same group file (their `[debugname]`
  1400→  blocks are appended to the same `.npc` / `.seq` file).
  1401→
  1402→- **`overwrite`** (default `false`): when false (the safe default), the
  1403→  writer NEVER overwrites existing files or pack entries — collisions
  1404→  trigger a fresh debugname with a `_2`, `_3` suffix instead. When true,
  1405→  the writer REPLACES existing `[debugname]` blocks in config files and
  1406→  overwrites binary files with the same path. NOT recommended for normal
  1407→  use — intended for re-importing after a sanitization rule change.
  1408→
  1409→### Sanitization rules
  1410→
  1411→The `sanitizeName(rawName, packType)` helper applies the following
  1412→transforms in order:
  1413→
  1414→1. **Lowercase** the entire string.
  1415→2. **Replace every char NOT in `[a-z0-9]` with `_`** — spaces, hyphens,
  1416→   apostrophes, etc. all become underscores.
  1417→3. **Collapse runs of multiple `_`** into a single `_` (so `"a  b"` →
  1418→   `"a_b"`, not `"a__b"`).
  1419→4. **Trim leading/trailing `_`** (so `"_foo_"` → `"foo"`).
  1420→5. **Prefix** with `namePrefix` (default `osrs_`).
  1421→6. **Dedupe** against the target pack file AND a session-reserved set
  1422→   (in-memory, populated by prior `sanitizeName()` calls in the same
  1423→   writer lifetime). If the candidate name is already taken, append
  1424→   `_2`, `_3`, `_4`, ... until unique.
  1425→
  1426→Examples:
  1427→- `"Tormented demon"` → `"osrs_tormented_demon"`
  1428→- `"td_stand"` → `"osrs_td_stand"`
  1429→- `"model_45000"` → `"osrs_model_45000"`
  1430→- `"anim_2005"` (after `osrs_anim_2005` already exists in `anim.pack`)
  1431→  → `"osrs_anim_2005_2"`
  1432→
  1433→The session-reserved set is critical for batch imports where two OSRS
  1434→assets sanitize to the same candidate name in the same run — without
  1435→it, the second asset would silently overwrite the first's pack entry.
  1436→
  1437→### Idempotency approach
  1438→
  1439→The writer is idempotent at three layers:
  1440→
  1441→1. **Deps-map layer** (primary): every OSRS `DepNode` has a
  1442→   `transformedFrom` field (set by Task 8 per the schema). On each run,
  1443→   the writer's `bootstrapResolverFromDepsMap()` walks the input deps
  1444→   map and pre-populates `NameResolver` from any node with
  1445→   `transformedFrom !== null` (looking up the debugname from the
  1446→   corresponding pack file). Then `processNode()` checks the field
  1447→   directly: if set, the node is recorded as `skipped` with reason
  1448→   `already_imported (newId=N)` and NOT re-processed.
  1449→
  1450→2. **NameResolver layer** (cross-NPC): if two NPCs in the same batch
  1451→   reference the same OSRS asset (e.g. both use OSRS model 45000), the
  1452→   first NPC's `processNode()` call registers the asset in
  1453→   `NameResolver`. When the second NPC's `processNode()` runs for the
  1454→   same OSRS ID, it finds the existing entry in NameResolver, records
  1455→   it as `skipped` with reason `shared_with_prior_npc`, and updates the
  1456→   deps map's `transformedFrom` field to point at the existing new ID.
  1457→
  1458→3. **Config-file layer** (defensive): `appendConfigBlock()` scans the
  1459→   target `.npc` / `.seq` config file for an existing `[<debugname>]`
  1460→   header before appending. If found and `overwrite=false`, the append
  1461→   is skipped (with a warning). If found and `overwrite=true`, the
  1462→   existing block is stripped via `stripConfigBlock()` before the new
  1463→   block is appended. This protects against the case where a prior run
  1464→   wrote the config file but failed before persisting the updated
  1465→   deps.json (so the next run's deps-map layer doesn't know about the
  1466→   prior import).
  1467→
  1468→The CLI (`Import.ts`) persists the updated deps.json (with
  1469→`transformedFrom` filled in) to `content/deps/<npcName>.deps.json`
  1470→after every successful non-dry-run import. Subsequent runs pick this
  1471→file up via `--deps=<path>` (or auto-trace + apply the persisted
  1472→`transformedFrom` from a prior run via the `--out=` round-trip).
  1473→
  1474→### Pack-file ID allocation strategy
  1475→
  1476→**Decision: `max(existing IDs) + 1` for every pack type.**
  1477→
  1478→The schema supports reserved ranges per pack type (e.g. "models
  1479→50000-59999 reserved for OSRS imports") but no ranges are reserved
  1480→today. The `allocateId(packType)` helper loads the pack file, finds
  1481→the maximum existing ID, and returns `max + 1` (or `0` for an empty
  1482→pack, matching LostCity's convention where every existing pack starts
  1483→at ID 0).
  1484→
  1485→Rationale:
  1486→- Simplest strategy that preserves backwards-compat (existing IDs are
  1487→  never reused or overwritten).
  1488→- The LostCity content folder doesn't have a "free ID range" convention
  1489→  — IDs are assigned sequentially as new assets are added.
  1490→- If a future task adds reserved ranges, `allocateId()` is the single
  1491→  function to change (per-pack-type strategy can be added without
  1492→  touching call sites).
  1493→
  1494→**Anims are registered in BOTH `anim.pack` AND `animset.pack`** (with
  1495→the same ID and name). This matches the existing LostCity convention
  1496→where both packs share the same `id=name` mapping for every anim file:
  1497→- `anim.pack` (AnimPack) is consulted by `SeqConfig.parseSeqConfig()`
  1498→  when resolving `frameN=<animname>` refs in `.seq` config files.
  1499→- `animset.pack` (AnimSetPack) is consulted by
  1500→  `engine/tools/pack/graphics/pack.ts` when mapping `.anim` files to
  1501→  archive 2 file IDs during cache packing.
  1502→
  1503→Registering in only one would break either the seq config parser or
  1504→the cache packer. The writer registers in both atomically (within the
  1505→same `updatePackFile()` call sequence, with both packs sharing the
  1506→same dirty/save batch).
  1507→
  1508→### NPC config line rewriting
  1509→
  1510→`OsrsNpcType.toLegacy377NpcConfig()` emits raw integer asset IDs in
  1511→its output (e.g. `model1=45000`, `readyanim=9200`,
  1512→`param=osrs_runanim,9102`). The writer's `rewriteNpcConfigLine()`
  1513→post-processes each line to rewrite those IDs to their newly-assigned
  1514→debugnames via `NameResolver`:
  1515→
  1516→| Line shape                              | Rewrite                                          |
  1517→|-----------------------------------------|--------------------------------------------------|
  1518→| `modelN=<id>`                           | `modelN=<modelDebugname>`                        |
  1519→| `headN=<id>`                            | `headN=<modelDebugname>`                         |
  1520→| `readyanim=<id>`                        | `readyanim=<seqDebugname>`                       |
  1521→| `walkanim=<id>[,<id>,<id>,<id>]`        | per-element rewrite to seq debugnames            |
  1522→| `param=osrs_runanim,<id>`               | `param=osrs_runanim,<seqDebugname>`              |
  1523→| `param=osrs_runanim_{l,r,b},<id>`       | `param=osrs_runanim_{l,r,b},<seqDebugname>`      |
  1524→| `param=osrs_crawlanim,<id>`             | `param=osrs_crawlanim,<seqDebugname>`            |
  1525→| `param=osrs_crawlanim_{l,r,b,b2},<id>`  | `param=osrs_crawlanim_{l,r,b,b2},<seqDebugname>` |
  1526→| `param=osrs_walkanim_b,<id>`            | `param=osrs_walkanim_b,<seqDebugname>`           |
  1527→
  1528→Lines whose OSRS ID can't be resolved via `NameResolver` (e.g. the
  1529→tracer didn't walk that model) fall back to the `model_<id>` /
  1530→`seq_<id>` / `anim_<id>` literal form — the LostCity config parser
  1531→accepts both forms (it tries `getByName(value)` first, falling back to
  1532→`parseInt(value)` if the name lookup returns -1).
  1533→
  1534→The header line (`[<osrs_debugname>]`) is also replaced with
  1535→`[<sanitized_debugname>]` so the pack entry matches the config block
  1536→header. (The OSRS emitter uses the OSRS debugname or `npc_<id>` as
  1537→the header — we override it with our `osrs_<name>` sanitized form.)
  1538→
