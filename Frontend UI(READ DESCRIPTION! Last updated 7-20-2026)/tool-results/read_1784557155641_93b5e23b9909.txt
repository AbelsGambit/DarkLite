   400‚Üí5. **`AnimBase.instances[]` is reused.** The OSRS skeleton format is
   401‚Üí   byte-identical to 377 for pre-HD content (same `length u8`, `types
   402‚Üí   u8[]`, `labels u8[][]` layout). Rather than duplicate the loader,
   403‚Üí   `OsrsAnimFrame` references the existing `AnimBase.instances[]`
   404‚Üí   registry. OSRS-only pipelines that don't load 377 anims must still
   405‚Üí   call `AnimBase.unpack()` on OSRS base files to populate this
   406‚Üí   registry before invoking `OsrsAnimFrame.decode()`. (This is the
   407‚Üí   task 6 dep tracer's responsibility.)
   408‚Üí
   409‚Üí### `toLegacy377()` transform ‚Äî limitations
   410‚Üí
   411‚ÜíThe transform emits a faithful 1-frame 377 anim blob that the existing
   412‚Üí377 client can render via `AnimFrame.unpack()`. The following OSRS-only
   413‚Üídata is **dropped**:
   414‚Üí
   415‚Üí| OSRS field             | 377 equivalent                | Impact |
   416‚Üí|------------------------|-------------------------------|--------|
   417‚Üí| `interpolationType`    | none                          | Blended/interpolated animations fall back to standard single-frame playback |
   418‚Üí| `leftHandWeight`       | none                          | Left-hand blend weight lost |
   419‚Üí| `rightHandWeight`      | none                          | Right-hand blend weight lost |
   420‚Üí
   421‚ÜíThe `frameId` field is REQUIRED by 377 (the head section stores it as a
   422‚Üí`u16`) but is NOT carried in the OSRS blob. `toLegacy377()` uses
   423‚Üí`this.frameId` (default 0). Callers that need a specific ID routed into
   424‚Üí`AnimFrame.instances[id]` must set `frame.frameId` before calling
   425‚Üí`toLegacy377()`.
   426‚Üí
   427‚ÜíThe skeleton (AnimBase) data is re-emitted byte-for-byte into the 377
   428‚Üíbase-data section. Since the OSRS skeleton format is identical to 377
   429‚Üífor pre-HD content, this is lossless. The 377 `AnimBase.unpack()` will
   430‚Üípush a DUPLICATE base entry into `AnimBase.instances[]` when re-decoding
   431‚Üíthe legacy blob ‚Äî the dep tracer / packer should deduplicate by content
   432‚Üíhash if memory matters.

   496‚Üí**Agent:** OsrsNpcConfigPorter

   498‚Üí
   499‚Üí### Files created / modified
   500‚Üí- **Created:** `engine/src/cache/osrs/OsrsNpcType.ts` (~836 lines, incl. extensive
   501‚Üí  JSDoc on every opcode + design-decision comments)
   502‚Üí- **Modified:** `engine/src/cache/osrs/index.ts` (added 1 named re-export:
   503‚Üí  `export { default as OsrsNpcType } from './OsrsNpcType.js';`)
   504‚Üí
   505‚Üí### Summary
   506‚ÜíPorted the OSRS NpcConfig opcode decoder to TypeScript as `OsrsNpcType`,
   507‚Üífollowing the LostCity conventions used by the existing 377 decoder at
   508‚Üí`engine/src/cache/config/NpcType.ts` and matching the sibling-class pattern
   509‚Üíestablished by tasks 5-a (`OsrsModel`) and 5-b (`OsrsAnimFrame` /
   510‚Üí`OsrsSeqType`).
   511‚Üí
   512‚ÜíThe OSRS NPC opcode set is a strict superset of 377: every 377 opcode
   513‚Üídecodes the same way, plus OSRS adds codes 23, 24, 25, 26, 27, 28, 29,
   514‚Üí118, 119, 121, 122, 123, 124, 125, 126, 127, 128, 129, and 130-134.
   515‚ÜíThis decoder handles BOTH sets so it can read an OSRS-format `npc.dat`
   516‚Üídirectly. LostCity server-side codes 200-214 are read-and-discarded
   517‚Üí(with debug log) so this decoder can also parse a LostCity npc.dat
   518‚Üíwithout crashing.
   519‚Üí
   520‚Üí### Public API
   521‚Üí```ts
   522‚Üíclass OsrsNpcType extends ConfigType {
   523‚Üí    // Static registry (mirror of OsrsSeqType's pattern):
   524‚Üí    static configNames: Map<string, number>;
   525‚Üí    static configs: OsrsNpcType[];
   526‚Üí    static get(id): OsrsNpcType;
   527‚Üí    static getId(name): number;
   528‚Üí    static getByName(name): OsrsNpcType | null;
   529‚Üí    static get count(): number;
   530‚Üí
   531‚Üí    // 377 shared fields (mirror of NpcType.ts):
   532‚Üí    name, desc, size, models, heads, hasanim, readyanim, walkanim,
   533‚Üí    walkanim_b, walkanim_r, walkanim_l, hasalpha, recol_s, recol_d,
   534‚Üí    op (size-10 lazy array), resizex, resizey, resizez, minimap,
   535‚Üí    vislevel, resizeh, resizev, alwaysontop, ambient, contrast,
   536‚Üí    headicon, turnspeed, multivarbit, multivarp, multinpc, active,
   537‚Üí    category, stats[6], params
   538‚Üí
   539‚Üí    // OSRS-only fields:
   540‚Üí    runanim, runanim_l, runanim_r, runanim_b           // code 24
   541‚Üí    crawlanim, crawlanim_l, crawlanim_r, crawlanim_b, // codes 25/26
   542‚Üí        crawlanim_b2                                    // (5th short of code 26)
   543‚Üí    recol_s_tex, recol_d_tex                           // codes 28/29
   544‚Üí    sizeH, sizeW                                       // codes 118/119
   545‚Üí    mapFunction, mapScene                              // code 121
   546‚Üí    heightmap                                          // code 122
   547‚Üí    hoverText                                          // code 123
   548‚Üí    flipX, flipY                                       // codes 124/125
   549‚Üí    forceRenderPriority, clipOnMinimap, clip, interactable  // codes 126-129
   550‚Üí    osrsExtensions: Int8Array(5)                       // codes 130-134
   551‚Üí
   552‚Üí    // Methods:
   553‚Üí    decode(code: number, dat: Packet): void;
   554‚Üí    toLegacy377NpcConfig(): string[];
   555‚Üí    extractDependencyRefs(): { models: number[]; heads: number[]; anims: number[]; params: number[] };
   556‚Üí}
   557‚Üí```
   558‚Üí
   559‚Üí### Key API decisions
   560‚Üí1. **`OsrsNpcType` extends `ConfigType` directly, NOT `NpcType`.** Same
   561‚Üí   reasoning as task 5-b's `OsrsSeqType`: subclassing `NpcType` would
   562‚Üí   inherit its private static `configs[]` / `configNames` and bound
   563‚Üí   `load/parse/get` methods, routing OSRS NPC configs into the shared
   564‚Üí   377 registry. A sibling class with a parallel registry keeps OSRS
   565‚Üí   configs isolated so the modular variant selector (task 9) can pick
   566‚Üí   which one to serve.
   567‚Üí
   568‚Üí2. **`op` array is lazily sized up to 10 entries.** Legacy opcodes 30-34
   569‚Üí   populate `op[0..4]` (op1..op5). OSRS opcode 27 reads a u8 index and
   570‚Üí   can populate any index (typically 5..9 for op6..op10). The array is
   571‚Üí   allocated on first use and grown dynamically if opcode 27's index
   572‚Üí   exceeds the current length. This avoids pre-allocating for the
   573‚Üí   common case (5 ops) while still supporting the extended set.
   574‚Üí
   575‚Üí3. **`toLegacy377NpcConfig()` emits raw asset IDs, NOT pack-file
   576‚Üí   names.** This deliberately decouples the engine class from
   577‚Üí   `#tools/pack/PackFile.js` lookups (SeqPack, NpcPack, ModelPack) ‚Äî
   578‚Üí   mirroring the existing `NpcType` class which also doesn't import
   579‚Üí   PackFile. Task 8 (content-folder writer) is expected to post-process
   580‚Üí   the emitted lines to rename IDs to debugnames via the pack files.
   581‚Üí   The format `key=<int>` survives the round-trip into the content
   582‚Üí   folder because LostCity's config parser accepts both names and
   583‚Üí   integer literals.
   584‚Üí
   585‚Üí4. **OSRS-only fields are emitted as `param=osrs_<field>,<value>`
   586‚Üí   entries.** This preserves them through the content-folder round-trip
   587‚Üí   even though the 377 client doesn't understand them. When task 9
   588‚Üí   (variant registry) registers these as well-known param IDs, the
   589‚Üí   re-packing path will preserve them as opaque param values.
   590‚Üí
   591‚Üí5. **LostCity server-side codes 200-214 are read-and-discarded, not
   592‚Üí   stored.** The spec describes these as "IGNORE them (skip silently,
   593‚Üí   log debug)" ‚Äî interpreted as: read the payload (using the known
   594‚Üí   format from `NpcType.ts`) and discard the values, so this decoder
   595‚Üí   can parse a LostCity npc.dat without crashing. The discarded values
   596‚Üí   are NOT stored on this class because OSRS-format NPCs don't carry
   597‚Üí   these fields; the modular pipeline is expected to layer server-side
   598‚Üí   config on top via the content folder rather than the cache.
   599‚Üí
   600‚Üí6. **`extractDependencyRefs()` excludes `multinpc[]`.** Multinpc
   601‚Üí   references other NPC config IDs, but those are walked via the NPC
   602‚Üí   config index, not via archive IDs (1/2 for models/anims). Including
   603‚Üí   them in the returned `params`/`anims` arrays would mix asset kinds.
   604‚Üí   Task 6 (dep tracer) is expected to walk multinpc separately by
   605‚Üí   recursing into the NPC config index.
   606‚Üí
   607‚Üí7. **`osrsExtensions` is a fixed 5-slot `Int8Array` indexed by
   608‚Üí   `code - 130`.** Codes 130-134 are described in the spec as "store
   609‚Üí   opaquely" with no semantic meaning documented upstream. A fixed
   610‚Üí   array is simpler than a `Map<number, number>` and gives O(1) access
   611‚Üí   for consumers that later want to interpret specific slots.
   612‚Üí
   613‚Üí### Opcode ambiguities resolved
   614‚Üí
   615‚Üí1. **Opcode 23 reads 5 shorts, NOT 6.** The task spec listed
   616‚Üí   "(stand, walk, walkL, walkR, walkB, run)" ‚Äî 6 names ‚Äî but said
   617‚Üí   "g2√ó5". The trailing "run" is a spec typo; standard RuneLite reads
   618‚Üí   exactly 5 shorts (stand, walk, walkLeft, walkRight, walkBack).
   619‚Üí   Decoded as: readyanim, walkanim, walkanim_l, walkanim_r, walkanim_b
   620‚Üí   (overwriting the same fields populated by 377 codes 13/14/17).
   621‚Üí
   622‚Üí2. **Opcode 26 field label is "runanim code 26" but values are all
   623‚Üí   crawl-related.** Treated as a spec typo ‚Äî opcode 26 is decoded as a
   624‚Üí   crawl-anim family extension (5 shorts: crawl, crawlL, crawlR,
   625‚Üí   crawlB, crawlBack2). Overwrites the 4 crawl fields from opcode 25
   626‚Üí   (last write wins) and additionally populates `crawlanim_b2`.
   627‚Üí
   628‚Üí3. **Opcode 17 byte order: spec vs `NpcType.ts`.** The task spec says
   629‚Üí   code 17 reads (walk, back, left, right). The existing 377
   630‚Üí   `NpcType.ts` reads (walk, back, right, left) ‚Äî left/right swapped.
   631‚Üí   The Java client NpcType.java reads in obfuscated order with unclear
   632‚Üí   semantics. LostCity's own unpack tool (`unpackNpcConfig`) reads in
   633‚Üí   the SPEC order (l before r), so two of three LostCity sources agree
   634‚Üí   with the spec. This decoder follows the spec (l before r) for code
   635‚Üí   17. Documented as a known discrepancy with `NpcType.ts`; if
   636‚Üí   LostCity ever regenerates its 377 npc.dat from RL sources, the 377
   637‚Üí   `NpcType.ts` should be updated to match.
   638‚Üí
   639‚Üí4. **Opcode 28 vs 29 (texture recolors).** Spec describes 28 as
   640‚Üí   "recol_d_texture (dst)" and 29 as "recol_s/d texture (paired with
   641‚Üí   28)" ‚Äî both with shape "g1 count, g2√ó2 per pair". The semantic
   642‚Üí   distinction between 28 and 29 is unclear from the spec; both are
   643‚Üí   decoded identically and stored in the same `recol_s_tex` /
   644‚Üí   `recol_d_tex` parallel arrays (last opcode wins, matching how
   645‚Üí   legacy opcode 40 behaves on repeat). If a future cache distinguishes
   646‚Üí   them semantically, the single `decodeTexRecolors()` helper is the
   647‚Üí   one place to split the storage.
   648‚Üí
   649‚Üí5. **`desc` field uses `gjstr()`, not `gjstrraw()`.** The task spec
   650‚Üí   says code 3 reads `gjstrraw`. The Java client's `gjstrraw()`
   651‚Üí   returns `byte[]` (raw bytes). The existing 377 `NpcType.ts` uses
   652‚Üí   `gjstr()` (which returns a string) and stores `desc` as a string.
   653‚Üí   `Packet.ts` does NOT have a `gjstrraw()` method. To match the
   654‚Üí   existing TS convention (and avoid touching `Packet.ts`), this
   655‚Üí   decoder uses `gjstr()`. If raw-bytes round-trip fidelity becomes
   656‚Üí   important, a future task should add `Packet.gjstrraw()` and store
   657‚Üí   `desc` as `Uint8Array`.
   658‚Üí
   659‚Üí6. **Opcode 101 contrast scaling.** The Java client reads `g1b() * 5`
   660‚Üí   for the effective contrast value. The existing 377 `NpcType.ts`
   661‚Üí   stores the raw `g1b()` byte (no scaling). This decoder matches the
   662‚Üí   377 TS convention (store raw byte; consumer scales). Documented in
   663‚Üí   a code comment.
   664‚Üí
   665‚Üí7. **Opcode 249 (params) inherited unchanged.** Uses
   666‚Üí   `ParamHelper.decodeParams(dat)` from the existing engine ‚Äî same
   667‚Üí   path as 377 NpcType. No OSRS-specific changes needed.
   668‚Üí
   669‚Üí### `extractDependencyRefs()` ‚Äî what it returns

   776‚Üí
   777‚Üí### Summary
   778‚ÜíImplements a depth-first dependency walker that records an OSRS NPC's full
   779‚Üítransitive dependency graph as a `DepsMap` (JSON-serializable). The walker
   780‚Üíhandles every node kind listed in the task spec (`npc`, `model`, `anim`,
   781‚Üí`anim-base`, `seq`, `obj`, `param`, `struct`, `script`, `texture`,
   782‚Üí`particle`, `sound`), with per-kind `visitXxx` methods that decode the
   783‚Üíasset via a `CacheReader` interface (abstract ‚Äî backed by either a real
   784‚ÜíOSRS cache or a test stub) and recurse into child refs.
   785‚Üí
   786‚ÜíCycle detection: per-branch `path: Set<string>` of node keys currently
   787‚Üíbeing walked. Re-entering a key records a `DepCycle` and skips recursion ‚Äî
   788‚Üíguarantees termination on pathological param cross-reference loops.
   789‚Üí
   790‚ÜíMemoization: visited nodes are cached for the lifetime of the tracer
   791‚Üíinstance, so `traceMany()` over multiple NPCs shares work across NPCs
   792‚Üíthat depend on the same model/seq/etc.
   793‚Üí
   794‚Üí### deps.json schema (TypeScript interfaces, pasted inline)
   795‚Üí
   796‚Üí```ts
   797‚Üíexport type NodeKind =
   798‚Üí    | 'npc' | 'model' | 'anim' | 'anim-base' | 'seq' | 'obj'
   799‚Üí    | 'param' | 'struct' | 'script' | 'texture' | 'particle' | 'sound';
   800‚Üí
   801‚Üíexport type NodeSource = 'osrs' | 'legacy377';
   802‚Üí
   803‚Üíexport interface DepRef {
   804‚Üí    kind: NodeKind;
   805‚Üí    id: number | string;        // string for scripts (debugname-as-id)
   806‚Üí    via?: string;               // e.g. 'readyanim', 'models[0]', 'param:attack_anim'
   807‚Üí    missing?: boolean;
   808‚Üí}
   809‚Üí
   810‚Üíexport interface DepNode {
   811‚Üí    kind: NodeKind;
   812‚Üí    id: number | string;
   813‚Üí    name: string | null;
   814‚Üí    source: NodeSource;
   815‚Üí    transformedFrom: number | string | null;  // set by Task 8 when emitting 377 twin
   816‚Üí    cycle?: boolean;
   817‚Üí    missing?: boolean;
   818‚Üí    skins?: boolean;            // model nodes only ‚Äî true if triangleSkin/vertexSkin non-empty
   819‚Üí    note?: string;              // e.g. 'TODO: deep obj walk not implemented'
   820‚Üí    deps: DepRef[];
   821‚Üí}
   822‚Üí
   823‚Üíexport interface DepCycle {
   824‚Üí    path: string[];             // node keys from walk root to reentry point
   825‚Üí    reentry: string;
   826‚Üí}
   827‚Üí
   828‚Üíexport interface DepRoot {
   829‚Üí    kind: NodeKind;
   830‚Üí    id: number | string;
   831‚Üí    name: string | null;
   832‚Üí}
   833‚Üí
   834‚Üíexport interface DepsMap {
   835‚Üí    version: 1;
   836‚Üí    root: DepRoot;
   837‚Üí    secondaryRoots?: DepRef[];  // present only for traceMany() batch output
   838‚Üí    nodes: Record<string, DepNode>;  // keyed by `${kind}:${id}` for O(1) lookup
   839‚Üí    cycles: DepCycle[];
   840‚Üí    missing: DepRef[];
   841‚Üí}
   842‚Üí
   843‚Üíexport const DEPS_SCHEMA_VERSION = 1 as const;
   844‚Üí
   845‚Üíexport function nodeKey(kind: NodeKind, id: number | string): string;
   846‚Üíexport function isRefParamType(type: string): boolean;
   847‚Üíexport function paramTypeToNodeKind(type: string): NodeKind | null;
   848‚Üí```

   850‚Üí### CacheReader interface signature (for Task 7 to implement)
   851‚Üí
   852‚Üí```ts
   853‚Üíexport interface CacheReader {
   854‚Üí    /** OSRS model blob (archive 1). Returns decoded OsrsModel or null. */
   855‚Üí    readModel(id: number): OsrsModel | null;
   856‚Üí    /** OSRS animation frame (archive 2 file). */
   857‚Üí    readAnim(id: number): OsrsAnimFrame | null;
   858‚Üí    /** OSRS animation skeleton (separate OSRS index; same byte layout as 377). */
   859‚Üí    readAnimBase(id: number): AnimBase | null;
   860‚Üí    /** OSRS sequence config (seq.dat entry). */
   861‚Üí    readSeq(id: number): OsrsSeqType | null;
   862‚Üí    /** OSRS NPC config (npc.dat entry). */
   863‚Üí    readNpc(id: number): OsrsNpcType | null;
   864‚Üí    /** Item config. Returns legacy ObjType for now (OsrsObjType not ported). */
   865‚Üí    readObj(id: number): ObjType | null;
   866‚Üí    /** Param type. Returns legacy ParamType if loaded, null for OSRS-native params. */
   867‚Üí    readParam(id: number): ParamType | null;
   868‚Üí    /** Struct config. */
   869‚Üí    readStruct(id: number): StructType | null;
   870‚Üí    /** Texture image bytes. */
   871‚Üí    readTexture(id: number): Uint8Array | null;
   872‚Üí    /** OSRS particle system bytes. */
   873‚Üí    readParticle(id: number): Uint8Array | null;
   874‚Üí    /** Sound effect bytes (synth or midi). */
   875‚Üí    readSound(id: number): Uint8Array | null;
   876‚Üí    /** Optional debugname lookup (consults content/pack/<kind>.pack). */
   877‚Üí    getName?(kind: NodeKind, id: number | string): string | null;
   878‚Üí}
   879‚Üí```
   880‚Üí
   881‚Üí**Task 7 implementation notes:**
   882‚Üí1. The OSRS cache stores skeletons (AnimBase) in a separate index from
   883‚Üí   frames. The reader MUST load bases into `AnimBase.instances[]` BEFORE
   884‚Üí   returning frames that reference them ‚Äî `OsrsAnimFrame.decode()`
   885‚Üí   requires the base to be present (see task 5-b worklog entry).
   886‚Üí2. Every method MUST return `null` on missing IDs ‚Äî do NOT throw. The
   887‚Üí   tracer records null returns as `missing: true` dep refs.
   888‚Üí3. `getName` is optional but recommended ‚Äî without it, `DepNode.name`
   889‚Üí   will be `null` for every node except NPCs (which carry their own
   890‚Üí   `name` field on `OsrsNpcType`) and seqs (which carry `debugname`).
   891‚Üí4. For `readParam`, prefer consulting an OSRS-side param registry if
   892‚Üí   one is built. Falling back to the legacy 377 `ParamType.get(id)`
   893‚Üí   works for shared params but will miss OSRS-native ones.
   894‚Üí
   895‚Üí### Key design decisions
   896‚Üí
   897‚Üí1. **`visitNode(kind, id, path)` is PUBLIC**, not private as the task
   898‚Üí   spec hinted. Rationale: the standalone `walkParams` helper needs to
   899‚Üí   call it (the spec explicitly specified `walkParams(params, tracer,
   900‚Üí   deps)` as a standalone function taking the tracer as an argument),
   901‚Üí   and future per-kind walkers (e.g. an `OsrsObjType` walker added by
   902‚Üí   task 8) should hook into the same cycle-detection + memoization
   903‚Üí   logic without duplicating it. The state it touches (`nodes`,
   904‚Üí   `cycles`, `missing`) is still encapsulated as private fields on
   905‚Üí   the tracer instance.
   906‚Üí
   907‚Üí2. **`walkParams` is BOTH a public method on the tracer (`tracer.walkParams(params, deps, path)`)
   908‚Üí   AND a standalone exported function.** The standalone wrapper exists
   909‚Üí   for parity with the task spec's `walkParams(params, tracer, deps)`
   910‚Üí   signature; it adds a required `path: Set<string>` argument (not in
   911‚Üí   the spec) because cycle detection depends on it. Callers should
   912‚Üí   prefer the method form.
   913‚Üí
   914‚Üí3. **`id` is `number | string` throughout the schema.** Numeric for
   915‚Üí   cache-indexed assets (npc/model/anim/seq/obj/param/struct/texture/
   916‚Üí   particle/sound); string for scripts (whose "id" is their debugname,
   917‚Üí   e.g. `'dragon'`). This is necessary because LostCity runescript
   918‚Üí   references assets by name (e.g. `npc_anim(dragon_attack, 0)`), and
   919‚Üí   the script scanner can't resolve names to numeric cache IDs without
   920‚Üí   the pack files (task 8's job). String-ID refs are recorded as
   921‚Üí   `missing: true` until task 8 resolves them.
   922‚Üí
   923‚Üí4. **`transformedFrom` is always `null` in the tracer's output.** It
   924‚Üí   will be populated by task 8 (content-folder writer) when it
   925‚Üí   actually generates the 377-transformed twin of an OSRS node. The
   926‚Üí   field exists in the schema now so task 8 doesn't need a schema
   927‚Üí   migration.
   928‚Üí
   929‚Üí5. **`skins` flag on model nodes.** Per task 5-a's next-action hook,
   930‚Üí   models with non-empty `triangleSkin`/`vertexSkin` arrays lose
   931‚Üí   rig-skinning data when transformed to 377. The tracer records this
   932‚Üí   as `skins: boolean` on model nodes so the pilot can prefer
   933‚Üí   `skins === false` models when alternatives exist.
   934‚Üí
   935‚Üí6. **`secondaryRoots` for batched traces.** `traceMany()` keeps the
   936‚Üí   single-`root` schema shape (backward compat with single-NPC
   937‚Üí   traces) by making the first NPC the root and listing any
   938‚Üí   additional NPCs in `secondaryRoots: DepRef[]`. The field is
   939‚Üí   omitted for single-NPC traces (the common case).
   940‚Üí
   941‚Üí7. **`ObjType` deep walk is stubbed.** Per task spec: "for now, treat
   942‚Üí   obj as a partial node and log 'TODO: deep obj walk not
   943‚Üí   implemented'". The `visitObj` method records the obj as a leaf
   944‚Üí   with `note: 'TODO: deep obj walk not implemented (OsrsObjType not
   945‚Üí   ported)'`. When task 8 ports the obj decoder, this method should
   946‚Üí   walk `obj.model`, `obj.manwear`, `obj.womanwear`, `obj.manhead`,
   947‚Üí   `obj.womanhead`, `obj.countobj[]`, `obj.recol_s/d`, and
   948‚Üí   `obj.params` (via `walkParams`).
   949‚Üí
   950‚Üí8. **`ParamType` lookup falls back to the legacy 377 registry.** When
   951‚Üí   `CacheReader.readParam(id)` returns null (the common case until
   952‚Üí   `OsrsParamType` is ported), the tracer tries `ParamType.get(id)`
   953‚Üí   (the legacy 377 static registry). If both miss, the param is
   954‚Üí   recorded with `note: 'param type unknown ‚Äî value not walked
   955‚Üí   (OsrsParamType not ported)'` and its value is NOT walked (the
   956‚Üí   importer will bring the param's value verbatim, but the tracer
   957‚Üí   can't tell what asset kind it points at).
   958‚Üí
   959‚Üí9. **Script scanning is best-effort regex.** LostCity runescript
   960‚Üí   references assets by debugname in call sites like
   961‚Üí   `npc_anim(dragon_attack, 0)`, `spotanim_npc(firebreath_attack, 92, 0)`,
   962‚Üí   `inv_total(worn, antidragonbreathshield)`, `sound_synth(dragonbreath, 0, 30)`,
   963‚Üí   `npc_param(attack_sound)`. The tracer scans for these patterns and
   964‚Üí   records the discovered names as string-ID dep refs (all marked
   965‚Üí   `missing: true` until task 8 resolves them via pack files). The
   966‚Üí   scan is intentionally conservative ‚Äî false positives would be
   967‚Üí   worse than false negatives here (the importer would try to fetch
   968‚Üí   non-existent assets).
   969‚Üí
   970‚Üí10. **Script discovery for an NPC uses TWO heuristics:**
   971‚Üí    (a) look for `content/scripts/npc/scripts/<npcName>.rs2` (primary
   972‚Üí    combat script convention); (b) scan all other `.rs2` files for
   973‚Üí    `[<trigger>,<npcName>]` dispatch directives (e.g.
   974‚Üí    `[ai_applayer2,babydragon] @dragon_ai_applayer2(false);`) and
   975‚Üí    register each matching file as a separate script dep with
   976‚Üí    `via: 'dispatch:<trigger>'`. The dispatch-scan is O(files √ó lines)
   977‚Üí    but runs only once per NPC and only over the `scripts/npc/scripts/`
   978‚Üí    subdir.
