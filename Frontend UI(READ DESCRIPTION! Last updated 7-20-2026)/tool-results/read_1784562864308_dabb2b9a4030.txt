  1700→
  1701→1. **Per-NPC player override** (`state.npcOverrides.get(npcId)`)
  1702→   - Honored UNCONDITIONALLY — the resolver does NOT check whether the chosen variant is actually available. The rationale: the override represents the player's explicit preference; `resolveNpcConfigForPlayer` on the registry is responsible for gracefully falling back to whatever's actually loadable (e.g. a player who forced `'osrs'` on an NPC with no OSRS import will see the legacy model via the fallback, but the resolver still reports `'osrs'` so the UI can flag the mismatch).
  1703→
  1704→2. **Per-region player override** (`state.regionOverrides.get(regionId)`)
  1705→   - STUB: today the registry always passes `regionId = -1` (which never matches any entry), so this step is effectively a no-op. The spawn-side wiring that would populate "current player region" is deferred to a later task. The `resolveNpcVariantForRegion(npcId, playerId, regionId)` method exists on the registry so the spawn-site migration is a one-line change once the region lookup is wired.
  1706→
  1707→3. **Player era preset** (`state.eraPreset`)
  1708→   - Respects availability (unlike steps 1 and 2). The `resolveEraPreset` helper translates:
  1709→     - `'05era'`   → `'legacy377'` (or `'osrs'` if the NPC has no legacy twin — i.e. brand-new OSRS content).
  1710→     - `'07era'`   → `'osrs'` if available, else `'legacy377'`. (Today equivalent to `'allOSRS'` because we don't track per-NPC "year added" — see the `EraPreset` docstring in `PlayerVariantState.ts`.)
  1711→     - `'allOSRS'` → `'osrs'` if available, else `'legacy377'`.
  1712→     - `'mixed'`   → same as `'05era'` (the override map is checked first; this is the fallback).
  1713→
  1714→4. **Server default** (`'05era'` → `'legacy377'`)
  1715→   - Defensive fallback — should never be reached because `PlayerVariantState` always has a valid `eraPreset`.
  1716→
  1717→**Backwards-compat guarantee:** a freshly-created state (default `eraPreset = '05era'`, empty override maps) selects `'legacy377'` for every NPC that has a legacy twin — i.e. the existing `NpcType.get(id)` path is preserved unchanged for any NPC that hasn't been imported. Only NPCs that were imported from OSRS via Task 8 (and registered in `content/deps/variants.json`) are eligible for the `'osrs'` variant — and only for players who explicitly chose an OSRS-era preset or set a per-NPC `'osrs'` override.

  1764→### Integration pattern with existing NpcType
  1765→
  1766→The variant registry is NOT a wrapper around `NpcType`. It's a side-channel: when an NPC is being spawned for a specific player, the spawn code calls `VariantRegistry.resolveNpcVariant(npcId, playerId)` to decide which config ID to use. The two paths are:
  1767→
  1768→- `'legacy377'` → use the existing `NpcType.get(npcId)` path UNCHANGED. This is the default for every NPC that hasn't been imported.
  1769→- `'osrs'`       → use `NpcType.get(osrsImportedNpcId)` — the imported NPC's new 377 pack-file ID. The imported NPC config has its own `models[]` array pointing at the OSRS-imported models' new 377 IDs — so no per-model swap is needed at the spawn site.
  1770→
  1771→A single helper does the lookup:
  1772→```ts
  1773→VariantRegistry.getInstance().resolveNpcConfigForPlayer(npcId, playerId): NpcType
  1774→```
  1775→
  1776→Returns the legacy NpcType (`NpcType.get(npcId)`) or the OSRS-imported NpcType (`NpcType.get(osrsImportedNpcId)`). Future engine work can migrate NPC spawn sites one at a time to call this helper instead of `NpcType.get(npcId)` directly. Until a spawn site is migrated, the existing `NpcType.get(npcId)` path continues to work unchanged — the variant system is invisible to it. The registry does NOT modify `NpcType.configs[]` or any other existing static registry; it only consults them (for the `legacyModelOwner` reverse index used by `resolveModelVariant`).
  1777→
  1778→The spawn-site migration is intentionally deferred — touching every `NpcType.get(...)` call in the codebase (7 files: `World.ts`, `entity/Npc.ts`, `entity/Player.ts`, `GameMap.ts`, `script/ScriptIterators.ts`, `script/handlers/NpcOps.ts`, `script/handlers/PlayerOps.ts`) is too invasive for one task. The helper is in place; the migration can happen one spawn site at a time.
  1779→
  1780→### Per-player state caching
  1781→
  1782→`resolveNpcVariant(npcId, playerId)` is SYNC (the spawn path is hot). The per-player `PlayerVariantState` is loaded asynchronously by `VariantPersistence.loadPlayerVariantState(playerId)` at login time and stashed in the registry's `playerStateCache` Map. The login handler is expected to call:
  1783→```ts
  1784→const state = await loadPlayerVariantState(playerId);
  1785→VariantRegistry.getInstance().setPlayerState(playerId, state);
  1786→```
  1787→On logout: `VariantRegistry.getInstance().clearPlayerState(playerId)`.
  1788→
  1789→If `resolveNpcVariant` is called for a playerId that's not in the cache (e.g. before the login handler has warmed it), the registry returns the server default (`'legacy377'` for everything with a legacy twin, `'osrs'` for brand-new OSRS NPCs). A warning is logged so the developer knows the cache wasn't warmed — but the engine doesn't crash.

  2080→### Design decisions / spec-ambiguity resolutions
  2081→
  2082→1. **Stub reader returns real `ParamType` stubs for params 1234/1235** —
  2083→   the dep tracer's `walkParams` only walks a param's VALUE if it can
  2084→   look up the param's TYPE (via `lookupParamType`). Without a stub
  2085→   `ParamType` with `type=ScriptVarType.SEQ`, the tracer would log
  2086→   "param type unknown" and NOT walk the value — meaning the
  2087→   `seq:9502 → anim:2006` path wouldn't be reached via params. The stub
  2088→   returns a real `ParamType` with `type=SEQ` so the param-value-walk
  2089→   code path is exercised. Mirrors what a real OSRS cache would provide
  2090→   via the (not-yet-ported) `OsrsParamType`.
  2091→
  2092→2. **Script-dispatch directive uses the OSRS display name verbatim**
  2093→   (`[ai_applayer2,Tormented demon]` with space + capital T). The dep
  2094→   tracer's `findScriptsForNpc()` uses `npc.name` (the display name) as
  2095→   the script-filename lookup key — which doesn't match LostCity's
  2096→   lowercase-with-underscores script naming convention. For the pilot,
  2097→   we sidestep this by also adding a `[ai_applayer2,Tormented demon]`
  2098→   dispatch directive to the script, which the tracer's dispatch-scan
  2099→   path picks up via the regex `^\[([a-zA-Z_][a-zA-Z0-9_]*),<escaped
  2100→   npcName>\]`. This is a known limitation of the dep tracer — the
  2101→   real-cache path will have the same behavior. Future cleanup: the
  2102→   tracer should consult `npc.debugname` (via `reader.getName`) instead
  2103→   of `npc.name` for script lookups.
  2104→
  2105→3. **NPC node name updated to sanitized debugname before registering the
  2106→   variant.** The `VariantRegistry.registerVariant()` and
  2107→   `UpdateVariantsIndex.processDepsFile()` both pull `osrsDebugname`
  2108→   from `npcNode.name`. The dep tracer sets `npcNode.name` to the OSRS
  2109→   display name (e.g. "Tormented demon") — but the variants.json
  2110→   schema expects the sanitized debugname (e.g.
  2111→   "osrs_tormented_demon"). The pilot sets `npcNode.name =
  2112→   writtenNpcEntry.debugname` (the sanitized form returned by the
  2113→   writer) before calling `registerVariant`. Future cleanup: have
  2114→   `registerVariant` look up the sanitized debugname from the npc pack
  2115→   file given the `osrsNpcId` (the `transformedFrom` value) instead
  2116→   of trusting `npcNode.name`.
  2117→
  2118→4. **Writer-skip counts include script + param nodes** — the assertion
  2119→   expects `skipped=3` (script + 2 params — kinds not yet supported by
  2120→   `ContentFolderWriter`, per Task 8 worklog "What's stubbed / deferred"
  2121→   section). The pilot's assertion text documents this explicitly so
  2122→   future readers don't mistake the skips for failures.
  2123→
  2124→5. **`UpdateVariantsIndex.ts` auto-run guarded by `import.meta.main`**
  2125→   — without this guard, importing the module for its `regenerateIndex`
  2126→   export (which the pilot does) would trigger the module-level
  2127→   `main(); process.exit()` calls as a side effect, killing the pilot
  2128→   process before any of its own code ran. The guard makes the module
  2129→   importable as a library while preserving its CLI behavior when
  2130→   invoked directly (`bun tools/osrs/UpdateVariantsIndex.ts`).
