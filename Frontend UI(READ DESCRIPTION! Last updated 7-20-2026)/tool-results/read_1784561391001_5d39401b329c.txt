    30‚Üí    30‚Üí## Architecture findings (from codebase exploration)
    31‚Üí    31‚Üí
    32‚Üí    32‚Üí### Engine side (TypeScript, `engine-ts`)
    33‚Üí    33‚Üí- **Cache reader**: `engine/src/io/FileStream.ts` ‚Äî 255-archive file store
    34‚Üí    34‚Üí  (archives 0..4). `read(archive, file, decompress)` / `write(...)`.
    35‚Üí    35‚Üí  Archives: 0=versionlist/jagfiles, 1=models, 2=anims, 3=midi, 4=map.
    36‚Üí    36‚Üí- **OnDemand**: `engine/src/engine/OnDemand.ts` ‚Äî serves cache files to client
    37‚Üí    37‚Üí  over the game socket. Three priority queues (urgent/extra/ingame).
    38‚Üí    38‚Üí- **Model decoder**: `engine/src/cache/graphics/Model.ts` ‚Äî 377-format reader.
    39‚Üí    39‚Üí  Reads 18-byte trailer (vertexCount, faceCount, texturedFaceCount, flags,
    40‚Üí    40‚Üí  data lengths), then decodes vertices/faces/textures.
    41‚Üí    41‚Üí- **Anim decoder**: `engine/src/cache/graphics/AnimFrame.ts` + `AnimBase.ts` ‚Äî
    42‚Üí    42‚Üí  reads 377 anim frames (4 sections: head/tran1/tran2/del + base data).
    43‚Üí    43‚Üí- **Config decoders**: `engine/src/cache/config/*.ts` ‚Äî NpcType, SeqType,
    44‚Üí    44‚Üí  ObjType, etc. Each has a `decode(code, dat)` opcode switch.
    45‚Üí    45‚Üí- **Pack files**: `content/pack/*.pack` ‚Äî hand-maintained `id=name` maps:
    46‚Üí    46‚Üí  `npc.pack`, `model.pack`, `anim.pack`, `seq.pack`, etc.
    47‚Üí    47‚Üí- **Pack tooling**: `engine/tools/pack/PackFileBase.ts` ‚Äî `PackFile` class
    48‚Üí    48‚Üí  with `register(id, name)`, `getByName(name)`, `save()`.
    49‚Üí    49‚Üí  `engine/tools/pack/graphics/pack.ts` writes models (`.ob2`) and anims
    50‚Üí    50‚Üí  (`.anim`) from content folder into `data/pack` cache, gzipped.
    51‚Üí    51‚Üí- **Config build**: `engine/tools/pack/config/NpcConfig.ts` etc ‚Äî compiles
    52‚Üí    52‚Üí  `[debugname]` ini-style configs into `npc.dat`/`npc.idx` for client.
    53‚Üí    53‚Üí- **Content layout**: `content/{models,pack,scripts,...}`. NPC combat scripts
    54‚Üí    54‚Üí  live in `content/scripts/npc/scripts/*.rs2` (runescript), NPC configs in
    55‚Üí    55‚Üí  `content/scripts/npc/configs/*.npc`. Params in `content/scripts/npc/configs/npc.param`.
    56‚Üí    56‚Üí
    57‚Üí    57‚Üí### Client side (Java, obfuscated)
    58‚Üí    58‚Üí- **Model decoder**: `client/src/main/java/jagex2/dash3d/Model.java` (2018 lines).
    59‚Üí    59‚Üí  `Metadata` class holds offsets; `Model(int id, int unused)` constructor
    60‚Üí    60‚Üí  decodes vertex/face arrays. `Model.method357(byte[] data, int id, byte 7)`
    61‚Üí    61‚Üí  populates `Metadata` from the 18-byte trailer.
    62‚Üí    62‚Üí- **AnimBase/AnimFrame**: `client/src/main/java/jagex2/dash3d/` ‚Äî mirrors the

    64‚Üí    64‚Üí- **NpcType**: `client/src/main/java/jagex2/config/NpcType.java` ‚Äî opcode-based
    65‚Üí    65‚Üí  decoder matching the 377 opcode set (codes 1..107).
    66‚Üí    66‚Üí- **OnDemand fetcher**: `client/src/main/java/jagex2/io/OnDemand*.java`.
    67‚Üí    67‚Üí
    68‚Üí    68‚Üí### Format diff: 377 vs OSRS (key for the port)
    69‚Üí    69‚ÜíThe OSRS model format is a **strict superset** of the 377 format:
    70‚Üí    70‚Üí- Header trailer is 18 bytes in 377, **23 bytes** in OSRS (extra 5 bytes:
    71‚Üí    71‚Üí  `version`, `textureUVCount`, `triangleSkinCount`, `vertexSkinCount`,
    72‚Üí    72‚Üí  `maxDepth`).
    73‚Üí    73‚Üí- OSRS adds: per-vertex skin data, per-triangle skin data, textured UV coords,
    74‚Üí    74‚Üí  `version` byte (used as a feature gate).
    75‚Üí    75‚Üí- Old models decode fine under OSRS rules by treating the new fields as
    76‚Üí    76‚Üí  zero-length when their flag bytes are 0.
    77‚Üí    77‚Üí- Animations changed more substantially: OSRS anim frames use a different
    78‚Üí    78‚Üí  layout (no shared `head` block ‚Äî each frame is self-contained).
    79‚Üí    79‚Üí
    80‚Üí    80‚Üí### Pack-list format (hand-maintained today)
    81‚Üí    81‚ÜíPlain text, one entry per line: `0=hans\n1=man\n...`. Generated/validated by
    82‚Üí    82‚Üí`PackFile.save()`. We will extend, not replace.
    83‚Üí    83‚Üí

   132‚Üí   132‚Üí## Task 5-a ‚Äî Port OSRS ModelDefinition decoder to TS
   133‚Üí   133‚Üí**Agent:** OsrsModelPorter
   134‚Üí   134‚Üí**Status:** done
   135‚Üí   135‚Üí**Files created** (both under `engine/src/cache/osrs/`):
   136‚Üí   136‚Üí- `OsrsModel.ts` (~822 lines incl. extensive comments)
   137‚Üí   137‚Üí- `index.ts` (barrel, 2 lines, re-exports default + `OsrsModelMetadata`)
   138‚Üí   138‚Üí
   139‚Üí   139‚Üí### Summary
   140‚Üí   140‚ÜíPorted RuneLite's `ModelDefinition.java` to TypeScript as `OsrsModel`, following
   141‚Üí   141‚Üíthe LostCity conventions used by the existing 377 decoder at
   142‚Üí   142‚Üí`engine/src/cache/graphics/Model.ts`. Output uses `Packet` from `#/io/Packet.js`
   143‚Üí   143‚Üífor all byte reading, `printWarning` from `#/util/Logger.js` for warnings, and
   144‚Üí   144‚Üíthe same 4-space / single-quote / semi style as the rest of the engine.
   145‚Üí   145‚Üí
   146‚Üí   146‚Üí### Public API
   147‚Üí   147‚Üí```ts
   148‚Üí   148‚Üíclass OsrsModel {
   149‚Üí   149‚Üí  static decode(id: number, data: Uint8Array | null): OsrsModel | null;
   150‚Üí   150‚Üí  static detect(data: Uint8Array): 'osrs' | 'legacy377';
   151‚Üí   151‚Üí  toLegacy377(): Uint8Array;
   152‚Üí   152‚Üí  // plus public decoded fields (typed arrays) and header counts
   153‚Üí   153‚Üí}
   154‚Üí   154‚Üí
   155‚Üí   155‚Üíclass OsrsModelMetadata {
   156‚Üí   156‚Üí  // all trailer fields + computed section offsets
   157‚Üí   157‚Üí}
   158‚Üí   158‚Üí```
   159‚Üí   159‚Üí
   160‚Üí   160‚ÜíThe class is **API-compatible in spirit** with the existing 377 `Model` class
   161‚Üí   161‚Üí(sibling `Metadata` class + static `decode()`), but the OSRS decoder is
   162‚Üí   162‚Üíone-shot (no separate `unpack`/`fromId` split) ‚Äî it is intended for pipeline
   163‚Üí   163‚Üíuse (decode ‚Üí transform ‚Üí pack), not for runtime client caching. The 377
   164‚Üí   164‚Üídecoder's two-stage split was an optimisation for the live game server; the
   165‚Üí   165‚ÜíOSRS pipeline doesn't need it.
   166‚Üí   166‚Üí
   167‚Üí   167‚Üí### Spec ambiguities resolved
   168‚Üí   168‚Üí
   169‚Üí   169‚Üí1. **Trailer byte count.** The task spec listed 12 trailer fields (17 bytes)
   170‚Üí   170‚Üí   but asserted the trailer is 23 bytes. Resolved by combining the spec list
   171‚Üí   171‚Üí   with the worklog note ("extra 5 bytes: `version`, `textureUVCount`,
   172‚Üí   172‚Üí   `triangleSkinCount`, `vertexSkinCount`, `maxDepth`") and keeping every 377
   173‚Üí   173‚Üí   field (including `priority`, which the spec omitted). Final layout = 17
   174‚Üí   174‚Üí   fields, 23 bytes total. The 5 NEW bytes are: `version`,
   175‚Üí   175‚Üí   `hasFaceTextures`, `triangleSkinCount`, `vertexSkinCount`, `maxDepth`
   176‚Üí   176‚Üí   (all u8). `textureUVCount` from the worklog is interpreted as the same
   177‚Üí   177‚Üí   field as `triangleTextureCount` (the count of textured triangles that
   178‚Üí   178‚Üí   carry UVs).
   179‚Üí   179‚Üí
   180‚Üí   180‚Üí2. **Triangle orientation/type byte.** The spec lists section 11
   181‚Üí   181‚Üí   ("triangle vertex indices") as using the "same 1/2/3/4 type encoding as
   182‚Üí   182‚Üí   377" but doesn't list a separate orientation section. In 377 the
   183‚Üí   183‚Üí   orientation byte lives in its own 1-byte-per-face section read BEFORE
   184‚Üí   184‚Üí   the index deltas. Resolved by adding a triangle-orientation section
   185‚Üí   185‚Üí   (1 byte per triangle) at body position 5.5 (between vertex labels and
   186‚Üí   186‚Üí   triangle colours) ‚Äî the same role as 377's `faceOrientationsOffset`.
   187‚Üí   187‚Üí   The body section order comment in the source makes this explicit.
   188‚Üí   188‚Üí
   189‚Üí   189‚Üí3. **`priority` byte.** 377 has a `priority` byte (255 = per-face priority
   190‚Üí   190‚Üí   array; otherwise shared priority value). The OSRS spec body section
   191‚Üí   191‚Üí   list has no priority section, but dropping the byte entirely would lose
   192‚Üí   192‚Üí   information needed for the 377 backward-compat path. Kept the byte in
   193‚Üí   193‚Üí   the OSRS trailer at the same trailer position as 377 (between
   194‚Üí   194‚Üí   `hasFaceTextures` and `hasFaceAlphas`).
   195‚Üí   195‚Üí
   196‚Üí   196‚Üí4. **Skin section encoding.** Spec body sections 13/14 say "only if
   197‚Üí   197‚Üí   `version >= 1` AND skinCount > 0" but don't say how each skin entry is
   198‚Üí   198‚Üí   encoded. Chose u8 per entry (matching the modern RuneLite decoder).
   199‚Üí   199‚Üí   If a future cache uses smart-encoded skin entries, this is the single
   200‚Üí   200‚Üí   line to change.
   201‚Üí   201‚Üí
   202‚Üí   202‚Üí5. **Detect heuristic.** `detect()` sniffs by reading the version byte at
   203‚Üí   203‚Üí   `data.length - 23`, then validates vertexCount/triangleCount/dataLen
   204‚Üí   204‚Üí   fields against the blob size. Falls back to `'legacy377'` on any
   205‚Üí   205‚Üí   sanity failure ‚Äî the legacy decoder will then either succeed or report
   206‚Üí   206‚Üí   its own error. The safe default prevents false-positive OSRS
   207‚Üí   207‚Üí   classification of malformed 377 blobs.
   208‚Üí   208‚Üí
   209‚Üí   209‚Üí### `toLegacy377()` transform ‚Äî limitations
   210‚Üí   210‚Üí
   211‚Üí   211‚ÜíThe transform emits a faithful 377 `.ob2` blob that the existing Java client
   212‚Üí   212‚Üícan render. The following OSRS-only data is **dropped**:
   213‚Üí   213‚Üí
   214‚Üí   214‚Üí| OSRS field             | 377 equivalent                | Impact |
   215‚Üí   215‚Üí|------------------------|-------------------------------|--------|
   216‚Üí   216‚Üí| `triangleMaterial`     | none                          | Faces with material textures fall back to flat colour |
   217‚Üí   217‚Üí| `triangleSkin`         | none                          | Rig-based triangle skinning lost |
   218‚Üí   218‚Üí| `vertexSkin`           | none                          | Rig-based vertex skinning lost |
   219‚Üí   219‚Üí| `maxDepth`             | none                          | Bounding-box hint lost (client recomputes) |
   220‚Üí   220‚Üí
   221‚Üí   221‚ÜíThe dependency tracer (task 6) MUST flag models that have non-zero
   222‚Üí   222‚Üí`triangleSkin`/`vertexSkin` arrays, because animations targeting those rigs
   223‚Üí   223‚Üíwill not play correctly when the model is rendered as a 377 blob. The pilot
   224‚Üí   224‚ÜíNPC (Tormented Demon) selection should prefer models with `triangleSkinCount
   225‚Üí   225‚Üí== 0 && vertexSkinCount == 0` if any are available.
   226‚Üí   226‚Üí
   227‚Üí   227‚ÜíRound-trip verified by smoke test:
   228‚Üí   228‚Üí  OSRS bytes ‚Üí `OsrsModel.decode()` ‚Üí `toLegacy377()` ‚Üí `Model.unpack()`
   229‚Üí   229‚Üí  ‚Üí `Model.fromId()` ‚Äî all vertex positions, triangle indices, colours,
   230‚Üí   230‚Üí  infos, alphas, tags, vertex labels, and per-face/shared priorities
   231‚Üí   231‚Üí  match exactly. Textured UV vertex indices also survive (377 has the
   232‚Üí   232‚Üí  same field shape ‚Äî 3 u16 vertex indices per textured triangle).
   233‚Üí   233‚Üí

   329‚Üí   329‚Üí### Key API decisions
   330‚Üí   330‚Üí1. **`OsrsAnimFrame` is a sibling class, not `extends AnimFrame`.**
   331‚Üí   331‚Üí   The 377 `AnimFrame` is heavily tied to its 4-section `unpack()` static
   332‚Üí   332‚Üí   loader and the `AnimFrame.instances[]` / `AnimFrame.order[]` registries
   333‚Üí   333‚Üí   managed by `OnDemand.cache`. Subclassing would inherit those statics
   334‚Üí   334‚Üí   verbatim and route OSRS frames into the live 377 registry ‚Äî exactly
   335‚Üí   335‚Üí   what the modular pipeline wants to avoid. A sibling class with matching
   336‚Üí   336‚Üí   field shape gives polymorphism without unwanted registry coupling.
   337‚Üí   337‚Üí   This mirrors the sibling-class pattern used by `OsrsModel` (task 5-a).
   338‚Üí   338‚Üí
   339‚Üí   339‚Üí2. **`OsrsSeqType` is a sibling class extending `ConfigType` directly**, not
   340‚Üí   340‚Üí   `extends SeqType`. Same reasoning as above ‚Äî `SeqType` has private
   341‚Üí   341‚Üí   static `configs[]` / `configNames` and bound `load/parse/get` methods
   342‚Üí   342‚Üí   that would route OSRS seqs into the shared 377 registry.
   343‚Üí   343‚Üí
   344‚Üí   344‚Üí3. **`OsrsAnimFrame.decode()` takes `baseId` as a parameter** even though
   345‚Üí   345‚Üí   the trailing 2 bytes of the blob also encode it. We cross-check and
   346‚Üí   346‚Üí   warn on mismatch but trust the parameter ‚Äî the caller is expected to
   347‚Üí   347‚Üí   have resolved the base ID through the cache index, and the trailing
   348‚Üí   348‚Üí   bytes are a redundant integrity check (sometimes zero in OSRS caches
   349‚Üí   349‚Üí   that store the base reference in a separate index file).
   350‚Üí   350‚Üí
   351‚Üí   351‚Üí4. **`frameId` defaults to 0** in `toLegacy377()` because the OSRS blob
   352‚Üí   352‚Üí   doesn't carry a frame ID. The modular packer is expected to assign a
   353‚Üí   353‚Üí   real ID (and rewrite it into the head section if needed) before the
   354‚Üí   354‚Üí   blob reaches the live client.
   355‚Üí   355‚Üí
   356‚Üí   356‚Üí5. **`OsrsSeqType.decode()` throws on codes 16+** (sub-frame references).
   357‚Üí   357‚Üí   The task spec describes them only as "16+ for sub-frame references"
   358‚Üí   358‚Üí   without a layout. Throwing lets the dep tracer (task 6) flag any seq
   359‚Üí   359‚Üí   that uses them, rather than silently dropping data.
   360‚Üí   360‚Üí
   361‚Üí   361‚Üí### Format ambiguities resolved
   362‚Üí   362‚Üí
   363‚Üí   363‚Üí1. **"For groups with no flag, the previous group's value carries forward
   364‚Üí   364‚Üí   (or 0 for translate, 128 for scale ‚Äî same default-value rule as 377)"**
   365‚Üí   365‚Üí   ‚Äî interpreted as: use the SAME default-value rule as the 377 decoder
   366‚Üí   366‚Üí   (0 for OP_TRANSLATE/OP_ROTATE/OP_ALPHA, 128 for OP_SCALE). The
   367‚Üí   367‚Üí   "carry forward" phrasing in the spec was a description of how the
   368‚Üí   368‚Üí   default value doesn't change between adjacent same-bone-type groups,
   369‚Üí   369‚Üí   not a request to track a per-axis running last-value. The synthetic
   370‚Üí   370‚Üí   OP_BASE pull-forward walk from `AnimFrame.unpack()` is preserved
   371‚Üí   371‚Üí   unchanged ‚Äî it emits a zero-value entry for the most recent OP_BASE
   372‚Üí   372‚Üí   bone when a non-OP_BASE bone has a non-zero flag.
   373‚Üí   373‚Üí
   374‚Üí   374‚Üí2. **OSRS body layout is INTERLEAVED, not sectioned.** The 377 decoder
   375‚Üí   375‚Üí   reads flags from a separate `tran1` section and smart values from a
   376‚Üí   376‚Üí   separate `tran2` section. The OSRS spec describes "For each group i:
   377‚Üí   377‚Üí   read typeFlag, then read axis values for set bits" ‚Äî i.e., the flag
   378‚Üí   378‚Üí   byte and the smart values are read sequentially from a single
   379‚Üí   379‚Üí   section. This matches RuneLite's `FrameDefinition` decoder. Verified
   380‚Üí   380‚Üí   by round-trip smoke test below.
   381‚Üí   381‚Üí
   382‚Üí   382‚Üí3. **Optional interpolation tail.** RuneLite's `FrameDefinition` reads
   383‚Üí   383‚Üí   (after the delay byte) an optional `interpolationType: u8`, then
   384‚Üí   384‚Üí   (if 4 more bytes are present) `leftHandItem: u16` and
   385‚Üí   385‚Üí   `rightHandItem: u16`. The task spec calls these `leftHandWeight` /
   386‚Üí   386‚Üí   `rightHandWeight` ‚Äî I followed the spec's naming but kept RuneLite's
   387‚Üí   387‚Üí   read layout (1 byte if any remain, then 4 bytes if at least 5 remain
   388‚Üí   388‚Üí   total). If a future cache uses a different layout, this is the
   389‚Üí   389‚Üí   single block to revisit.
   390‚Üí   390‚Üí
   391‚Üí   391‚Üí4. **Opcode 12 (`field790` in the Java client).** The 377 TS
   392‚Üí   392‚Üí   `SeqType.decode()` does NOT handle code 12 (it throws). The Java
   393‚Üí   393‚Üí   client does, reading a `u32`. The OSRS spec doesn't mention code 12
   394‚Üí   394‚Üí   explicitly, but since OSRS is a superset of the 377 client opcode
   395‚Üí   395‚Üí   set, code 12 should be handled. I added it as `field12: u32` with a
   396‚Üí   396‚Üí   comment documenting it as an opaque field carried over from the
   397‚Üí   397‚Üí   Java client (`field790`) ‚Äî its semantic meaning isn't documented
   398‚Üí   398‚Üí   upstream.
   399‚Üí   399‚Üí
   400‚Üí   400‚Üí5. **`AnimBase.instances[]` is reused.** The OSRS skeleton format is
   401‚Üí   401‚Üí   byte-identical to 377 for pre-HD content (same `length u8`, `types
   402‚Üí   402‚Üí   u8[]`, `labels u8[][]` layout). Rather than duplicate the loader,
   403‚Üí   403‚Üí   `OsrsAnimFrame` references the existing `AnimBase.instances[]`
   404‚Üí   404‚Üí   registry. OSRS-only pipelines that don't load 377 anims must still
   405‚Üí   405‚Üí   call `AnimBase.unpack()` on OSRS base files to populate this
   406‚Üí   406‚Üí   registry before invoking `OsrsAnimFrame.decode()`. (This is the
   407‚Üí   407‚Üí   task 6 dep tracer's responsibility.)
   408‚Üí   408‚Üí
   409‚Üí   409‚Üí### `toLegacy377()` transform ‚Äî limitations
   410‚Üí   410‚Üí
   411‚Üí   411‚ÜíThe transform emits a faithful 1-frame 377 anim blob that the existing
   412‚Üí   412‚Üí377 client can render via `AnimFrame.unpack()`. The following OSRS-only
   413‚Üí   413‚Üídata is **dropped**:
   414‚Üí   414‚Üí
   415‚Üí   415‚Üí| OSRS field             | 377 equivalent                | Impact |
   416‚Üí   416‚Üí|------------------------|-------------------------------|--------|
   417‚Üí   417‚Üí| `interpolationType`    | none                          | Blended/interpolated animations fall back to standard single-frame playback |
   418‚Üí   418‚Üí| `leftHandWeight`       | none                          | Left-hand blend weight lost |
   419‚Üí   419‚Üí| `rightHandWeight`      | none                          | Right-hand blend weight lost |
   420‚Üí   420‚Üí
   421‚Üí   421‚ÜíThe `frameId` field is REQUIRED by 377 (the head section stores it as a
   422‚Üí   422‚Üí`u16`) but is NOT carried in the OSRS blob. `toLegacy377()` uses
   423‚Üí   423‚Üí`this.frameId` (default 0). Callers that need a specific ID routed into
   424‚Üí   424‚Üí`AnimFrame.instances[id]` must set `frame.frameId` before calling
   425‚Üí   425‚Üí`toLegacy377()`.
   426‚Üí   426‚Üí
   427‚Üí   427‚ÜíThe skeleton (AnimBase) data is re-emitted byte-for-byte into the 377
   428‚Üí   428‚Üíbase-data section. Since the OSRS skeleton format is identical to 377
   429‚Üí   429‚Üífor pre-HD content, this is lossless. The 377 `AnimBase.unpack()` will
   430‚Üí   430‚Üípush a DUPLICATE base entry into `AnimBase.instances[]` when re-decoding
   431‚Üí   431‚Üíthe legacy blob ‚Äî the dep tracer / packer should deduplicate by content
   432‚Üí   432‚Üíhash if memory matters.

   542‚Üí   542‚Üí        crawlanim_b2                                    // (5th short of code 26)
   543‚Üí   543‚Üí    recol_s_tex, recol_d_tex                           // codes 28/29
   544‚Üí   544‚Üí    sizeH, sizeW                                       // codes 118/119
   545‚Üí   545‚Üí    mapFunction, mapScene                              // code 121
   546‚Üí   546‚Üí    heightmap                                          // code 122
   547‚Üí   547‚Üí    hoverText                                          // code 123
   548‚Üí   548‚Üí    flipX, flipY                                       // codes 124/125
   549‚Üí   549‚Üí    forceRenderPriority, clipOnMinimap, clip, interactable  // codes 126-129
   550‚Üí   550‚Üí    osrsExtensions: Int8Array(5)                       // codes 130-134
   551‚Üí   551‚Üí
   552‚Üí   552‚Üí    // Methods:
   553‚Üí   553‚Üí    decode(code: number, dat: Packet): void;
   554‚Üí   554‚Üí    toLegacy377NpcConfig(): string[];
   555‚Üí   555‚Üí    extractDependencyRefs(): { models: number[]; heads: number[]; anims: number[]; params: number[] };
   556‚Üí   556‚Üí}
   557‚Üí   557‚Üí```
   558‚Üí   558‚Üí
   559‚Üí   559‚Üí### Key API decisions
   560‚Üí   560‚Üí1. **`OsrsNpcType` extends `ConfigType` directly, NOT `NpcType`.** Same
   561‚Üí   561‚Üí   reasoning as task 5-b's `OsrsSeqType`: subclassing `NpcType` would
   562‚Üí   562‚Üí   inherit its private static `configs[]` / `configNames` and bound
   563‚Üí   563‚Üí   `load/parse/get` methods, routing OSRS NPC configs into the shared
   564‚Üí   564‚Üí   377 registry. A sibling class with a parallel registry keeps OSRS
   565‚Üí   565‚Üí   configs isolated so the modular variant selector (task 9) can pick
   566‚Üí   566‚Üí   which one to serve.
   567‚Üí   567‚Üí
   568‚Üí   568‚Üí2. **`op` array is lazily sized up to 10 entries.** Legacy opcodes 30-34
   569‚Üí   569‚Üí   populate `op[0..4]` (op1..op5). OSRS opcode 27 reads a u8 index and
   570‚Üí   570‚Üí   can populate any index (typically 5..9 for op6..op10). The array is
   571‚Üí   571‚Üí   allocated on first use and grown dynamically if opcode 27's index
   572‚Üí   572‚Üí   exceeds the current length. This avoids pre-allocating for the
   573‚Üí   573‚Üí   common case (5 ops) while still supporting the extended set.
   574‚Üí   574‚Üí
   575‚Üí   575‚Üí3. **`toLegacy377NpcConfig()` emits raw asset IDs, NOT pack-file
   576‚Üí   576‚Üí   names.** This deliberately decouples the engine class from
   577‚Üí   577‚Üí   `#tools/pack/PackFile.js` lookups (SeqPack, NpcPack, ModelPack) ‚Äî
   578‚Üí   578‚Üí   mirroring the existing `NpcType` class which also doesn't import
   579‚Üí   579‚Üí   PackFile. Task 8 (content-folder writer) is expected to post-process
   580‚Üí   580‚Üí   the emitted lines to rename IDs to debugnames via the pack files.
   581‚Üí   581‚Üí   The format `key=<int>` survives the round-trip into the content
   582‚Üí   582‚Üí   folder because LostCity's config parser accepts both names and
   583‚Üí   583‚Üí   integer literals.
   584‚Üí   584‚Üí
   585‚Üí   585‚Üí4. **OSRS-only fields are emitted as `param=osrs_<field>,<value>`
   586‚Üí   586‚Üí   entries.** This preserves them through the content-folder round-trip
   587‚Üí   587‚Üí   even though the 377 client doesn't understand them. When task 9
   588‚Üí   588‚Üí   (variant registry) registers these as well-known param IDs, the
   589‚Üí   589‚Üí   re-packing path will preserve them as opaque param values.
   590‚Üí   590‚Üí
   591‚Üí   591‚Üí5. **LostCity server-side codes 200-214 are read-and-discarded, not
   592‚Üí   592‚Üí   stored.** The spec describes these as "IGNORE them (skip silently,
   593‚Üí   593‚Üí   log debug)" ‚Äî interpreted as: read the payload (using the known
   594‚Üí   594‚Üí   format from `NpcType.ts`) and discard the values, so this decoder
   595‚Üí   595‚Üí   can parse a LostCity npc.dat without crashing. The discarded values
   596‚Üí   596‚Üí   are NOT stored on this class because OSRS-format NPCs don't carry
   597‚Üí   597‚Üí   these fields; the modular pipeline is expected to layer server-side
   598‚Üí   598‚Üí   config on top via the content folder rather than the cache.
   599‚Üí   599‚Üí
   600‚Üí   600‚Üí6. **`extractDependencyRefs()` excludes `multinpc[]`.** Multinpc
   601‚Üí   601‚Üí   references other NPC config IDs, but those are walked via the NPC
   602‚Üí   602‚Üí   config index, not via archive IDs (1/2 for models/anims). Including
   603‚Üí   603‚Üí   them in the returned `params`/`anims` arrays would mix asset kinds.
   604‚Üí   604‚Üí   Task 6 (dep tracer) is expected to walk multinpc separately by
   605‚Üí   605‚Üí   recursing into the NPC config index.
   606‚Üí   606‚Üí
   607‚Üí   607‚Üí7. **`osrsExtensions` is a fixed 5-slot `Int8Array` indexed by
   608‚Üí   608‚Üí   `code - 130`.** Codes 130-134 are described in the spec as "store
   609‚Üí   609‚Üí   opaquely" with no semantic meaning documented upstream. A fixed
   610‚Üí   610‚Üí   array is simpler than a `Map<number, number>` and gives O(1) access
   611‚Üí   611‚Üí   for consumers that later want to interpret specific slots.
   612‚Üí   612‚Üí
   613‚Üí   613‚Üí### Opcode ambiguities resolved
   614‚Üí   614‚Üí
   615‚Üí   615‚Üí1. **Opcode 23 reads 5 shorts, NOT 6.** The task spec listed
   616‚Üí   616‚Üí   "(stand, walk, walkL, walkR, walkB, run)" ‚Äî 6 names ‚Äî but said
   617‚Üí   617‚Üí   "g2√ó5". The trailing "run" is a spec typo; standard RuneLite reads
   618‚Üí   618‚Üí   exactly 5 shorts (stand, walk, walkLeft, walkRight, walkBack).
   619‚Üí   619‚Üí   Decoded as: readyanim, walkanim, walkanim_l, walkanim_r, walkanim_b
   620‚Üí   620‚Üí   (overwriting the same fields populated by 377 codes 13/14/17).
   621‚Üí   621‚Üí
   622‚Üí   622‚Üí2. **Opcode 26 field label is "runanim code 26" but values are all
   623‚Üí   623‚Üí   crawl-related.** Treated as a spec typo ‚Äî opcode 26 is decoded as a
   624‚Üí   624‚Üí   crawl-anim family extension (5 shorts: crawl, crawlL, crawlR,
   625‚Üí   625‚Üí   crawlB, crawlBack2). Overwrites the 4 crawl fields from opcode 25
   626‚Üí   626‚Üí   (last write wins) and additionally populates `crawlanim_b2`.
   627‚Üí   627‚Üí
   628‚Üí   628‚Üí3. **Opcode 17 byte order: spec vs `NpcType.ts`.** The task spec says
   629‚Üí   629‚Üí   code 17 reads (walk, back, left, right). The existing 377
   630‚Üí   630‚Üí   `NpcType.ts` reads (walk, back, right, left) ‚Äî left/right swapped.
   631‚Üí   631‚Üí   The Java client NpcType.java reads in obfuscated order with unclear
   632‚Üí   632‚Üí   semantics. LostCity's own unpack tool (`unpackNpcConfig`) reads in
   633‚Üí   633‚Üí   the SPEC order (l before r), so two of three LostCity sources agree
   634‚Üí   634‚Üí   with the spec. This decoder follows the spec (l before r) for code
   635‚Üí   635‚Üí   17. Documented as a known discrepancy with `NpcType.ts`; if
   636‚Üí   636‚Üí   LostCity ever regenerates its 377 npc.dat from RL sources, the 377
   637‚Üí   637‚Üí   `NpcType.ts` should be updated to match.
   638‚Üí   638‚Üí
   639‚Üí   639‚Üí4. **Opcode 28 vs 29 (texture recolors).** Spec describes 28 as
   640‚Üí   640‚Üí   "recol_d_texture (dst)" and 29 as "recol_s/d texture (paired with
   641‚Üí   641‚Üí   28)" ‚Äî both with shape "g1 count, g2√ó2 per pair". The semantic
   642‚Üí   642‚Üí   distinction between 28 and 29 is unclear from the spec; both are
   643‚Üí   643‚Üí   decoded identically and stored in the same `recol_s_tex` /
   644‚Üí   644‚Üí   `recol_d_tex` parallel arrays (last opcode wins, matching how
   645‚Üí   645‚Üí   legacy opcode 40 behaves on repeat). If a future cache distinguishes
   646‚Üí   646‚Üí   them semantically, the single `decodeTexRecolors()` helper is the
   647‚Üí   647‚Üí   one place to split the storage.
   648‚Üí   648‚Üí
   649‚Üí   649‚Üí5. **`desc` field uses `gjstr()`, not `gjstrraw()`.** The task spec
   650‚Üí   650‚Üí   says code 3 reads `gjstrraw`. The Java client's `gjstrraw()`
   651‚Üí   651‚Üí   returns `byte[]` (raw bytes). The existing 377 `NpcType.ts` uses
   652‚Üí   652‚Üí   `gjstr()` (which returns a string) and stores `desc` as a string.
   653‚Üí   653‚Üí   `Packet.ts` does NOT have a `gjstrraw()` method. To match the
   654‚Üí   654‚Üí   existing TS convention (and avoid touching `Packet.ts`), this
   655‚Üí   655‚Üí   decoder uses `gjstr()`. If raw-bytes round-trip fidelity becomes
   656‚Üí   656‚Üí   important, a future task should add `Packet.gjstrraw()` and store
   657‚Üí   657‚Üí   `desc` as `Uint8Array`.
   658‚Üí   658‚Üí
   659‚Üí   659‚Üí6. **Opcode 101 contrast scaling.** The Java client reads `g1b() * 5`
   660‚Üí   660‚Üí   for the effective contrast value. The existing 377 `NpcType.ts`
   661‚Üí   661‚Üí   stores the raw `g1b()` byte (no scaling). This decoder matches the
   662‚Üí   662‚Üí   377 TS convention (store raw byte; consumer scales). Documented in
   663‚Üí   663‚Üí   a code comment.
   664‚Üí   664‚Üí
   665‚Üí   665‚Üí7. **Opcode 249 (params) inherited unchanged.** Uses
   666‚Üí   666‚Üí   `ParamHelper.decodeParams(dat)` from the existing engine ‚Äî same
   667‚Üí   667‚Üí   path as 377 NpcType. No OSRS-specific changes needed.
   668‚Üí   668‚Üí
   669‚Üí   669‚Üí### `extractDependencyRefs()` ‚Äî what it returns
   670‚Üí   670‚Üí
   671‚Üí   671‚Üí```ts
   672‚Üí   672‚Üí{
   673‚Üí   673‚Üí    models: number[];   // archive-1 IDs from `models[]` (code 1)

   777‚Üí   777‚Üí### Summary
   778‚Üí   778‚ÜíImplements a depth-first dependency walker that records an OSRS NPC's full
   779‚Üí   779‚Üítransitive dependency graph as a `DepsMap` (JSON-serializable). The walker
   780‚Üí   780‚Üíhandles every node kind listed in the task spec (`npc`, `model`, `anim`,
   781‚Üí   781‚Üí`anim-base`, `seq`, `obj`, `param`, `struct`, `script`, `texture`,
   782‚Üí   782‚Üí`particle`, `sound`), with per-kind `visitXxx` methods that decode the
   783‚Üí   783‚Üíasset via a `CacheReader` interface (abstract ‚Äî backed by either a real
   784‚Üí   784‚ÜíOSRS cache or a test stub) and recurse into child refs.
   785‚Üí   785‚Üí
   786‚Üí   786‚ÜíCycle detection: per-branch `path: Set<string>` of node keys currently
   787‚Üí   787‚Üíbeing walked. Re-entering a key records a `DepCycle` and skips recursion ‚Äî
   788‚Üí   788‚Üíguarantees termination on pathological param cross-reference loops.
   789‚Üí   789‚Üí
   790‚Üí   790‚ÜíMemoization: visited nodes are cached for the lifetime of the tracer
   791‚Üí   791‚Üíinstance, so `traceMany()` over multiple NPCs shares work across NPCs
   792‚Üí   792‚Üíthat depend on the same model/seq/etc.
   793‚Üí   793‚Üí
   794‚Üí   794‚Üí### deps.json schema (TypeScript interfaces, pasted inline)
   795‚Üí   795‚Üí
   796‚Üí   796‚Üí```ts
   797‚Üí   797‚Üíexport type NodeKind =

   880‚Üí   880‚Üí
   881‚Üí   881‚Üí**Task 7 implementation notes:**
   882‚Üí   882‚Üí1. The OSRS cache stores skeletons (AnimBase) in a separate index from
   883‚Üí   883‚Üí   frames. The reader MUST load bases into `AnimBase.instances[]` BEFORE
   884‚Üí   884‚Üí   returning frames that reference them ‚Äî `OsrsAnimFrame.decode()`
   885‚Üí   885‚Üí   requires the base to be present (see task 5-b worklog entry).
   886‚Üí   886‚Üí2. Every method MUST return `null` on missing IDs ‚Äî do NOT throw. The
   887‚Üí   887‚Üí   tracer records null returns as `missing: true` dep refs.
   888‚Üí   888‚Üí3. `getName` is optional but recommended ‚Äî without it, `DepNode.name`
   889‚Üí   889‚Üí   will be `null` for every node except NPCs (which carry their own
   890‚Üí   890‚Üí   `name` field on `OsrsNpcType`) and seqs (which carry `debugname`).
   891‚Üí   891‚Üí4. For `readParam`, prefer consulting an OSRS-side param registry if
   892‚Üí   892‚Üí   one is built. Falling back to the legacy 377 `ParamType.get(id)`
   893‚Üí   893‚Üí   works for shared params but will miss OSRS-native ones.
   894‚Üí   894‚Üí
   895‚Üí   895‚Üí### Key design decisions
   896‚Üí   896‚Üí
   897‚Üí   897‚Üí1. **`visitNode(kind, id, path)` is PUBLIC**, not private as the task
   898‚Üí   898‚Üí   spec hinted. Rationale: the standalone `walkParams` helper needs to
   899‚Üí   899‚Üí   call it (the spec explicitly specified `walkParams(params, tracer,
   900‚Üí   900‚Üí   deps)` as a standalone function taking the tracer as an argument),
   901‚Üí   901‚Üí   and future per-kind walkers (e.g. an `OsrsObjType` walker added by
   902‚Üí   902‚Üí   task 8) should hook into the same cycle-detection + memoization
   903‚Üí   903‚Üí   logic without duplicating it. The state it touches (`nodes`,
   904‚Üí   904‚Üí   `cycles`, `missing`) is still encapsulated as private fields on
   905‚Üí   905‚Üí   the tracer instance.
   906‚Üí   906‚Üí
   907‚Üí   907‚Üí2. **`walkParams` is BOTH a public method on the tracer (`tracer.walkParams(params, deps, path)`)
   908‚Üí   908‚Üí   AND a standalone exported function.** The standalone wrapper exists
   909‚Üí   909‚Üí   for parity with the task spec's `walkParams(params, tracer, deps)`
   910‚Üí   910‚Üí   signature; it adds a required `path: Set<string>` argument (not in
   911‚Üí   911‚Üí   the spec) because cycle detection depends on it. Callers should
   912‚Üí   912‚Üí   prefer the method form.
   913‚Üí   913‚Üí
   914‚Üí   914‚Üí3. **`id` is `number | string` throughout the schema.** Numeric for
   915‚Üí   915‚Üí   cache-indexed assets (npc/model/anim/seq/obj/param/struct/texture/
   916‚Üí   916‚Üí   particle/sound); string for scripts (whose "id" is their debugname,
   917‚Üí   917‚Üí   e.g. `'dragon'`). This is necessary because LostCity runescript
   918‚Üí   918‚Üí   references assets by name (e.g. `npc_anim(dragon_attack, 0)`), and
   919‚Üí   919‚Üí   the script scanner can't resolve names to numeric cache IDs without
   920‚Üí   920‚Üí   the pack files (task 8's job). String-ID refs are recorded as
   921‚Üí   921‚Üí   `missing: true` until task 8 resolves them.
   922‚Üí   922‚Üí
   923‚Üí   923‚Üí4. **`transformedFrom` is always `null` in the tracer's output.** It
   924‚Üí   924‚Üí   will be populated by task 8 (content-folder writer) when it
   925‚Üí   925‚Üí   actually generates the 377-transformed twin of an OSRS node. The
   926‚Üí   926‚Üí   field exists in the schema now so task 8 doesn't need a schema
   927‚Üí   927‚Üí   migration.
   928‚Üí   928‚Üí
   929‚Üí   929‚Üí5. **`skins` flag on model nodes.** Per task 5-a's next-action hook,
   930‚Üí   930‚Üí   models with non-empty `triangleSkin`/`vertexSkin` arrays lose
   931‚Üí   931‚Üí   rig-skinning data when transformed to 377. The tracer records this
   932‚Üí   932‚Üí   as `skins: boolean` on model nodes so the pilot can prefer
   933‚Üí   933‚Üí   `skins === false` models when alternatives exist.
   934‚Üí   934‚Üí
   935‚Üí   935‚Üí6. **`secondaryRoots` for batched traces.** `traceMany()` keeps the
   936‚Üí   936‚Üí   single-`root` schema shape (backward compat with single-NPC
   937‚Üí   937‚Üí   traces) by making the first NPC the root and listing any
   938‚Üí   938‚Üí   additional NPCs in `secondaryRoots: DepRef[]`. The field is
   939‚Üí   939‚Üí   omitted for single-NPC traces (the common case).
   940‚Üí   940‚Üí
   941‚Üí   941‚Üí7. **`ObjType` deep walk is stubbed.** Per task spec: "for now, treat
   942‚Üí   942‚Üí   obj as a partial node and log 'TODO: deep obj walk not
   943‚Üí   943‚Üí   implemented'". The `visitObj` method records the obj as a leaf
   944‚Üí   944‚Üí   with `note: 'TODO: deep obj walk not implemented (OsrsObjType not
   945‚Üí   945‚Üí   ported)'`. When task 8 ports the obj decoder, this method should
   946‚Üí   946‚Üí   walk `obj.model`, `obj.manwear`, `obj.womanwear`, `obj.manhead`,
   947‚Üí   947‚Üí   `obj.womanhead`, `obj.countobj[]`, `obj.recol_s/d`, and
   948‚Üí   948‚Üí   `obj.params` (via `walkParams`).
   949‚Üí   949‚Üí
   950‚Üí   950‚Üí8. **`ParamType` lookup falls back to the legacy 377 registry.** When
   951‚Üí   951‚Üí   `CacheReader.readParam(id)` returns null (the common case until
   952‚Üí   952‚Üí   `OsrsParamType` is ported), the tracer tries `ParamType.get(id)`
   953‚Üí   953‚Üí   (the legacy 377 static registry). If both miss, the param is
   954‚Üí   954‚Üí   recorded with `note: 'param type unknown ‚Äî value not walked
   955‚Üí   955‚Üí   (OsrsParamType not ported)'` and its value is NOT walked (the
   956‚Üí   956‚Üí   importer will bring the param's value verbatim, but the tracer
   957‚Üí   957‚Üí   can't tell what asset kind it points at).
   958‚Üí   958‚Üí
   959‚Üí   959‚Üí9. **Script scanning is best-effort regex.** LostCity runescript
   960‚Üí   960‚Üí   references assets by debugname in call sites like
   961‚Üí   961‚Üí   `npc_anim(dragon_attack, 0)`, `spotanim_npc(firebreath_attack, 92, 0)`,
   962‚Üí   962‚Üí   `inv_total(worn, antidragonbreathshield)`, `sound_synth(dragonbreath, 0, 30)`,
   963‚Üí   963‚Üí   `npc_param(attack_sound)`. The tracer scans for these patterns and
   964‚Üí   964‚Üí   records the discovered names as string-ID dep refs (all marked
   965‚Üí   965‚Üí   `missing: true` until task 8 resolves them via pack files). The
   966‚Üí   966‚Üí   scan is intentionally conservative ‚Äî false positives would be
   967‚Üí   967‚Üí   worse than false negatives here (the importer would try to fetch
   968‚Üí   968‚Üí   non-existent assets).
   969‚Üí   969‚Üí
   970‚Üí   970‚Üí10. **Script discovery for an NPC uses TWO heuristics:**
   971‚Üí   971‚Üí    (a) look for `content/scripts/npc/scripts/<npcName>.rs2` (primary
   972‚Üí   972‚Üí    combat script convention); (b) scan all other `.rs2` files for
   973‚Üí   973‚Üí    `[<trigger>,<npcName>]` dispatch directives (e.g.
   974‚Üí   974‚Üí    `[ai_applayer2,babydragon] @dragon_ai_applayer2(false);`) and
   975‚Üí   975‚Üí    register each matching file as a separate script dep with
   976‚Üí   976‚Üí    `via: 'dispatch:<trigger>'`. The dispatch-scan is O(files √ó lines)
   977‚Üí   977‚Üí    but runs only once per NPC and only over the `scripts/npc/scripts/`
   978‚Üí   978‚Üí    subdir.

  1154‚Üí  1154‚Üí### JS5 archive ‚Üí content mapping assumed
  1155‚Üí  1155‚Üí
  1156‚Üí  1156‚ÜíThe OSRS cache uses this layout (post-rev ~150 era, stable across modern
  1157‚Üí  1157‚Üídumps). The reader hard-codes these constants in
  1158‚Üí  1158‚Üí`OsrsCacheAssetReader.ts`:
  1159‚Üí  1159‚Üí
  1160‚Üí  1160‚Üí| Archive | Contents | Used by |
  1161‚Üí  1161‚Üí|---------|----------|---------|
  1162‚Üí  1162‚Üí| 0       | Animations (frames AND skeletons, disjoint file IDs) | `readAnim` + `readAnimBase` |
  1163‚Üí  1163‚Üí| 1       | Models (one `.ob2` per file ID) | `readModel` |
  1164‚Üí  1164‚Üí| 2       | Music ‚Äî older OSRS dumps use this for config (fallback) | `ensureConfigChildren` fallback |
  1165‚Üí  1165‚Üí| 5       | Textures (older) ‚Äî fallback | `readTexture` fallback |
  1166‚Üí  1166‚Üí| 9       | Configs (multi-child bundle: `npc.dat`, `seq.dat`, etc.) | `readNpc`/`readSeq`/etc. (primary) |
  1167‚Üí  1167‚Üí| 11      | Textures (modern) | `readTexture` primary |
  1168‚Üí  1168‚Üí| 25      | Audio (synth sound effects) | `readSound` (stub ‚Äî not yet read) |
  1169‚Üí  1169‚Üí
  1170‚Üí  1170‚ÜíThe config bundle (`npc.dat`/`seq.dat`/`obj.dat`/etc.) is loaded from
  1171‚Üí  1171‚Üíarchive 9 by default, with a fallback to archive 2 for older cache
  1172‚Üí  1172‚Üídumps. Child names are resolved via the master index's per-child name
  1173‚Üí  1173‚Üíhashes (using the same `genHash()` algorithm as `Jagfile.ts`). If the
  1174‚Üí  1174‚Üímaster index doesn't have name hashes, a sequential fallback is used.
  1175‚Üí  1175‚Üí
  1176‚Üí  1176‚Üí### BZIP2 path notes
  1177‚Üí  1177‚Üí
  1178‚Üí  1178‚ÜíThe OSRS BZIP2 path uses the existing `BZip2` wasm wrapper at
  1179‚Üí  1179‚Üí`engine/src/io/BZip2.ts` (already used by `Jagfile.ts`). OSRS strips
  1180‚Üí  1180‚Üíthe leading `BZh1` magic from BZIP2-compressed container payloads; we
  1181‚Üí  1181‚Üírestore it via `BZip2.decompress(payload, uncompressedLength,
  1182‚Üí  1182‚ÜíprependHeader=true)`. No external `bunzip2` calls needed.
  1183‚Üí  1183‚Üí
  1184‚Üí  1184‚ÜíGZIP path uses the existing `decompressGz` from `engine/src/io/GZip.ts`.

  1186‚Üí  1186‚Üí### Limitations / stubs
  1187‚Üí  1187‚Üí
  1188‚Üí  1188‚Üí1. **`readObj`** returns a STUB `ObjType(id)` with only the ID populated
  1189‚Üí  1189‚Üí   ‚Äî the legacy `ObjType.decode()` calls `printFatalError` (which calls
  1190‚Üí  1190‚Üí   `process.exit(1)`) on unrecognized opcodes, and OSRS obj configs use
  1191‚Üí  1191‚Üí   opcodes the 377 decoder doesn't know. The dep tracer treats objs as
  1192‚Üí  1192‚Üí   leaves anyway (Task 6 worklog item 7). Swap to a real decode once
  1193‚Üí  1193‚Üí   `OsrsObjType` is ported (Task 8+).
  1194‚Üí  1194‚Üí
  1195‚Üí  1195‚Üí2. **`readParam` / `readStruct`** return null ‚Äî the legacy decoders
  1196‚Üí  1196‚Üí   `throw` on unrecognized opcodes (OSRS uses new codes). The dep
  1197‚Üí  1197‚Üí   tracer falls back to the legacy 377 `ParamType.get(id)` /
  1198‚Üí  1198‚Üí   `StructType.get(id)` static registries when these return null,
  1199‚Üí  1199‚Üí   which is the documented behavior in the Task 6 worklog.
  1200‚Üí  1200‚Üí
  1201‚Üí  1201‚Üí3. **`readParticle` / `readSound`** return null ‚Äî OSRS particle systems
  1202‚Üí  1202‚Üí   and sound formats are complex and undocumented. The dep tracer
  1203‚Üí  1203‚Üí   records these as missing leaves, which is fine for the pilot.
  1204‚Üí  1204‚Üí
  1205‚Üí  1205‚Üí4. **`readAnimBase`** assumes skeletons live in archive 0 (the same
  1206‚Üí  1206‚Üí   archive as anim frames). This is correct for modern OSRS but may
  1207‚Üí  1207‚Üí   differ in some cache revisions. If the base isn't found at
  1208‚Üí  1208‚Üí   archive 0 file `id`, the method returns null.
  1209‚Üí  1209‚Üí
  1210‚Üí  1210‚Üí5. **Config bundle child-name resolution** uses the master index's
  1211‚Üí  1211‚Üí   per-child name hashes. If the master index was written with
  1212‚Üí  1212‚Üí   `named = false`, falls back to sequential child-ID assignment
  1213‚Üí  1213‚Üí   based on the standard OSRS child ordering
  1214‚Üí  1214‚Üí   (`npc.dat`, `npc.idx`, `seq.dat`, `seq.idx`, ...). This is
  1215‚Üí  1215‚Üí   best-effort and may need adjustment for unusual cache dumps.
  1216‚Üí  1216‚Üí
  1217‚Üí  1217‚Üí6. **Master index reading** assumes the master index is stored as a
  1218‚Üí  1218‚Üí   single compressed container at archive 255, file 0 (read via the
  1219‚Üí  1219‚Üí   normal sector-walking protocol). The sector's archiveType byte is
  1220‚Üí  1220‚Üí   NOT validated (only `id` and `part` are), to remain compatible
  1221‚Üí  1221‚Üí   with both the `archive + 1` convention (used for normal archives)
  1222‚Üí  1222‚Üí   and the `archiveType = 255` convention (sometimes used for the
  1223‚Üí  1223‚Üí   master index). This matches OpenRS2's behavior.

  1384‚Üí  1384‚Üí### WriterOptions design
  1385‚Üí  1385‚Üí
  1386‚Üí  1386‚Üí- **`dryRun`** (default `false`): when true, the writer walks the deps map,
  1387‚Üí  1387‚Üí  allocates IDs in memory, and produces an `ImportResult` showing what
  1388‚Üí  1388‚Üí  WOULD be written ‚Äî but does NOT touch the filesystem or any pack files.
  1389‚Üí  1389‚Üí  Every `WrittenEntry.path` is set to the sentinel string `'<dryrun>'`.
  1390‚Üí  1390‚Üí  Useful for previewing an import before committing it.
  1391‚Üí  1391‚Üí
  1392‚Üí  1392‚Üí- **`namePrefix`** (default `'osrs_'`): prepended to every sanitized
  1393‚Üí  1393‚Üí  debugname. Combined with the sanitized OSRS name/ID to form the final
  1394‚Üí  1394‚Üí  debugname (e.g. `osrs_tormented_demon`, `osrs_model_45000`).
  1395‚Üí  1395‚Üí
  1396‚Üí  1396‚Üí- **`groupFile`** (default `'osrs_imports'`): subfolder under
  1397‚Üí  1397‚Üí  `content/models/` where binary assets (.ob2, .anim, .base) are written,
  1398‚Üí  1398‚Üí  AND the basename of the `.npc` / `.seq` config files. Multiple NPCs
  1399‚Üí  1399‚Üí  imported in one batch share the same group file (their `[debugname]`
  1400‚Üí  1400‚Üí  blocks are appended to the same `.npc` / `.seq` file).
  1401‚Üí  1401‚Üí
  1402‚Üí  1402‚Üí- **`overwrite`** (default `false`): when false (the safe default), the
  1403‚Üí  1403‚Üí  writer NEVER overwrites existing files or pack entries ‚Äî collisions
  1404‚Üí  1404‚Üí  trigger a fresh debugname with a `_2`, `_3` suffix instead. When true,
  1405‚Üí  1405‚Üí  the writer REPLACES existing `[debugname]` blocks in config files and
  1406‚Üí  1406‚Üí  overwrites binary files with the same path. NOT recommended for normal
  1407‚Üí  1407‚Üí  use ‚Äî intended for re-importing after a sanitization rule change.
  1408‚Üí  1408‚Üí
  1409‚Üí  1409‚Üí### Sanitization rules
  1410‚Üí  1410‚Üí
  1411‚Üí  1411‚ÜíThe `sanitizeName(rawName, packType)` helper applies the following
  1412‚Üí  1412‚Üítransforms in order:
  1413‚Üí  1413‚Üí
  1414‚Üí  1414‚Üí1. **Lowercase** the entire string.
  1415‚Üí  1415‚Üí2. **Replace every char NOT in `[a-z0-9]` with `_`** ‚Äî spaces, hyphens,
  1416‚Üí  1416‚Üí   apostrophes, etc. all become underscores.
  1417‚Üí  1417‚Üí3. **Collapse runs of multiple `_`** into a single `_` (so `"a  b"` ‚Üí
  1418‚Üí  1418‚Üí   `"a_b"`, not `"a__b"`).
  1419‚Üí  1419‚Üí4. **Trim leading/trailing `_`** (so `"_foo_"` ‚Üí `"foo"`).
  1420‚Üí  1420‚Üí5. **Prefix** with `namePrefix` (default `osrs_`).
  1421‚Üí  1421‚Üí6. **Dedupe** against the target pack file AND a session-reserved set
  1422‚Üí  1422‚Üí   (in-memory, populated by prior `sanitizeName()` calls in the same
  1423‚Üí  1423‚Üí   writer lifetime). If the candidate name is already taken, append
  1424‚Üí  1424‚Üí   `_2`, `_3`, `_4`, ... until unique.
  1425‚Üí  1425‚Üí
  1426‚Üí  1426‚ÜíExamples:
  1427‚Üí  1427‚Üí- `"Tormented demon"` ‚Üí `"osrs_tormented_demon"`
  1428‚Üí  1428‚Üí- `"td_stand"` ‚Üí `"osrs_td_stand"`
  1429‚Üí  1429‚Üí- `"model_45000"` ‚Üí `"osrs_model_45000"`
  1430‚Üí  1430‚Üí- `"anim_2005"` (after `osrs_anim_2005` already exists in `anim.pack`)
  1431‚Üí  1431‚Üí  ‚Üí `"osrs_anim_2005_2"`
  1432‚Üí  1432‚Üí
  1433‚Üí  1433‚ÜíThe session-reserved set is critical for batch imports where two OSRS
  1434‚Üí  1434‚Üíassets sanitize to the same candidate name in the same run ‚Äî without
  1435‚Üí  1435‚Üíit, the second asset would silently overwrite the first's pack entry.
  1436‚Üí  1436‚Üí
  1437‚Üí  1437‚Üí### Idempotency approach
  1438‚Üí  1438‚Üí
  1439‚Üí  1439‚ÜíThe writer is idempotent at three layers:
  1440‚Üí  1440‚Üí
  1441‚Üí  1441‚Üí1. **Deps-map layer** (primary): every OSRS `DepNode` has a
  1442‚Üí  1442‚Üí   `transformedFrom` field (set by Task 8 per the schema). On each run,
  1443‚Üí  1443‚Üí   the writer's `bootstrapResolverFromDepsMap()` walks the input deps
  1444‚Üí  1444‚Üí   map and pre-populates `NameResolver` from any node with
  1445‚Üí  1445‚Üí   `transformedFrom !== null` (looking up the debugname from the
  1446‚Üí  1446‚Üí   corresponding pack file). Then `processNode()` checks the field
  1447‚Üí  1447‚Üí   directly: if set, the node is recorded as `skipped` with reason
  1448‚Üí  1448‚Üí   `already_imported (newId=N)` and NOT re-processed.
  1449‚Üí  1449‚Üí
  1450‚Üí  1450‚Üí2. **NameResolver layer** (cross-NPC): if two NPCs in the same batch
  1451‚Üí  1451‚Üí   reference the same OSRS asset (e.g. both use OSRS model 45000), the
  1452‚Üí  1452‚Üí   first NPC's `processNode()` call registers the asset in
  1453‚Üí  1453‚Üí   `NameResolver`. When the second NPC's `processNode()` runs for the
  1454‚Üí  1454‚Üí   same OSRS ID, it finds the existing entry in NameResolver, records
  1455‚Üí  1455‚Üí   it as `skipped` with reason `shared_with_prior_npc`, and updates the
  1456‚Üí  1456‚Üí   deps map's `transformedFrom` field to point at the existing new ID.
  1457‚Üí  1457‚Üí
  1458‚Üí  1458‚Üí3. **Config-file layer** (defensive): `appendConfigBlock()` scans the
  1459‚Üí  1459‚Üí   target `.npc` / `.seq` config file for an existing `[<debugname>]`
  1460‚Üí  1460‚Üí   header before appending. If found and `overwrite=false`, the append
  1461‚Üí  1461‚Üí   is skipped (with a warning). If found and `overwrite=true`, the
  1462‚Üí  1462‚Üí   existing block is stripped via `stripConfigBlock()` before the new
  1463‚Üí  1463‚Üí   block is appended. This protects against the case where a prior run
  1464‚Üí  1464‚Üí   wrote the config file but failed before persisting the updated
  1465‚Üí  1465‚Üí   deps.json (so the next run's deps-map layer doesn't know about the
  1466‚Üí  1466‚Üí   prior import).
  1467‚Üí  1467‚Üí
  1468‚Üí  1468‚ÜíThe CLI (`Import.ts`) persists the updated deps.json (with
  1469‚Üí  1469‚Üí`transformedFrom` filled in) to `content/deps/<npcName>.deps.json`
  1470‚Üí  1470‚Üíafter every successful non-dry-run import. Subsequent runs pick this
  1471‚Üí  1471‚Üífile up via `--deps=<path>` (or auto-trace + apply the persisted
  1472‚Üí  1472‚Üí`transformedFrom` from a prior run via the `--out=` round-trip).
  1473‚Üí  1473‚Üí
  1474‚Üí  1474‚Üí### Pack-file ID allocation strategy
  1475‚Üí  1475‚Üí
  1476‚Üí  1476‚Üí**Decision: `max(existing IDs) + 1` for every pack type.**
  1477‚Üí  1477‚Üí
  1478‚Üí  1478‚ÜíThe schema supports reserved ranges per pack type (e.g. "models
  1479‚Üí  1479‚Üí50000-59999 reserved for OSRS imports") but no ranges are reserved
  1480‚Üí  1480‚Üítoday. The `allocateId(packType)` helper loads the pack file, finds
  1481‚Üí  1481‚Üíthe maximum existing ID, and returns `max + 1` (or `0` for an empty
  1482‚Üí  1482‚Üípack, matching LostCity's convention where every existing pack starts
  1483‚Üí  1483‚Üíat ID 0).
  1484‚Üí  1484‚Üí
  1485‚Üí  1485‚ÜíRationale:
  1486‚Üí  1486‚Üí- Simplest strategy that preserves backwards-compat (existing IDs are
  1487‚Üí  1487‚Üí  never reused or overwritten).
  1488‚Üí  1488‚Üí- The LostCity content folder doesn't have a "free ID range" convention
  1489‚Üí  1489‚Üí  ‚Äî IDs are assigned sequentially as new assets are added.
  1490‚Üí  1490‚Üí- If a future task adds reserved ranges, `allocateId()` is the single
  1491‚Üí  1491‚Üí  function to change (per-pack-type strategy can be added without
  1492‚Üí  1492‚Üí  touching call sites).
  1493‚Üí  1493‚Üí
  1494‚Üí  1494‚Üí**Anims are registered in BOTH `anim.pack` AND `animset.pack`** (with
  1495‚Üí  1495‚Üíthe same ID and name). This matches the existing LostCity convention
  1496‚Üí  1496‚Üíwhere both packs share the same `id=name` mapping for every anim file:
  1497‚Üí  1497‚Üí- `anim.pack` (AnimPack) is consulted by `SeqConfig.parseSeqConfig()`
  1498‚Üí  1498‚Üí  when resolving `frameN=<animname>` refs in `.seq` config files.
  1499‚Üí  1499‚Üí- `animset.pack` (AnimSetPack) is consulted by
  1500‚Üí  1500‚Üí  `engine/tools/pack/graphics/pack.ts` when mapping `.anim` files to
  1501‚Üí  1501‚Üí  archive 2 file IDs during cache packing.
  1502‚Üí  1502‚Üí
  1503‚Üí  1503‚ÜíRegistering in only one would break either the seq config parser or
  1504‚Üí  1504‚Üíthe cache packer. The writer registers in both atomically (within the
  1505‚Üí  1505‚Üísame `updatePackFile()` call sequence, with both packs sharing the
  1506‚Üí  1506‚Üísame dirty/save batch).
  1507‚Üí  1507‚Üí
  1508‚Üí  1508‚Üí### NPC config line rewriting
  1509‚Üí  1509‚Üí
  1510‚Üí  1510‚Üí`OsrsNpcType.toLegacy377NpcConfig()` emits raw integer asset IDs in
  1511‚Üí  1511‚Üíits output (e.g. `model1=45000`, `readyanim=9200`,
  1512‚Üí  1512‚Üí`param=osrs_runanim,9102`). The writer's `rewriteNpcConfigLine()`
  1513‚Üí  1513‚Üípost-processes each line to rewrite those IDs to their newly-assigned
  1514‚Üí  1514‚Üídebugnames via `NameResolver`:
  1515‚Üí  1515‚Üí
  1516‚Üí  1516‚Üí| Line shape                              | Rewrite                                          |
  1517‚Üí  1517‚Üí|-----------------------------------------|--------------------------------------------------|
  1518‚Üí  1518‚Üí| `modelN=<id>`                           | `modelN=<modelDebugname>`                        |
  1519‚Üí  1519‚Üí| `headN=<id>`                            | `headN=<modelDebugname>`                         |
  1520‚Üí  1520‚Üí| `readyanim=<id>`                        | `readyanim=<seqDebugname>`                       |
  1521‚Üí  1521‚Üí| `walkanim=<id>[,<id>,<id>,<id>]`        | per-element rewrite to seq debugnames            |
  1522‚Üí  1522‚Üí| `param=osrs_runanim,<id>`               | `param=osrs_runanim,<seqDebugname>`              |
  1523‚Üí  1523‚Üí| `param=osrs_runanim_{l,r,b},<id>`       | `param=osrs_runanim_{l,r,b},<seqDebugname>`      |
  1524‚Üí  1524‚Üí| `param=osrs_crawlanim,<id>`             | `param=osrs_crawlanim,<seqDebugname>`            |
  1525‚Üí  1525‚Üí| `param=osrs_crawlanim_{l,r,b,b2},<id>`  | `param=osrs_crawlanim_{l,r,b,b2},<seqDebugname>` |
  1526‚Üí  1526‚Üí| `param=osrs_walkanim_b,<id>`            | `param=osrs_walkanim_b,<seqDebugname>`           |
  1527‚Üí  1527‚Üí
  1528‚Üí  1528‚ÜíLines whose OSRS ID can't be resolved via `NameResolver` (e.g. the
  1529‚Üí  1529‚Üítracer didn't walk that model) fall back to the `model_<id>` /
  1530‚Üí  1530‚Üí`seq_<id>` / `anim_<id>` literal form ‚Äî the LostCity config parser
  1531‚Üí  1531‚Üíaccepts both forms (it tries `getByName(value)` first, falling back to
  1532‚Üí  1532‚Üí`parseInt(value)` if the name lookup returns -1).
  1533‚Üí  1533‚Üí
  1534‚Üí  1534‚ÜíThe header line (`[<osrs_debugname>]`) is also replaced with
  1535‚Üí  1535‚Üí`[<sanitized_debugname>]` so the pack entry matches the config block
  1536‚Üí  1536‚Üíheader. (The OSRS emitter uses the OSRS debugname or `npc_<id>` as
  1537‚Üí  1537‚Üíthe header ‚Äî we override it with our `osrs_<name>` sanitized form.)
  1538‚Üí  1538‚Üí
  1539‚Üí  1539‚Üí### Seq config emission
  1540‚Üí  1540‚Üí
  1541‚Üí  1541‚Üí`writeSeq()` emits an ini-style `[debugname]` block per seq, mirroring
  1542‚Üí  1542‚Üíthe format of `content/scripts/areas/area_kalphite/configs/kalphite.seq`
  1543‚Üí  1543‚Üí(existing LostCity convention):
  1544‚Üí  1544‚Üí
  1545‚Üí  1545‚Üí```
  1546‚Üí  1546‚Üí[osrs_td_attack]

  1551‚Üí  1551‚Üí```
  1552‚Üí  1552‚Üí
  1553‚Üí  1553‚ÜíThe 377 shared fields (loops, walkmerge, stretches, priority,

  1558‚Üí  1558‚Üí`param=osrs_<field>,<value>` entries, matching the convention used by
  1559‚Üí  1559‚Üí`OsrsNpcType.toLegacy377NpcConfig()` for OSRS-only NPC fields.
  1560‚Üí  1560‚Üí
  1561‚Üí  1561‚ÜíFrame references (`frameN=<animId>`) and interpolated-frame refs
  1562‚Üí  1562‚Üí(`iframeN=<animId>`) are rewritten to their newly-assigned debugnames
  1563‚Üí  1563‚Üívia `NameResolver`, the same way NPC config lines are.

  1699‚Üí  1699‚ÜíApplied top-to-bottom (first hit wins):
  1700‚Üí  1700‚Üí
  1701‚Üí  1701‚Üí1. **Per-NPC player override** (`state.npcOverrides.get(npcId)`)
  1702‚Üí  1702‚Üí   - Honored UNCONDITIONALLY ‚Äî the resolver does NOT check whether the chosen variant is actually available. The rationale: the override represents the player's explicit preference; `resolveNpcConfigForPlayer` on the registry is responsible for gracefully falling back to whatever's actually loadable (e.g. a player who forced `'osrs'` on an NPC with no OSRS import will see the legacy model via the fallback, but the resolver still reports `'osrs'` so the UI can flag the mismatch).
  1703‚Üí  1703‚Üí
  1704‚Üí  1704‚Üí2. **Per-region player override** (`state.regionOverrides.get(regionId)`)
  1705‚Üí  1705‚Üí   - STUB: today the registry always passes `regionId = -1` (which never matches any entry), so this step is effectively a no-op. The spawn-side wiring that would populate "current player region" is deferred to a later task. The `resolveNpcVariantForRegion(npcId, playerId, regionId)` method exists on the registry so the spawn-site migration is a one-line change once the region lookup is wired.
  1706‚Üí  1706‚Üí
  1707‚Üí  1707‚Üí3. **Player era preset** (`state.eraPreset`)
  1708‚Üí  1708‚Üí   - Respects availability (unlike steps 1 and 2). The `resolveEraPreset` helper translates:
  1709‚Üí  1709‚Üí     - `'05era'`   ‚Üí `'legacy377'` (or `'osrs'` if the NPC has no legacy twin ‚Äî i.e. brand-new OSRS content).
  1710‚Üí  1710‚Üí     - `'07era'`   ‚Üí `'osrs'` if available, else `'legacy377'`. (Today equivalent to `'allOSRS'` because we don't track per-NPC "year added" ‚Äî see the `EraPreset` docstring in `PlayerVariantState.ts`.)
  1711‚Üí  1711‚Üí     - `'allOSRS'` ‚Üí `'osrs'` if available, else `'legacy377'`.
  1712‚Üí  1712‚Üí     - `'mixed'`   ‚Üí same as `'05era'` (the override map is checked first; this is the fallback).
  1713‚Üí  1713‚Üí
  1714‚Üí  1714‚Üí4. **Server default** (`'05era'` ‚Üí `'legacy377'`)
  1715‚Üí  1715‚Üí   - Defensive fallback ‚Äî should never be reached because `PlayerVariantState` always has a valid `eraPreset`.
  1716‚Üí  1716‚Üí

  1764‚Üí  1764‚Üí### Integration pattern with existing NpcType
  1765‚Üí  1765‚Üí
  1766‚Üí  1766‚ÜíThe variant registry is NOT a wrapper around `NpcType`. It's a side-channel: when an NPC is being spawned for a specific player, the spawn code calls `VariantRegistry.resolveNpcVariant(npcId, playerId)` to decide which config ID to use. The two paths are:
  1767‚Üí  1767‚Üí
  1768‚Üí  1768‚Üí- `'legacy377'` ‚Üí use the existing `NpcType.get(npcId)` path UNCHANGED. This is the default for every NPC that hasn't been imported.
  1769‚Üí  1769‚Üí- `'osrs'`       ‚Üí use `NpcType.get(osrsImportedNpcId)` ‚Äî the imported NPC's new 377 pack-file ID. The imported NPC config has its own `models[]` array pointing at the OSRS-imported models' new 377 IDs ‚Äî so no per-model swap is needed at the spawn site.
  1770‚Üí  1770‚Üí
  1771‚Üí  1771‚ÜíA single helper does the lookup:
  1772‚Üí  1772‚Üí```ts
  1773‚Üí  1773‚ÜíVariantRegistry.getInstance().resolveNpcConfigForPlayer(npcId, playerId): NpcType
  1774‚Üí  1774‚Üí```
  1775‚Üí  1775‚Üí
  1776‚Üí  1776‚ÜíReturns the legacy NpcType (`NpcType.get(npcId)`) or the OSRS-imported NpcType (`NpcType.get(osrsImportedNpcId)`). Future engine work can migrate NPC spawn sites one at a time to call this helper instead of `NpcType.get(npcId)` directly. Until a spawn site is migrated, the existing `NpcType.get(npcId)` path continues to work unchanged ‚Äî the variant system is invisible to it. The registry does NOT modify `NpcType.configs[]` or any other existing static registry; it only consults them (for the `legacyModelOwner` reverse index used by `resolveModelVariant`).
  1777‚Üí  1777‚Üí
  1778‚Üí  1778‚ÜíThe spawn-site migration is intentionally deferred ‚Äî touching every `NpcType.get(...)` call in the codebase (7 files: `World.ts`, `entity/Npc.ts`, `entity/Player.ts`, `GameMap.ts`, `script/ScriptIterators.ts`, `script/handlers/NpcOps.ts`, `script/handlers/PlayerOps.ts`) is too invasive for one task. The helper is in place; the migration can happen one spawn site at a time.

  1780‚Üí  1780‚Üí### Per-player state caching
  1781‚Üí  1781‚Üí
  1782‚Üí  1782‚Üí`resolveNpcVariant(npcId, playerId)` is SYNC (the spawn path is hot). The per-player `PlayerVariantState` is loaded asynchronously by `VariantPersistence.loadPlayerVariantState(playerId)` at login time and stashed in the registry's `playerStateCache` Map. The login handler is expected to call:
  1783‚Üí  1783‚Üí```ts
  1784‚Üí  1784‚Üíconst state = await loadPlayerVariantState(playerId);
  1785‚Üí  1785‚ÜíVariantRegistry.getInstance().setPlayerState(playerId, state);
  1786‚Üí  1786‚Üí```
  1787‚Üí  1787‚ÜíOn logout: `VariantRegistry.getInstance().clearPlayerState(playerId)`.
  1788‚Üí  1788‚Üí
  1789‚Üí  1789‚ÜíIf `resolveNpcVariant` is called for a playerId that's not in the cache (e.g. before the login handler has warmed it), the registry returns the server default (`'legacy377'` for everything with a legacy twin, `'osrs'` for brand-new OSRS NPCs). A warning is logged so the developer knows the cache wasn't warmed ‚Äî but the engine doesn't crash.

  1823‚Üí  1823‚Üí### What's stubbed for the UI (later task)
  1824‚Üí  1824‚Üí
  1825‚Üí  1825‚Üí1. **Start-screen configuration UI** ‚Äî not built. The `PlayerVariantState` API (`setEra`, `setNpcOverride`, `clearNpcOverride`, `clearAll`) is the contract the UI will call. The `VariantRegistry.getAvailableVariants(npcId)` and `getVariantMetadata(npcId)` methods provide the data the UI needs to render the picker (which NPCs have OSRS variants, what their debugnames are, when they were imported).
  1826‚Üí  1826‚Üí
  1827‚Üí  1827‚Üí2. **Per-region/town override** ‚Äî interface only, no implementation. The `PlayerVariantState.regionOverrides` map and `setRegionOverride`/`clearRegionOverride` methods exist; the resolver consults the map (step 2 of the precedence). But the spawn-side wiring that would pass the player's current region ID is not in place ‚Äî `VariantRegistry.resolveNpcVariant` always passes `regionId = -1` today. The `resolveNpcVariantForRegion(npcId, playerId, regionId)` method exists for the future spawn-site migration.
  1828‚Üí  1828‚Üí
  1829‚Üí  1829‚Üí3. **Spawn-site migration** ‚Äî the `resolveNpcConfigForPlayer(npcId, playerId)` helper is in place but no spawn sites have been migrated to call it. The 7 files that call `NpcType.get(...)` directly (`World.ts`, `entity/Npc.ts`, `entity/Player.ts`, `GameMap.ts`, `script/ScriptIterators.ts`, `script/handlers/NpcOps.ts`, `script/handlers/PlayerOps.ts`) continue to use the legacy path. This is intentional ‚Äî the system is opt-in, and migrating one spawn site at a time is the safer path.
  1830‚Üí  1830‚Üí
  1831‚Üí  1831‚Üí4. **`resolveModelVariant` per-model swap** ‚Äî uses a parallel-array heuristic (legacy NPC has N models, OSRS NPC has N models ‚Üí pair by index). If the counts differ, NO per-model swap is done (the legacy model ID is returned unchanged). The caller should use `resolveNpcConfigForPlayer` instead, which swaps the entire NPC config (and thus the entire `models[]` array). This limitation is documented in the `VariantRegistry.resolveModelVariant` JSDoc.
  1832‚Üí  1832‚Üí
  1833‚Üí  1833‚Üí5. **`'07era'` preset semantic** ‚Äî today equivalent to `'allOSRS'` because we don't track per-NPC "year added" metadata. When more OSRS imports land, the registry can distinguish "added in 2007+" NPCs via a future `addedAt` field on the variant metadata, and `'07era'` can be tightened to use OSRS only for those.
