    67Ôćĺ    67Ôćĺ
    68Ôćĺ    68Ôćĺ### Format diff: 377 vs OSRS (key for the port)
    69Ôćĺ    69ÔćĺThe OSRS model format is a **strict superset** of the 377 format:
    70Ôćĺ    70Ôćĺ- Header trailer is 18 bytes in 377, **23 bytes** in OSRS (extra 5 bytes:
    71Ôćĺ    71Ôćĺ  `version`, `textureUVCount`, `triangleSkinCount`, `vertexSkinCount`,
    72Ôćĺ    72Ôćĺ  `maxDepth`).
    73Ôćĺ    73Ôćĺ- OSRS adds: per-vertex skin data, per-triangle skin data, textured UV coords,
    74Ôćĺ    74Ôćĺ  `version` byte (used as a feature gate).
    75Ôćĺ    75Ôćĺ- Old models decode fine under OSRS rules by treating the new fields as
    76Ôćĺ    76Ôćĺ  zero-length when their flag bytes are 0.
    77Ôćĺ    77Ôćĺ- Animations changed more substantially: OSRS anim frames use a different
    78Ôćĺ    78Ôćĺ  layout (no shared `head` block ÔÇö each frame is self-contained).
    79Ôćĺ    79Ôćĺ
    80Ôćĺ    80Ôćĺ### Pack-list format (hand-maintained today)
    81Ôćĺ    81ÔćĺPlain text, one entry per line: `0=hans\n1=man\n...`. Generated/validated by
    82Ôćĺ    82Ôćĺ`PackFile.save()`. We will extend, not replace.

   132Ôćĺ   132Ôćĺ## Task 5-a ÔÇö Port OSRS ModelDefinition decoder to TS
   133Ôćĺ   133Ôćĺ**Agent:** OsrsModelPorter
   134Ôćĺ   134Ôćĺ**Status:** done
   135Ôćĺ   135Ôćĺ**Files created** (both under `engine/src/cache/osrs/`):
   136Ôćĺ   136Ôćĺ- `OsrsModel.ts` (~822 lines incl. extensive comments)
   137Ôćĺ   137Ôćĺ- `index.ts` (barrel, 2 lines, re-exports default + `OsrsModelMetadata`)
   138Ôćĺ   138Ôćĺ
   139Ôćĺ   139Ôćĺ### Summary
   140Ôćĺ   140ÔćĺPorted RuneLite's `ModelDefinition.java` to TypeScript as `OsrsModel`, following
   141Ôćĺ   141Ôćĺthe LostCity conventions used by the existing 377 decoder at
   142Ôćĺ   142Ôćĺ`engine/src/cache/graphics/Model.ts`. Output uses `Packet` from `#/io/Packet.js`
   143Ôćĺ   143Ôćĺfor all byte reading, `printWarning` from `#/util/Logger.js` for warnings, and
   144Ôćĺ   144Ôćĺthe same 4-space / single-quote / semi style as the rest of the engine.
   145Ôćĺ   145Ôćĺ
   146Ôćĺ   146Ôćĺ### Public API
   147Ôćĺ   147Ôćĺ```ts
   148Ôćĺ   148Ôćĺclass OsrsModel {
   149Ôćĺ   149Ôćĺ  static decode(id: number, data: Uint8Array | null): OsrsModel | null;
   150Ôćĺ   150Ôćĺ  static detect(data: Uint8Array): 'osrs' | 'legacy377';
   151Ôćĺ   151Ôćĺ  toLegacy377(): Uint8Array;
   152Ôćĺ   152Ôćĺ  // plus public decoded fields (typed arrays) and header counts
   153Ôćĺ   153Ôćĺ}
   154Ôćĺ   154Ôćĺ
   155Ôćĺ   155Ôćĺclass OsrsModelMetadata {
   156Ôćĺ   156Ôćĺ  // all trailer fields + computed section offsets
   157Ôćĺ   157Ôćĺ}
   158Ôćĺ   158Ôćĺ```
   159Ôćĺ   159Ôćĺ
   160Ôćĺ   160ÔćĺThe class is **API-compatible in spirit** with the existing 377 `Model` class
   161Ôćĺ   161Ôćĺ(sibling `Metadata` class + static `decode()`), but the OSRS decoder is
   162Ôćĺ   162Ôćĺone-shot (no separate `unpack`/`fromId` split) ÔÇö it is intended for pipeline
   163Ôćĺ   163Ôćĺuse (decode Ôćĺ transform Ôćĺ pack), not for runtime client caching. The 377
   164Ôćĺ   164Ôćĺdecoder's two-stage split was an optimisation for the live game server; the
   165Ôćĺ   165ÔćĺOSRS pipeline doesn't need it.
   166Ôćĺ   166Ôćĺ
   167Ôćĺ   167Ôćĺ### Spec ambiguities resolved
   168Ôćĺ   168Ôćĺ
   169Ôćĺ   169Ôćĺ1. **Trailer byte count.** The task spec listed 12 trailer fields (17 bytes)
   170Ôćĺ   170Ôćĺ   but asserted the trailer is 23 bytes. Resolved by combining the spec list
   171Ôćĺ   171Ôćĺ   with the worklog note ("extra 5 bytes: `version`, `textureUVCount`,
   172Ôćĺ   172Ôćĺ   `triangleSkinCount`, `vertexSkinCount`, `maxDepth`") and keeping every 377
   173Ôćĺ   173Ôćĺ   field (including `priority`, which the spec omitted). Final layout = 17
   174Ôćĺ   174Ôćĺ   fields, 23 bytes total. The 5 NEW bytes are: `version`,
   175Ôćĺ   175Ôćĺ   `hasFaceTextures`, `triangleSkinCount`, `vertexSkinCount`, `maxDepth`
   176Ôćĺ   176Ôćĺ   (all u8). `textureUVCount` from the worklog is interpreted as the same
   177Ôćĺ   177Ôćĺ   field as `triangleTextureCount` (the count of textured triangles that
   178Ôćĺ   178Ôćĺ   carry UVs).
   179Ôćĺ   179Ôćĺ
   180Ôćĺ   180Ôćĺ2. **Triangle orientation/type byte.** The spec lists section 11
   181Ôćĺ   181Ôćĺ   ("triangle vertex indices") as using the "same 1/2/3/4 type encoding as
   182Ôćĺ   182Ôćĺ   377" but doesn't list a separate orientation section. In 377 the
   183Ôćĺ   183Ôćĺ   orientation byte lives in its own 1-byte-per-face section read BEFORE
   184Ôćĺ   184Ôćĺ   the index deltas. Resolved by adding a triangle-orientation section
   185Ôćĺ   185Ôćĺ   (1 byte per triangle) at body position 5.5 (between vertex labels and
   186Ôćĺ   186Ôćĺ   triangle colours) ÔÇö the same role as 377's `faceOrientationsOffset`.
   187Ôćĺ   187Ôćĺ   The body section order comment in the source makes this explicit.
   188Ôćĺ   188Ôćĺ
   189Ôćĺ   189Ôćĺ3. **`priority` byte.** 377 has a `priority` byte (255 = per-face priority
   190Ôćĺ   190Ôćĺ   array; otherwise shared priority value). The OSRS spec body section
   191Ôćĺ   191Ôćĺ   list has no priority section, but dropping the byte entirely would lose
   192Ôćĺ   192Ôćĺ   information needed for the 377 backward-compat path. Kept the byte in
   193Ôćĺ   193Ôćĺ   the OSRS trailer at the same trailer position as 377 (between
   194Ôćĺ   194Ôćĺ   `hasFaceTextures` and `hasFaceAlphas`).
   195Ôćĺ   195Ôćĺ
   196Ôćĺ   196Ôćĺ4. **Skin section encoding.** Spec body sections 13/14 say "only if
   197Ôćĺ   197Ôćĺ   `version >= 1` AND skinCount > 0" but don't say how each skin entry is
   198Ôćĺ   198Ôćĺ   encoded. Chose u8 per entry (matching the modern RuneLite decoder).
   199Ôćĺ   199Ôćĺ   If a future cache uses smart-encoded skin entries, this is the single
   200Ôćĺ   200Ôćĺ   line to change.
   201Ôćĺ   201Ôćĺ
   202Ôćĺ   202Ôćĺ5. **Detect heuristic.** `detect()` sniffs by reading the version byte at
   203Ôćĺ   203Ôćĺ   `data.length - 23`, then validates vertexCount/triangleCount/dataLen
   204Ôćĺ   204Ôćĺ   fields against the blob size. Falls back to `'legacy377'` on any
   205Ôćĺ   205Ôćĺ   sanity failure ÔÇö the legacy decoder will then either succeed or report
   206Ôćĺ   206Ôćĺ   its own error. The safe default prevents false-positive OSRS
   207Ôćĺ   207Ôćĺ   classification of malformed 377 blobs.
   208Ôćĺ   208Ôćĺ
   209Ôćĺ   209Ôćĺ### `toLegacy377()` transform ÔÇö limitations
   210Ôćĺ   210Ôćĺ
   211Ôćĺ   211ÔćĺThe transform emits a faithful 377 `.ob2` blob that the existing Java client
   212Ôćĺ   212Ôćĺcan render. The following OSRS-only data is **dropped**:
   213Ôćĺ   213Ôćĺ
   214Ôćĺ   214Ôćĺ| OSRS field             | 377 equivalent                | Impact |
   215Ôćĺ   215Ôćĺ|------------------------|-------------------------------|--------|
   216Ôćĺ   216Ôćĺ| `triangleMaterial`     | none                          | Faces with material textures fall back to flat colour |
   217Ôćĺ   217Ôćĺ| `triangleSkin`         | none                          | Rig-based triangle skinning lost |
   218Ôćĺ   218Ôćĺ| `vertexSkin`           | none                          | Rig-based vertex skinning lost |
   219Ôćĺ   219Ôćĺ| `maxDepth`             | none                          | Bounding-box hint lost (client recomputes) |
   220Ôćĺ   220Ôćĺ
   221Ôćĺ   221ÔćĺThe dependency tracer (task 6) MUST flag models that have non-zero
   222Ôćĺ   222Ôćĺ`triangleSkin`/`vertexSkin` arrays, because animations targeting those rigs
   223Ôćĺ   223Ôćĺwill not play correctly when the model is rendered as a 377 blob. The pilot
   224Ôćĺ   224ÔćĺNPC (Tormented Demon) selection should prefer models with `triangleSkinCount
   225Ôćĺ   225Ôćĺ== 0 && vertexSkinCount == 0` if any are available.
   226Ôćĺ   226Ôćĺ
   227Ôćĺ   227ÔćĺRound-trip verified by smoke test:
   228Ôćĺ   228Ôćĺ  OSRS bytes Ôćĺ `OsrsModel.decode()` Ôćĺ `toLegacy377()` Ôćĺ `Model.unpack()`
   229Ôćĺ   229Ôćĺ  Ôćĺ `Model.fromId()` ÔÇö all vertex positions, triangle indices, colours,
   230Ôćĺ   230Ôćĺ  infos, alphas, tags, vertex labels, and per-face/shared priorities
   231Ôćĺ   231Ôćĺ  match exactly. Textured UV vertex indices also survive (377 has the
   232Ôćĺ   232Ôćĺ  same field shape ÔÇö 3 u16 vertex indices per textured triangle).

   329Ôćĺ   329Ôćĺ### Key API decisions
   330Ôćĺ   330Ôćĺ1. **`OsrsAnimFrame` is a sibling class, not `extends AnimFrame`.**
   331Ôćĺ   331Ôćĺ   The 377 `AnimFrame` is heavily tied to its 4-section `unpack()` static
   332Ôćĺ   332Ôćĺ   loader and the `AnimFrame.instances[]` / `AnimFrame.order[]` registries
   333Ôćĺ   333Ôćĺ   managed by `OnDemand.cache`. Subclassing would inherit those statics
   334Ôćĺ   334Ôćĺ   verbatim and route OSRS frames into the live 377 registry ÔÇö exactly
   335Ôćĺ   335Ôćĺ   what the modular pipeline wants to avoid. A sibling class with matching
   336Ôćĺ   336Ôćĺ   field shape gives polymorphism without unwanted registry coupling.
   337Ôćĺ   337Ôćĺ   This mirrors the sibling-class pattern used by `OsrsModel` (task 5-a).
   338Ôćĺ   338Ôćĺ
   339Ôćĺ   339Ôćĺ2. **`OsrsSeqType` is a sibling class extending `ConfigType` directly**, not
   340Ôćĺ   340Ôćĺ   `extends SeqType`. Same reasoning as above ÔÇö `SeqType` has private
   341Ôćĺ   341Ôćĺ   static `configs[]` / `configNames` and bound `load/parse/get` methods
   342Ôćĺ   342Ôćĺ   that would route OSRS seqs into the shared 377 registry.
   343Ôćĺ   343Ôćĺ
   344Ôćĺ   344Ôćĺ3. **`OsrsAnimFrame.decode()` takes `baseId` as a parameter** even though
   345Ôćĺ   345Ôćĺ   the trailing 2 bytes of the blob also encode it. We cross-check and
   346Ôćĺ   346Ôćĺ   warn on mismatch but trust the parameter ÔÇö the caller is expected to
   347Ôćĺ   347Ôćĺ   have resolved the base ID through the cache index, and the trailing
   348Ôćĺ   348Ôćĺ   bytes are a redundant integrity check (sometimes zero in OSRS caches
   349Ôćĺ   349Ôćĺ   that store the base reference in a separate index file).
   350Ôćĺ   350Ôćĺ
   351Ôćĺ   351Ôćĺ4. **`frameId` defaults to 0** in `toLegacy377()` because the OSRS blob
   352Ôćĺ   352Ôćĺ   doesn't carry a frame ID. The modular packer is expected to assign a
   353Ôćĺ   353Ôćĺ   real ID (and rewrite it into the head section if needed) before the
   354Ôćĺ   354Ôćĺ   blob reaches the live client.
   355Ôćĺ   355Ôćĺ
   356Ôćĺ   356Ôćĺ5. **`OsrsSeqType.decode()` throws on codes 16+** (sub-frame references).
   357Ôćĺ   357Ôćĺ   The task spec describes them only as "16+ for sub-frame references"
   358Ôćĺ   358Ôćĺ   without a layout. Throwing lets the dep tracer (task 6) flag any seq
   359Ôćĺ   359Ôćĺ   that uses them, rather than silently dropping data.
   360Ôćĺ   360Ôćĺ
   361Ôćĺ   361Ôćĺ### Format ambiguities resolved
   362Ôćĺ   362Ôćĺ
   363Ôćĺ   363Ôćĺ1. **"For groups with no flag, the previous group's value carries forward
   364Ôćĺ   364Ôćĺ   (or 0 for translate, 128 for scale ÔÇö same default-value rule as 377)"**
   365Ôćĺ   365Ôćĺ   ÔÇö interpreted as: use the SAME default-value rule as the 377 decoder
   366Ôćĺ   366Ôćĺ   (0 for OP_TRANSLATE/OP_ROTATE/OP_ALPHA, 128 for OP_SCALE). The
   367Ôćĺ   367Ôćĺ   "carry forward" phrasing in the spec was a description of how the
   368Ôćĺ   368Ôćĺ   default value doesn't change between adjacent same-bone-type groups,
   369Ôćĺ   369Ôćĺ   not a request to track a per-axis running last-value. The synthetic
   370Ôćĺ   370Ôćĺ   OP_BASE pull-forward walk from `AnimFrame.unpack()` is preserved
   371Ôćĺ   371Ôćĺ   unchanged ÔÇö it emits a zero-value entry for the most recent OP_BASE
   372Ôćĺ   372Ôćĺ   bone when a non-OP_BASE bone has a non-zero flag.
   373Ôćĺ   373Ôćĺ
   374Ôćĺ   374Ôćĺ2. **OSRS body layout is INTERLEAVED, not sectioned.** The 377 decoder
   375Ôćĺ   375Ôćĺ   reads flags from a separate `tran1` section and smart values from a
   376Ôćĺ   376Ôćĺ   separate `tran2` section. The OSRS spec describes "For each group i:
   377Ôćĺ   377Ôćĺ   read typeFlag, then read axis values for set bits" ÔÇö i.e., the flag
   378Ôćĺ   378Ôćĺ   byte and the smart values are read sequentially from a single
   379Ôćĺ   379Ôćĺ   section. This matches RuneLite's `FrameDefinition` decoder. Verified
   380Ôćĺ   380Ôćĺ   by round-trip smoke test below.
   381Ôćĺ   381Ôćĺ
   382Ôćĺ   382Ôćĺ3. **Optional interpolation tail.** RuneLite's `FrameDefinition` reads
   383Ôćĺ   383Ôćĺ   (after the delay byte) an optional `interpolationType: u8`, then
   384Ôćĺ   384Ôćĺ   (if 4 more bytes are present) `leftHandItem: u16` and
   385Ôćĺ   385Ôćĺ   `rightHandItem: u16`. The task spec calls these `leftHandWeight` /
   386Ôćĺ   386Ôćĺ   `rightHandWeight` ÔÇö I followed the spec's naming but kept RuneLite's
   387Ôćĺ   387Ôćĺ   read layout (1 byte if any remain, then 4 bytes if at least 5 remain
   388Ôćĺ   388Ôćĺ   total). If a future cache uses a different layout, this is the
   389Ôćĺ   389Ôćĺ   single block to revisit.
   390Ôćĺ   390Ôćĺ
   391Ôćĺ   391Ôćĺ4. **Opcode 12 (`field790` in the Java client).** The 377 TS
   392Ôćĺ   392Ôćĺ   `SeqType.decode()` does NOT handle code 12 (it throws). The Java
   393Ôćĺ   393Ôćĺ   client does, reading a `u32`. The OSRS spec doesn't mention code 12
   394Ôćĺ   394Ôćĺ   explicitly, but since OSRS is a superset of the 377 client opcode
   395Ôćĺ   395Ôćĺ   set, code 12 should be handled. I added it as `field12: u32` with a
   396Ôćĺ   396Ôćĺ   comment documenting it as an opaque field carried over from the
   397Ôćĺ   397Ôćĺ   Java client (`field790`) ÔÇö its semantic meaning isn't documented
   398Ôćĺ   398Ôćĺ   upstream.
   399Ôćĺ   399Ôćĺ
   400Ôćĺ   400Ôćĺ5. **`AnimBase.instances[]` is reused.** The OSRS skeleton format is
   401Ôćĺ   401Ôćĺ   byte-identical to 377 for pre-HD content (same `length u8`, `types
   402Ôćĺ   402Ôćĺ   u8[]`, `labels u8[][]` layout). Rather than duplicate the loader,
   403Ôćĺ   403Ôćĺ   `OsrsAnimFrame` references the existing `AnimBase.instances[]`
   404Ôćĺ   404Ôćĺ   registry. OSRS-only pipelines that don't load 377 anims must still
   405Ôćĺ   405Ôćĺ   call `AnimBase.unpack()` on OSRS base files to populate this
   406Ôćĺ   406Ôćĺ   registry before invoking `OsrsAnimFrame.decode()`. (This is the
   407Ôćĺ   407Ôćĺ   task 6 dep tracer's responsibility.)
   408Ôćĺ   408Ôćĺ
   409Ôćĺ   409Ôćĺ### `toLegacy377()` transform ÔÇö limitations
   410Ôćĺ   410Ôćĺ
   411Ôćĺ   411ÔćĺThe transform emits a faithful 1-frame 377 anim blob that the existing
   412Ôćĺ   412Ôćĺ377 client can render via `AnimFrame.unpack()`. The following OSRS-only
   413Ôćĺ   413Ôćĺdata is **dropped**:
   414Ôćĺ   414Ôćĺ
   415Ôćĺ   415Ôćĺ| OSRS field             | 377 equivalent                | Impact |
   416Ôćĺ   416Ôćĺ|------------------------|-------------------------------|--------|
   417Ôćĺ   417Ôćĺ| `interpolationType`    | none                          | Blended/interpolated animations fall back to standard single-frame playback |
   418Ôćĺ   418Ôćĺ| `leftHandWeight`       | none                          | Left-hand blend weight lost |
   419Ôćĺ   419Ôćĺ| `rightHandWeight`      | none                          | Right-hand blend weight lost |
   420Ôćĺ   420Ôćĺ
   421Ôćĺ   421ÔćĺThe `frameId` field is REQUIRED by 377 (the head section stores it as a
   422Ôćĺ   422Ôćĺ`u16`) but is NOT carried in the OSRS blob. `toLegacy377()` uses
   423Ôćĺ   423Ôćĺ`this.frameId` (default 0). Callers that need a specific ID routed into
   424Ôćĺ   424Ôćĺ`AnimFrame.instances[id]` must set `frame.frameId` before calling
   425Ôćĺ   425Ôćĺ`toLegacy377()`.
   426Ôćĺ   426Ôćĺ
   427Ôćĺ   427ÔćĺThe skeleton (AnimBase) data is re-emitted byte-for-byte into the 377
   428Ôćĺ   428Ôćĺbase-data section. Since the OSRS skeleton format is identical to 377
   429Ôćĺ   429Ôćĺfor pre-HD content, this is lossless. The 377 `AnimBase.unpack()` will
   430Ôćĺ   430Ôćĺpush a DUPLICATE base entry into `AnimBase.instances[]` when re-decoding
   431Ôćĺ   431Ôćĺthe legacy blob ÔÇö the dep tracer / packer should deduplicate by content
   432Ôćĺ   432Ôćĺhash if memory matters.

   558Ôćĺ   558Ôćĺ
   559Ôćĺ   559Ôćĺ### Key API decisions
   560Ôćĺ   560Ôćĺ1. **`OsrsNpcType` extends `ConfigType` directly, NOT `NpcType`.** Same
   561Ôćĺ   561Ôćĺ   reasoning as task 5-b's `OsrsSeqType`: subclassing `NpcType` would
   562Ôćĺ   562Ôćĺ   inherit its private static `configs[]` / `configNames` and bound
   563Ôćĺ   563Ôćĺ   `load/parse/get` methods, routing OSRS NPC configs into the shared
   564Ôćĺ   564Ôćĺ   377 registry. A sibling class with a parallel registry keeps OSRS
   565Ôćĺ   565Ôćĺ   configs isolated so the modular variant selector (task 9) can pick
   566Ôćĺ   566Ôćĺ   which one to serve.
   567Ôćĺ   567Ôćĺ
   568Ôćĺ   568Ôćĺ2. **`op` array is lazily sized up to 10 entries.** Legacy opcodes 30-34
   569Ôćĺ   569Ôćĺ   populate `op[0..4]` (op1..op5). OSRS opcode 27 reads a u8 index and
   570Ôćĺ   570Ôćĺ   can populate any index (typically 5..9 for op6..op10). The array is
   571Ôćĺ   571Ôćĺ   allocated on first use and grown dynamically if opcode 27's index
   572Ôćĺ   572Ôćĺ   exceeds the current length. This avoids pre-allocating for the
   573Ôćĺ   573Ôćĺ   common case (5 ops) while still supporting the extended set.
   574Ôćĺ   574Ôćĺ
   575Ôćĺ   575Ôćĺ3. **`toLegacy377NpcConfig()` emits raw asset IDs, NOT pack-file
   576Ôćĺ   576Ôćĺ   names.** This deliberately decouples the engine class from
   577Ôćĺ   577Ôćĺ   `#tools/pack/PackFile.js` lookups (SeqPack, NpcPack, ModelPack) ÔÇö
   578Ôćĺ   578Ôćĺ   mirroring the existing `NpcType` class which also doesn't import
   579Ôćĺ   579Ôćĺ   PackFile. Task 8 (content-folder writer) is expected to post-process
   580Ôćĺ   580Ôćĺ   the emitted lines to rename IDs to debugnames via the pack files.
   581Ôćĺ   581Ôćĺ   The format `key=<int>` survives the round-trip into the content
   582Ôćĺ   582Ôćĺ   folder because LostCity's config parser accepts both names and
   583Ôćĺ   583Ôćĺ   integer literals.
   584Ôćĺ   584Ôćĺ
   585Ôćĺ   585Ôćĺ4. **OSRS-only fields are emitted as `param=osrs_<field>,<value>`
   586Ôćĺ   586Ôćĺ   entries.** This preserves them through the content-folder round-trip
   587Ôćĺ   587Ôćĺ   even though the 377 client doesn't understand them. When task 9
   588Ôćĺ   588Ôćĺ   (variant registry) registers these as well-known param IDs, the
   589Ôćĺ   589Ôćĺ   re-packing path will preserve them as opaque param values.
   590Ôćĺ   590Ôćĺ
   591Ôćĺ   591Ôćĺ5. **LostCity server-side codes 200-214 are read-and-discarded, not
   592Ôćĺ   592Ôćĺ   stored.** The spec describes these as "IGNORE them (skip silently,
   593Ôćĺ   593Ôćĺ   log debug)" ÔÇö interpreted as: read the payload (using the known
   594Ôćĺ   594Ôćĺ   format from `NpcType.ts`) and discard the values, so this decoder
   595Ôćĺ   595Ôćĺ   can parse a LostCity npc.dat without crashing. The discarded values
   596Ôćĺ   596Ôćĺ   are NOT stored on this class because OSRS-format NPCs don't carry
   597Ôćĺ   597Ôćĺ   these fields; the modular pipeline is expected to layer server-side
   598Ôćĺ   598Ôćĺ   config on top via the content folder rather than the cache.
   599Ôćĺ   599Ôćĺ
   600Ôćĺ   600Ôćĺ6. **`extractDependencyRefs()` excludes `multinpc[]`.** Multinpc
   601Ôćĺ   601Ôćĺ   references other NPC config IDs, but those are walked via the NPC
   602Ôćĺ   602Ôćĺ   config index, not via archive IDs (1/2 for models/anims). Including
   603Ôćĺ   603Ôćĺ   them in the returned `params`/`anims` arrays would mix asset kinds.
   604Ôćĺ   604Ôćĺ   Task 6 (dep tracer) is expected to walk multinpc separately by
   605Ôćĺ   605Ôćĺ   recursing into the NPC config index.
   606Ôćĺ   606Ôćĺ
   607Ôćĺ   607Ôćĺ7. **`osrsExtensions` is a fixed 5-slot `Int8Array` indexed by
   608Ôćĺ   608Ôćĺ   `code - 130`.** Codes 130-134 are described in the spec as "store
   609Ôćĺ   609Ôćĺ   opaquely" with no semantic meaning documented upstream. A fixed
   610Ôćĺ   610Ôćĺ   array is simpler than a `Map<number, number>` and gives O(1) access
   611Ôćĺ   611Ôćĺ   for consumers that later want to interpret specific slots.
   612Ôćĺ   612Ôćĺ
   613Ôćĺ   613Ôćĺ### Opcode ambiguities resolved
   614Ôćĺ   614Ôćĺ
   615Ôćĺ   615Ôćĺ1. **Opcode 23 reads 5 shorts, NOT 6.** The task spec listed
   616Ôćĺ   616Ôćĺ   "(stand, walk, walkL, walkR, walkB, run)" ÔÇö 6 names ÔÇö but said
   617Ôćĺ   617Ôćĺ   "g2├Ś5". The trailing "run" is a spec typo; standard RuneLite reads
   618Ôćĺ   618Ôćĺ   exactly 5 shorts (stand, walk, walkLeft, walkRight, walkBack).
   619Ôćĺ   619Ôćĺ   Decoded as: readyanim, walkanim, walkanim_l, walkanim_r, walkanim_b
   620Ôćĺ   620Ôćĺ   (overwriting the same fields populated by 377 codes 13/14/17).
   621Ôćĺ   621Ôćĺ
   622Ôćĺ   622Ôćĺ2. **Opcode 26 field label is "runanim code 26" but values are all
   623Ôćĺ   623Ôćĺ   crawl-related.** Treated as a spec typo ÔÇö opcode 26 is decoded as a
   624Ôćĺ   624Ôćĺ   crawl-anim family extension (5 shorts: crawl, crawlL, crawlR,
   625Ôćĺ   625Ôćĺ   crawlB, crawlBack2). Overwrites the 4 crawl fields from opcode 25
   626Ôćĺ   626Ôćĺ   (last write wins) and additionally populates `crawlanim_b2`.
   627Ôćĺ   627Ôćĺ
   628Ôćĺ   628Ôćĺ3. **Opcode 17 byte order: spec vs `NpcType.ts`.** The task spec says
   629Ôćĺ   629Ôćĺ   code 17 reads (walk, back, left, right). The existing 377
   630Ôćĺ   630Ôćĺ   `NpcType.ts` reads (walk, back, right, left) ÔÇö left/right swapped.
   631Ôćĺ   631Ôćĺ   The Java client NpcType.java reads in obfuscated order with unclear
   632Ôćĺ   632Ôćĺ   semantics. LostCity's own unpack tool (`unpackNpcConfig`) reads in
   633Ôćĺ   633Ôćĺ   the SPEC order (l before r), so two of three LostCity sources agree
   634Ôćĺ   634Ôćĺ   with the spec. This decoder follows the spec (l before r) for code
   635Ôćĺ   635Ôćĺ   17. Documented as a known discrepancy with `NpcType.ts`; if
   636Ôćĺ   636Ôćĺ   LostCity ever regenerates its 377 npc.dat from RL sources, the 377
   637Ôćĺ   637Ôćĺ   `NpcType.ts` should be updated to match.
   638Ôćĺ   638Ôćĺ
   639Ôćĺ   639Ôćĺ4. **Opcode 28 vs 29 (texture recolors).** Spec describes 28 as
   640Ôćĺ   640Ôćĺ   "recol_d_texture (dst)" and 29 as "recol_s/d texture (paired with
   641Ôćĺ   641Ôćĺ   28)" ÔÇö both with shape "g1 count, g2├Ś2 per pair". The semantic
   642Ôćĺ   642Ôćĺ   distinction between 28 and 29 is unclear from the spec; both are
   643Ôćĺ   643Ôćĺ   decoded identically and stored in the same `recol_s_tex` /
   644Ôćĺ   644Ôćĺ   `recol_d_tex` parallel arrays (last opcode wins, matching how
   645Ôćĺ   645Ôćĺ   legacy opcode 40 behaves on repeat). If a future cache distinguishes
   646Ôćĺ   646Ôćĺ   them semantically, the single `decodeTexRecolors()` helper is the
   647Ôćĺ   647Ôćĺ   one place to split the storage.
   648Ôćĺ   648Ôćĺ
   649Ôćĺ   649Ôćĺ5. **`desc` field uses `gjstr()`, not `gjstrraw()`.** The task spec
   650Ôćĺ   650Ôćĺ   says code 3 reads `gjstrraw`. The Java client's `gjstrraw()`
   651Ôćĺ   651Ôćĺ   returns `byte[]` (raw bytes). The existing 377 `NpcType.ts` uses
   652Ôćĺ   652Ôćĺ   `gjstr()` (which returns a string) and stores `desc` as a string.
   653Ôćĺ   653Ôćĺ   `Packet.ts` does NOT have a `gjstrraw()` method. To match the
   654Ôćĺ   654Ôćĺ   existing TS convention (and avoid touching `Packet.ts`), this
   655Ôćĺ   655Ôćĺ   decoder uses `gjstr()`. If raw-bytes round-trip fidelity becomes
   656Ôćĺ   656Ôćĺ   important, a future task should add `Packet.gjstrraw()` and store
   657Ôćĺ   657Ôćĺ   `desc` as `Uint8Array`.
   658Ôćĺ   658Ôćĺ
   659Ôćĺ   659Ôćĺ6. **Opcode 101 contrast scaling.** The Java client reads `g1b() * 5`
   660Ôćĺ   660Ôćĺ   for the effective contrast value. The existing 377 `NpcType.ts`
   661Ôćĺ   661Ôćĺ   stores the raw `g1b()` byte (no scaling). This decoder matches the
   662Ôćĺ   662Ôćĺ   377 TS convention (store raw byte; consumer scales). Documented in
   663Ôćĺ   663Ôćĺ   a code comment.
   664Ôćĺ   664Ôćĺ
   665Ôćĺ   665Ôćĺ7. **Opcode 249 (params) inherited unchanged.** Uses
   666Ôćĺ   666Ôćĺ   `ParamHelper.decodeParams(dat)` from the existing engine ÔÇö same
   667Ôćĺ   667Ôćĺ   path as 377 NpcType. No OSRS-specific changes needed.

   778Ôćĺ   778ÔćĺImplements a depth-first dependency walker that records an OSRS NPC's full
   779Ôćĺ   779Ôćĺtransitive dependency graph as a `DepsMap` (JSON-serializable). The walker
   780Ôćĺ   780Ôćĺhandles every node kind listed in the task spec (`npc`, `model`, `anim`,

   881Ôćĺ   881Ôćĺ**Task 7 implementation notes:**
   882Ôćĺ   882Ôćĺ1. The OSRS cache stores skeletons (AnimBase) in a separate index from
   883Ôćĺ   883Ôćĺ   frames. The reader MUST load bases into `AnimBase.instances[]` BEFORE
   884Ôćĺ   884Ôćĺ   returning frames that reference them ÔÇö `OsrsAnimFrame.decode()`
   885Ôćĺ   885Ôćĺ   requires the base to be present (see task 5-b worklog entry).
   886Ôćĺ   886Ôćĺ2. Every method MUST return `null` on missing IDs ÔÇö do NOT throw. The
   887Ôćĺ   887Ôćĺ   tracer records null returns as `missing: true` dep refs.
   888Ôćĺ   888Ôćĺ3. `getName` is optional but recommended ÔÇö without it, `DepNode.name`
   889Ôćĺ   889Ôćĺ   will be `null` for every node except NPCs (which carry their own
   890Ôćĺ   890Ôćĺ   `name` field on `OsrsNpcType`) and seqs (which carry `debugname`).
   891Ôćĺ   891Ôćĺ4. For `readParam`, prefer consulting an OSRS-side param registry if
   892Ôćĺ   892Ôćĺ   one is built. Falling back to the legacy 377 `ParamType.get(id)`
   893Ôćĺ   893Ôćĺ   works for shared params but will miss OSRS-native ones.
   894Ôćĺ   894Ôćĺ
   895Ôćĺ   895Ôćĺ### Key design decisions
   896Ôćĺ   896Ôćĺ
   897Ôćĺ   897Ôćĺ1. **`visitNode(kind, id, path)` is PUBLIC**, not private as the task
   898Ôćĺ   898Ôćĺ   spec hinted. Rationale: the standalone `walkParams` helper needs to
   899Ôćĺ   899Ôćĺ   call it (the spec explicitly specified `walkParams(params, tracer,
   900Ôćĺ   900Ôćĺ   deps)` as a standalone function taking the tracer as an argument),
   901Ôćĺ   901Ôćĺ   and future per-kind walkers (e.g. an `OsrsObjType` walker added by
   902Ôćĺ   902Ôćĺ   task 8) should hook into the same cycle-detection + memoization
   903Ôćĺ   903Ôćĺ   logic without duplicating it. The state it touches (`nodes`,
   904Ôćĺ   904Ôćĺ   `cycles`, `missing`) is still encapsulated as private fields on
   905Ôćĺ   905Ôćĺ   the tracer instance.
   906Ôćĺ   906Ôćĺ
   907Ôćĺ   907Ôćĺ2. **`walkParams` is BOTH a public method on the tracer (`tracer.walkParams(params, deps, path)`)
   908Ôćĺ   908Ôćĺ   AND a standalone exported function.** The standalone wrapper exists
   909Ôćĺ   909Ôćĺ   for parity with the task spec's `walkParams(params, tracer, deps)`
   910Ôćĺ   910Ôćĺ   signature; it adds a required `path: Set<string>` argument (not in
   911Ôćĺ   911Ôćĺ   the spec) because cycle detection depends on it. Callers should
   912Ôćĺ   912Ôćĺ   prefer the method form.
   913Ôćĺ   913Ôćĺ
   914Ôćĺ   914Ôćĺ3. **`id` is `number | string` throughout the schema.** Numeric for
   915Ôćĺ   915Ôćĺ   cache-indexed assets (npc/model/anim/seq/obj/param/struct/texture/
   916Ôćĺ   916Ôćĺ   particle/sound); string for scripts (whose "id" is their debugname,
   917Ôćĺ   917Ôćĺ   e.g. `'dragon'`). This is necessary because LostCity runescript
   918Ôćĺ   918Ôćĺ   references assets by name (e.g. `npc_anim(dragon_attack, 0)`), and
   919Ôćĺ   919Ôćĺ   the script scanner can't resolve names to numeric cache IDs without
   920Ôćĺ   920Ôćĺ   the pack files (task 8's job). String-ID refs are recorded as
   921Ôćĺ   921Ôćĺ   `missing: true` until task 8 resolves them.
   922Ôćĺ   922Ôćĺ
   923Ôćĺ   923Ôćĺ4. **`transformedFrom` is always `null` in the tracer's output.** It
   924Ôćĺ   924Ôćĺ   will be populated by task 8 (content-folder writer) when it
   925Ôćĺ   925Ôćĺ   actually generates the 377-transformed twin of an OSRS node. The
   926Ôćĺ   926Ôćĺ   field exists in the schema now so task 8 doesn't need a schema
   927Ôćĺ   927Ôćĺ   migration.
   928Ôćĺ   928Ôćĺ
   929Ôćĺ   929Ôćĺ5. **`skins` flag on model nodes.** Per task 5-a's next-action hook,
   930Ôćĺ   930Ôćĺ   models with non-empty `triangleSkin`/`vertexSkin` arrays lose
   931Ôćĺ   931Ôćĺ   rig-skinning data when transformed to 377. The tracer records this
   932Ôćĺ   932Ôćĺ   as `skins: boolean` on model nodes so the pilot can prefer
   933Ôćĺ   933Ôćĺ   `skins === false` models when alternatives exist.
   934Ôćĺ   934Ôćĺ
   935Ôćĺ   935Ôćĺ6. **`secondaryRoots` for batched traces.** `traceMany()` keeps the
   936Ôćĺ   936Ôćĺ   single-`root` schema shape (backward compat with single-NPC
   937Ôćĺ   937Ôćĺ   traces) by making the first NPC the root and listing any
   938Ôćĺ   938Ôćĺ   additional NPCs in `secondaryRoots: DepRef[]`. The field is
   939Ôćĺ   939Ôćĺ   omitted for single-NPC traces (the common case).
   940Ôćĺ   940Ôćĺ
   941Ôćĺ   941Ôćĺ7. **`ObjType` deep walk is stubbed.** Per task spec: "for now, treat
   942Ôćĺ   942Ôćĺ   obj as a partial node and log 'TODO: deep obj walk not
   943Ôćĺ   943Ôćĺ   implemented'". The `visitObj` method records the obj as a leaf
   944Ôćĺ   944Ôćĺ   with `note: 'TODO: deep obj walk not implemented (OsrsObjType not
   945Ôćĺ   945Ôćĺ   ported)'`. When task 8 ports the obj decoder, this method should
   946Ôćĺ   946Ôćĺ   walk `obj.model`, `obj.manwear`, `obj.womanwear`, `obj.manhead`,
   947Ôćĺ   947Ôćĺ   `obj.womanhead`, `obj.countobj[]`, `obj.recol_s/d`, and
   948Ôćĺ   948Ôćĺ   `obj.params` (via `walkParams`).
   949Ôćĺ   949Ôćĺ
   950Ôćĺ   950Ôćĺ8. **`ParamType` lookup falls back to the legacy 377 registry.** When
   951Ôćĺ   951Ôćĺ   `CacheReader.readParam(id)` returns null (the common case until
   952Ôćĺ   952Ôćĺ   `OsrsParamType` is ported), the tracer tries `ParamType.get(id)`
   953Ôćĺ   953Ôćĺ   (the legacy 377 static registry). If both miss, the param is
   954Ôćĺ   954Ôćĺ   recorded with `note: 'param type unknown ÔÇö value not walked
   955Ôćĺ   955Ôćĺ   (OsrsParamType not ported)'` and its value is NOT walked (the
   956Ôćĺ   956Ôćĺ   importer will bring the param's value verbatim, but the tracer
   957Ôćĺ   957Ôćĺ   can't tell what asset kind it points at).
   958Ôćĺ   958Ôćĺ
   959Ôćĺ   959Ôćĺ9. **Script scanning is best-effort regex.** LostCity runescript
   960Ôćĺ   960Ôćĺ   references assets by debugname in call sites like
   961Ôćĺ   961Ôćĺ   `npc_anim(dragon_attack, 0)`, `spotanim_npc(firebreath_attack, 92, 0)`,
   962Ôćĺ   962Ôćĺ   `inv_total(worn, antidragonbreathshield)`, `sound_synth(dragonbreath, 0, 30)`,
   963Ôćĺ   963Ôćĺ   `npc_param(attack_sound)`. The tracer scans for these patterns and
   964Ôćĺ   964Ôćĺ   records the discovered names as string-ID dep refs (all marked
   965Ôćĺ   965Ôćĺ   `missing: true` until task 8 resolves them via pack files). The
   966Ôćĺ   966Ôćĺ   scan is intentionally conservative ÔÇö false positives would be
   967Ôćĺ   967Ôćĺ   worse than false negatives here (the importer would try to fetch
   968Ôćĺ   968Ôćĺ   non-existent assets).
   969Ôćĺ   969Ôćĺ
   970Ôćĺ   970Ôćĺ10. **Script discovery for an NPC uses TWO heuristics:**
   971Ôćĺ   971Ôćĺ    (a) look for `content/scripts/npc/scripts/<npcName>.rs2` (primary
   972Ôćĺ   972Ôćĺ    combat script convention); (b) scan all other `.rs2` files for
   973Ôćĺ   973Ôćĺ    `[<trigger>,<npcName>]` dispatch directives (e.g.
   974Ôćĺ   974Ôćĺ    `[ai_applayer2,babydragon] @dragon_ai_applayer2(false);`) and
   975Ôćĺ   975Ôćĺ    register each matching file as a separate script dep with
   976Ôćĺ   976Ôćĺ    `via: 'dispatch:<trigger>'`. The dispatch-scan is O(files ├Ś lines)
   977Ôćĺ   977Ôćĺ    but runs only once per NPC and only over the `scripts/npc/scripts/`
   978Ôćĺ   978Ôćĺ    subdir.

  1155Ôćĺ  1155Ôćĺ
  1156Ôćĺ  1156ÔćĺThe OSRS cache uses this layout (post-rev ~150 era, stable across modern
  1157Ôćĺ  1157Ôćĺdumps). The reader hard-codes these constants in
  1158Ôćĺ  1158Ôćĺ`OsrsCacheAssetReader.ts`:
  1159Ôćĺ  1159Ôćĺ
  1160Ôćĺ  1160Ôćĺ| Archive | Contents | Used by |

  1162Ôćĺ  1162Ôćĺ| 0       | Animations (frames AND skeletons, disjoint file IDs) | `readAnim` + `readAnimBase` |
  1163Ôćĺ  1163Ôćĺ| 1       | Models (one `.ob2` per file ID) | `readModel` |
  1164Ôćĺ  1164Ôćĺ| 2       | Music ÔÇö older OSRS dumps use this for config (fallback) | `ensureConfigChildren` fallback |
  1165Ôćĺ  1165Ôćĺ| 5       | Textures (older) ÔÇö fallback | `readTexture` fallback |
  1166Ôćĺ  1166Ôćĺ| 9       | Configs (multi-child bundle: `npc.dat`, `seq.dat`, etc.) | `readNpc`/`readSeq`/etc. (primary) |
  1167Ôćĺ  1167Ôćĺ| 11      | Textures (modern) | `readTexture` primary |
  1168Ôćĺ  1168Ôćĺ| 25      | Audio (synth sound effects) | `readSound` (stub ÔÇö not yet read) |
  1169Ôćĺ  1169Ôćĺ
  1170Ôćĺ  1170ÔćĺThe config bundle (`npc.dat`/`seq.dat`/`obj.dat`/etc.) is loaded from
  1171Ôćĺ  1171Ôćĺarchive 9 by default, with a fallback to archive 2 for older cache
  1172Ôćĺ  1172Ôćĺdumps. Child names are resolved via the master index's per-child name
  1173Ôćĺ  1173Ôćĺhashes (using the same `genHash()` algorithm as `Jagfile.ts`). If the
  1174Ôćĺ  1174Ôćĺmaster index doesn't have name hashes, a sequential fallback is used.

  1186Ôćĺ  1186Ôćĺ### Limitations / stubs
  1187Ôćĺ  1187Ôćĺ
  1188Ôćĺ  1188Ôćĺ1. **`readObj`** returns a STUB `ObjType(id)` with only the ID populated
  1189Ôćĺ  1189Ôćĺ   ÔÇö the legacy `ObjType.decode()` calls `printFatalError` (which calls
  1190Ôćĺ  1190Ôćĺ   `process.exit(1)`) on unrecognized opcodes, and OSRS obj configs use
  1191Ôćĺ  1191Ôćĺ   opcodes the 377 decoder doesn't know. The dep tracer treats objs as
  1192Ôćĺ  1192Ôćĺ   leaves anyway (Task 6 worklog item 7). Swap to a real decode once
  1193Ôćĺ  1193Ôćĺ   `OsrsObjType` is ported (Task 8+).
  1194Ôćĺ  1194Ôćĺ
  1195Ôćĺ  1195Ôćĺ2. **`readParam` / `readStruct`** return null ÔÇö the legacy decoders
  1196Ôćĺ  1196Ôćĺ   `throw` on unrecognized opcodes (OSRS uses new codes). The dep
  1197Ôćĺ  1197Ôćĺ   tracer falls back to the legacy 377 `ParamType.get(id)` /
  1198Ôćĺ  1198Ôćĺ   `StructType.get(id)` static registries when these return null,
  1199Ôćĺ  1199Ôćĺ   which is the documented behavior in the Task 6 worklog.
  1200Ôćĺ  1200Ôćĺ
  1201Ôćĺ  1201Ôćĺ3. **`readParticle` / `readSound`** return null ÔÇö OSRS particle systems
  1202Ôćĺ  1202Ôćĺ   and sound formats are complex and undocumented. The dep tracer
  1203Ôćĺ  1203Ôćĺ   records these as missing leaves, which is fine for the pilot.
  1204Ôćĺ  1204Ôćĺ
  1205Ôćĺ  1205Ôćĺ4. **`readAnimBase`** assumes skeletons live in archive 0 (the same
  1206Ôćĺ  1206Ôćĺ   archive as anim frames). This is correct for modern OSRS but may
  1207Ôćĺ  1207Ôćĺ   differ in some cache revisions. If the base isn't found at
  1208Ôćĺ  1208Ôćĺ   archive 0 file `id`, the method returns null.
  1209Ôćĺ  1209Ôćĺ
  1210Ôćĺ  1210Ôćĺ5. **Config bundle child-name resolution** uses the master index's
  1211Ôćĺ  1211Ôćĺ   per-child name hashes. If the master index was written with
  1212Ôćĺ  1212Ôćĺ   `named = false`, falls back to sequential child-ID assignment
  1213Ôćĺ  1213Ôćĺ   based on the standard OSRS child ordering
  1214Ôćĺ  1214Ôćĺ   (`npc.dat`, `npc.idx`, `seq.dat`, `seq.idx`, ...). This is
  1215Ôćĺ  1215Ôćĺ   best-effort and may need adjustment for unusual cache dumps.
  1216Ôćĺ  1216Ôćĺ
  1217Ôćĺ  1217Ôćĺ6. **Master index reading** assumes the master index is stored as a
  1218Ôćĺ  1218Ôćĺ   single compressed container at archive 255, file 0 (read via the
  1219Ôćĺ  1219Ôćĺ   normal sector-walking protocol). The sector's archiveType byte is
  1220Ôćĺ  1220Ôćĺ   NOT validated (only `id` and `part` are), to remain compatible
  1221Ôćĺ  1221Ôćĺ   with both the `archive + 1` convention (used for normal archives)
  1222Ôćĺ  1222Ôćĺ   and the `archiveType = 255` convention (sometimes used for the
  1223Ôćĺ  1223Ôćĺ   master index). This matches OpenRS2's behavior.

  1384Ôćĺ  1384Ôćĺ### WriterOptions design
  1385Ôćĺ  1385Ôćĺ
  1386Ôćĺ  1386Ôćĺ- **`dryRun`** (default `false`): when true, the writer walks the deps map,
  1387Ôćĺ  1387Ôćĺ  allocates IDs in memory, and produces an `ImportResult` showing what
  1388Ôćĺ  1388Ôćĺ  WOULD be written ÔÇö but does NOT touch the filesystem or any pack files.
  1389Ôćĺ  1389Ôćĺ  Every `WrittenEntry.path` is set to the sentinel string `'<dryrun>'`.
  1390Ôćĺ  1390Ôćĺ  Useful for previewing an import before committing it.
  1391Ôćĺ  1391Ôćĺ
  1392Ôćĺ  1392Ôćĺ- **`namePrefix`** (default `'osrs_'`): prepended to every sanitized
  1393Ôćĺ  1393Ôćĺ  debugname. Combined with the sanitized OSRS name/ID to form the final
  1394Ôćĺ  1394Ôćĺ  debugname (e.g. `osrs_tormented_demon`, `osrs_model_45000`).
  1395Ôćĺ  1395Ôćĺ
  1396Ôćĺ  1396Ôćĺ- **`groupFile`** (default `'osrs_imports'`): subfolder under
  1397Ôćĺ  1397Ôćĺ  `content/models/` where binary assets (.ob2, .anim, .base) are written,
  1398Ôćĺ  1398Ôćĺ  AND the basename of the `.npc` / `.seq` config files. Multiple NPCs
  1399Ôćĺ  1399Ôćĺ  imported in one batch share the same group file (their `[debugname]`
  1400Ôćĺ  1400Ôćĺ  blocks are appended to the same `.npc` / `.seq` file).
  1401Ôćĺ  1401Ôćĺ
  1402Ôćĺ  1402Ôćĺ- **`overwrite`** (default `false`): when false (the safe default), the
  1403Ôćĺ  1403Ôćĺ  writer NEVER overwrites existing files or pack entries ÔÇö collisions
  1404Ôćĺ  1404Ôćĺ  trigger a fresh debugname with a `_2`, `_3` suffix instead. When true,
  1405Ôćĺ  1405Ôćĺ  the writer REPLACES existing `[debugname]` blocks in config files and
  1406Ôćĺ  1406Ôćĺ  overwrites binary files with the same path. NOT recommended for normal
  1407Ôćĺ  1407Ôćĺ  use ÔÇö intended for re-importing after a sanitization rule change.
  1408Ôćĺ  1408Ôćĺ
  1409Ôćĺ  1409Ôćĺ### Sanitization rules
  1410Ôćĺ  1410Ôćĺ
  1411Ôćĺ  1411ÔćĺThe `sanitizeName(rawName, packType)` helper applies the following
  1412Ôćĺ  1412Ôćĺtransforms in order:
  1413Ôćĺ  1413Ôćĺ
  1414Ôćĺ  1414Ôćĺ1. **Lowercase** the entire string.
  1415Ôćĺ  1415Ôćĺ2. **Replace every char NOT in `[a-z0-9]` with `_`** ÔÇö spaces, hyphens,
  1416Ôćĺ  1416Ôćĺ   apostrophes, etc. all become underscores.
  1417Ôćĺ  1417Ôćĺ3. **Collapse runs of multiple `_`** into a single `_` (so `"a  b"` Ôćĺ
  1418Ôćĺ  1418Ôćĺ   `"a_b"`, not `"a__b"`).
  1419Ôćĺ  1419Ôćĺ4. **Trim leading/trailing `_`** (so `"_foo_"` Ôćĺ `"foo"`).
  1420Ôćĺ  1420Ôćĺ5. **Prefix** with `namePrefix` (default `osrs_`).
  1421Ôćĺ  1421Ôćĺ6. **Dedupe** against the target pack file AND a session-reserved set
  1422Ôćĺ  1422Ôćĺ   (in-memory, populated by prior `sanitizeName()` calls in the same
  1423Ôćĺ  1423Ôćĺ   writer lifetime). If the candidate name is already taken, append
  1424Ôćĺ  1424Ôćĺ   `_2`, `_3`, `_4`, ... until unique.
  1425Ôćĺ  1425Ôćĺ
  1426Ôćĺ  1426ÔćĺExamples:
  1427Ôćĺ  1427Ôćĺ- `"Tormented demon"` Ôćĺ `"osrs_tormented_demon"`
  1428Ôćĺ  1428Ôćĺ- `"td_stand"` Ôćĺ `"osrs_td_stand"`
  1429Ôćĺ  1429Ôćĺ- `"model_45000"` Ôćĺ `"osrs_model_45000"`
  1430Ôćĺ  1430Ôćĺ- `"anim_2005"` (after `osrs_anim_2005` already exists in `anim.pack`)
  1431Ôćĺ  1431Ôćĺ  Ôćĺ `"osrs_anim_2005_2"`
  1432Ôćĺ  1432Ôćĺ
  1433Ôćĺ  1433ÔćĺThe session-reserved set is critical for batch imports where two OSRS
  1434Ôćĺ  1434Ôćĺassets sanitize to the same candidate name in the same run ÔÇö without
  1435Ôćĺ  1435Ôćĺit, the second asset would silently overwrite the first's pack entry.
  1436Ôćĺ  1436Ôćĺ
  1437Ôćĺ  1437Ôćĺ### Idempotency approach
  1438Ôćĺ  1438Ôćĺ
  1439Ôćĺ  1439ÔćĺThe writer is idempotent at three layers:
  1440Ôćĺ  1440Ôćĺ
  1441Ôćĺ  1441Ôćĺ1. **Deps-map layer** (primary): every OSRS `DepNode` has a
  1442Ôćĺ  1442Ôćĺ   `transformedFrom` field (set by Task 8 per the schema). On each run,
  1443Ôćĺ  1443Ôćĺ   the writer's `bootstrapResolverFromDepsMap()` walks the input deps
  1444Ôćĺ  1444Ôćĺ   map and pre-populates `NameResolver` from any node with
  1445Ôćĺ  1445Ôćĺ   `transformedFrom !== null` (looking up the debugname from the
  1446Ôćĺ  1446Ôćĺ   corresponding pack file). Then `processNode()` checks the field
  1447Ôćĺ  1447Ôćĺ   directly: if set, the node is recorded as `skipped` with reason
  1448Ôćĺ  1448Ôćĺ   `already_imported (newId=N)` and NOT re-processed.
  1449Ôćĺ  1449Ôćĺ
  1450Ôćĺ  1450Ôćĺ2. **NameResolver layer** (cross-NPC): if two NPCs in the same batch
  1451Ôćĺ  1451Ôćĺ   reference the same OSRS asset (e.g. both use OSRS model 45000), the
  1452Ôćĺ  1452Ôćĺ   first NPC's `processNode()` call registers the asset in
  1453Ôćĺ  1453Ôćĺ   `NameResolver`. When the second NPC's `processNode()` runs for the
  1454Ôćĺ  1454Ôćĺ   same OSRS ID, it finds the existing entry in NameResolver, records
  1455Ôćĺ  1455Ôćĺ   it as `skipped` with reason `shared_with_prior_npc`, and updates the
  1456Ôćĺ  1456Ôćĺ   deps map's `transformedFrom` field to point at the existing new ID.
  1457Ôćĺ  1457Ôćĺ
  1458Ôćĺ  1458Ôćĺ3. **Config-file layer** (defensive): `appendConfigBlock()` scans the
  1459Ôćĺ  1459Ôćĺ   target `.npc` / `.seq` config file for an existing `[<debugname>]`
  1460Ôćĺ  1460Ôćĺ   header before appending. If found and `overwrite=false`, the append
  1461Ôćĺ  1461Ôćĺ   is skipped (with a warning). If found and `overwrite=true`, the
  1462Ôćĺ  1462Ôćĺ   existing block is stripped via `stripConfigBlock()` before the new
  1463Ôćĺ  1463Ôćĺ   block is appended. This protects against the case where a prior run
  1464Ôćĺ  1464Ôćĺ   wrote the config file but failed before persisting the updated
  1465Ôćĺ  1465Ôćĺ   deps.json (so the next run's deps-map layer doesn't know about the
  1466Ôćĺ  1466Ôćĺ   prior import).
  1467Ôćĺ  1467Ôćĺ
  1468Ôćĺ  1468ÔćĺThe CLI (`Import.ts`) persists the updated deps.json (with
  1469Ôćĺ  1469Ôćĺ`transformedFrom` filled in) to `content/deps/<npcName>.deps.json`
  1470Ôćĺ  1470Ôćĺafter every successful non-dry-run import. Subsequent runs pick this
  1471Ôćĺ  1471Ôćĺfile up via `--deps=<path>` (or auto-trace + apply the persisted
  1472Ôćĺ  1472Ôćĺ`transformedFrom` from a prior run via the `--out=` round-trip).
  1473Ôćĺ  1473Ôćĺ
  1474Ôćĺ  1474Ôćĺ### Pack-file ID allocation strategy
  1475Ôćĺ  1475Ôćĺ
  1476Ôćĺ  1476Ôćĺ**Decision: `max(existing IDs) + 1` for every pack type.**
  1477Ôćĺ  1477Ôćĺ
  1478Ôćĺ  1478ÔćĺThe schema supports reserved ranges per pack type (e.g. "models
  1479Ôćĺ  1479Ôćĺ50000-59999 reserved for OSRS imports") but no ranges are reserved
  1480Ôćĺ  1480Ôćĺtoday. The `allocateId(packType)` helper loads the pack file, finds
  1481Ôćĺ  1481Ôćĺthe maximum existing ID, and returns `max + 1` (or `0` for an empty
  1482Ôćĺ  1482Ôćĺpack, matching LostCity's convention where every existing pack starts
  1483Ôćĺ  1483Ôćĺat ID 0).
  1484Ôćĺ  1484Ôćĺ
  1485Ôćĺ  1485ÔćĺRationale:
  1486Ôćĺ  1486Ôćĺ- Simplest strategy that preserves backwards-compat (existing IDs are
  1487Ôćĺ  1487Ôćĺ  never reused or overwritten).
  1488Ôćĺ  1488Ôćĺ- The LostCity content folder doesn't have a "free ID range" convention
  1489Ôćĺ  1489Ôćĺ  ÔÇö IDs are assigned sequentially as new assets are added.
  1490Ôćĺ  1490Ôćĺ- If a future task adds reserved ranges, `allocateId()` is the single
  1491Ôćĺ  1491Ôćĺ  function to change (per-pack-type strategy can be added without
  1492Ôćĺ  1492Ôćĺ  touching call sites).
  1493Ôćĺ  1493Ôćĺ
  1494Ôćĺ  1494Ôćĺ**Anims are registered in BOTH `anim.pack` AND `animset.pack`** (with
  1495Ôćĺ  1495Ôćĺthe same ID and name). This matches the existing LostCity convention
  1496Ôćĺ  1496Ôćĺwhere both packs share the same `id=name` mapping for every anim file:
  1497Ôćĺ  1497Ôćĺ- `anim.pack` (AnimPack) is consulted by `SeqConfig.parseSeqConfig()`
  1498Ôćĺ  1498Ôćĺ  when resolving `frameN=<animname>` refs in `.seq` config files.
  1499Ôćĺ  1499Ôćĺ- `animset.pack` (AnimSetPack) is consulted by
  1500Ôćĺ  1500Ôćĺ  `engine/tools/pack/graphics/pack.ts` when mapping `.anim` files to
  1501Ôćĺ  1501Ôćĺ  archive 2 file IDs during cache packing.
  1502Ôćĺ  1502Ôćĺ
  1503Ôćĺ  1503ÔćĺRegistering in only one would break either the seq config parser or
  1504Ôćĺ  1504Ôćĺthe cache packer. The writer registers in both atomically (within the
  1505Ôćĺ  1505Ôćĺsame `updatePackFile()` call sequence, with both packs sharing the
  1506Ôćĺ  1506Ôćĺsame dirty/save batch).
  1507Ôćĺ  1507Ôćĺ
  1508Ôćĺ  1508Ôćĺ### NPC config line rewriting
  1509Ôćĺ  1509Ôćĺ
  1510Ôćĺ  1510Ôćĺ`OsrsNpcType.toLegacy377NpcConfig()` emits raw integer asset IDs in
  1511Ôćĺ  1511Ôćĺits output (e.g. `model1=45000`, `readyanim=9200`,
  1512Ôćĺ  1512Ôćĺ`param=osrs_runanim,9102`). The writer's `rewriteNpcConfigLine()`
  1513Ôćĺ  1513Ôćĺpost-processes each line to rewrite those IDs to their newly-assigned
  1514Ôćĺ  1514Ôćĺdebugnames via `NameResolver`:
  1515Ôćĺ  1515Ôćĺ
  1516Ôćĺ  1516Ôćĺ| Line shape                              | Rewrite                                          |
  1517Ôćĺ  1517Ôćĺ|-----------------------------------------|--------------------------------------------------|
  1518Ôćĺ  1518Ôćĺ| `modelN=<id>`                           | `modelN=<modelDebugname>`                        |
  1519Ôćĺ  1519Ôćĺ| `headN=<id>`                            | `headN=<modelDebugname>`                         |
  1520Ôćĺ  1520Ôćĺ| `readyanim=<id>`                        | `readyanim=<seqDebugname>`                       |
  1521Ôćĺ  1521Ôćĺ| `walkanim=<id>[,<id>,<id>,<id>]`        | per-element rewrite to seq debugnames            |
  1522Ôćĺ  1522Ôćĺ| `param=osrs_runanim,<id>`               | `param=osrs_runanim,<seqDebugname>`              |
  1523Ôćĺ  1523Ôćĺ| `param=osrs_runanim_{l,r,b},<id>`       | `param=osrs_runanim_{l,r,b},<seqDebugname>`      |
  1524Ôćĺ  1524Ôćĺ| `param=osrs_crawlanim,<id>`             | `param=osrs_crawlanim,<seqDebugname>`            |
  1525Ôćĺ  1525Ôćĺ| `param=osrs_crawlanim_{l,r,b,b2},<id>`  | `param=osrs_crawlanim_{l,r,b,b2},<seqDebugname>` |
  1526Ôćĺ  1526Ôćĺ| `param=osrs_walkanim_b,<id>`            | `param=osrs_walkanim_b,<seqDebugname>`           |
  1527Ôćĺ  1527Ôćĺ
  1528Ôćĺ  1528ÔćĺLines whose OSRS ID can't be resolved via `NameResolver` (e.g. the
  1529Ôćĺ  1529Ôćĺtracer didn't walk that model) fall back to the `model_<id>` /
  1530Ôćĺ  1530Ôćĺ`seq_<id>` / `anim_<id>` literal form ÔÇö the LostCity config parser
  1531Ôćĺ  1531Ôćĺaccepts both forms (it tries `getByName(value)` first, falling back to
  1532Ôćĺ  1532Ôćĺ`parseInt(value)` if the name lookup returns -1).
  1533Ôćĺ  1533Ôćĺ
  1534Ôćĺ  1534ÔćĺThe header line (`[<osrs_debugname>]`) is also replaced with
  1535Ôćĺ  1535Ôćĺ`[<sanitized_debugname>]` so the pack entry matches the config block
  1536Ôćĺ  1536Ôćĺheader. (The OSRS emitter uses the OSRS debugname or `npc_<id>` as
  1537Ôćĺ  1537Ôćĺthe header ÔÇö we override it with our `osrs_<name>` sanitized form.)
  1538Ôćĺ  1538Ôćĺ
  1539Ôćĺ  1539Ôćĺ### Seq config emission
  1540Ôćĺ  1540Ôćĺ
  1541Ôćĺ  1541Ôćĺ`writeSeq()` emits an ini-style `[debugname]` block per seq, mirroring
  1542Ôćĺ  1542Ôćĺthe format of `content/scripts/areas/area_kalphite/configs/kalphite.seq`
  1543Ôćĺ  1543Ôćĺ(existing LostCity convention):
  1544Ôćĺ  1544Ôćĺ
  1545Ôćĺ  1545Ôćĺ```
  1546Ôćĺ  1546Ôćĺ[osrs_td_attack]

  1551Ôćĺ  1551Ôćĺ```
  1552Ôćĺ  1552Ôćĺ
  1553Ôćĺ  1553ÔćĺThe 377 shared fields (loops, walkmerge, stretches, priority,

  1559Ôćĺ  1559Ôćĺ`OsrsNpcType.toLegacy377NpcConfig()` for OSRS-only NPC fields.
  1560Ôćĺ  1560Ôćĺ
  1561Ôćĺ  1561ÔćĺFrame references (`frameN=<animId>`) and interpolated-frame refs
  1562Ôćĺ  1562Ôćĺ(`iframeN=<animId>`) are rewritten to their newly-assigned debugnames
  1563Ôćĺ  1563Ôćĺvia `NameResolver`, the same way NPC config lines are.
