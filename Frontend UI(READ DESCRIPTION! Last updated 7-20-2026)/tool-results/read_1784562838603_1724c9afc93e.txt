    31→    31→
    32→    32→### Engine side (TypeScript, `engine-ts`)
    33→    33→- **Cache reader**: `engine/src/io/FileStream.ts` — 255-archive file store
    34→    34→  (archives 0..4). `read(archive, file, decompress)` / `write(...)`.
    35→    35→  Archives: 0=versionlist/jagfiles, 1=models, 2=anims, 3=midi, 4=map.
    36→    36→- **OnDemand**: `engine/src/engine/OnDemand.ts` — serves cache files to client

    38→    38→- **Model decoder**: `engine/src/cache/graphics/Model.ts` — 377-format reader.
    39→    39→  Reads 18-byte trailer (vertexCount, faceCount, texturedFaceCount, flags,
    40→    40→  data lengths), then decodes vertices/faces/textures.

    44→    44→  ObjType, etc. Each has a `decode(code, dat)` opcode switch.
    45→    45→- **Pack files**: `content/pack/*.pack` — hand-maintained `id=name` maps:
    46→    46→  `npc.pack`, `model.pack`, `anim.pack`, `seq.pack`, etc.

    48→    48→  with `register(id, name)`, `getByName(name)`, `save()`.
    49→    49→  `engine/tools/pack/graphics/pack.ts` writes models (`.ob2`) and anims
    50→    50→  (`.anim`) from content folder into `data/pack` cache, gzipped.
    51→    51→- **Config build**: `engine/tools/pack/config/NpcConfig.ts` etc — compiles
    52→    52→  `[debugname]` ini-style configs into `npc.dat`/`npc.idx` for client.
    53→    53→- **Content layout**: `content/{models,pack,scripts,...}`. NPC combat scripts
    54→    54→  live in `content/scripts/npc/scripts/*.rs2` (runescript), NPC configs in
    55→    55→  `content/scripts/npc/configs/*.npc`. Params in `content/scripts/npc/configs/npc.param`.

    67→    67→
    68→    68→### Format diff: 377 vs OSRS (key for the port)
    69→    69→The OSRS model format is a **strict superset** of the 377 format:
    70→    70→- Header trailer is 18 bytes in 377, **23 bytes** in OSRS (extra 5 bytes:
    71→    71→  `version`, `textureUVCount`, `triangleSkinCount`, `vertexSkinCount`,
    72→    72→  `maxDepth`).
    73→    73→- OSRS adds: per-vertex skin data, per-triangle skin data, textured UV coords,
    74→    74→  `version` byte (used as a feature gate).
    75→    75→- Old models decode fine under OSRS rules by treating the new fields as
    76→    76→  zero-length when their flag bytes are 0.
    77→    77→- Animations changed more substantially: OSRS anim frames use a different
    78→    78→  layout (no shared `head` block — each frame is self-contained).
    79→    79→
    80→    80→### Pack-list format (hand-maintained today)
    81→    81→Plain text, one entry per line: `0=hans\n1=man\n...`. Generated/validated by
    82→    82→`PackFile.save()`. We will extend, not replace.

   139→   139→### Summary
   140→   140→Ported RuneLite's `ModelDefinition.java` to TypeScript as `OsrsModel`, following
   141→   141→the LostCity conventions used by the existing 377 decoder at
   142→   142→`engine/src/cache/graphics/Model.ts`. Output uses `Packet` from `#/io/Packet.js`
   143→   143→for all byte reading, `printWarning` from `#/util/Logger.js` for warnings, and

   145→   145→
   146→   146→### Public API
   147→   147→```ts
   148→   148→class OsrsModel {
   149→   149→  static decode(id: number, data: Uint8Array | null): OsrsModel | null;
   150→   150→  static detect(data: Uint8Array): 'osrs' | 'legacy377';
   151→   151→  toLegacy377(): Uint8Array;
   152→   152→  // plus public decoded fields (typed arrays) and header counts
   153→   153→}
   154→   154→
   155→   155→class OsrsModelMetadata {
   156→   156→  // all trailer fields + computed section offsets
   157→   157→}
   158→   158→```
   159→   159→
   160→   160→The class is **API-compatible in spirit** with the existing 377 `Model` class
   161→   161→(sibling `Metadata` class + static `decode()`), but the OSRS decoder is
   162→   162→one-shot (no separate `unpack`/`fromId` split) — it is intended for pipeline
   163→   163→use (decode → transform → pack), not for runtime client caching. The 377
   164→   164→decoder's two-stage split was an optimisation for the live game server; the
   165→   165→OSRS pipeline doesn't need it.
   166→   166→
   167→   167→### Spec ambiguities resolved
   168→   168→
   169→   169→1. **Trailer byte count.** The task spec listed 12 trailer fields (17 bytes)
   170→   170→   but asserted the trailer is 23 bytes. Resolved by combining the spec list
   171→   171→   with the worklog note ("extra 5 bytes: `version`, `textureUVCount`,
   172→   172→   `triangleSkinCount`, `vertexSkinCount`, `maxDepth`") and keeping every 377
   173→   173→   field (including `priority`, which the spec omitted). Final layout = 17
   174→   174→   fields, 23 bytes total. The 5 NEW bytes are: `version`,
   175→   175→   `hasFaceTextures`, `triangleSkinCount`, `vertexSkinCount`, `maxDepth`
   176→   176→   (all u8). `textureUVCount` from the worklog is interpreted as the same
   177→   177→   field as `triangleTextureCount` (the count of textured triangles that
   178→   178→   carry UVs).
   179→   179→
   180→   180→2. **Triangle orientation/type byte.** The spec lists section 11
   181→   181→   ("triangle vertex indices") as using the "same 1/2/3/4 type encoding as
   182→   182→   377" but doesn't list a separate orientation section. In 377 the
   183→   183→   orientation byte lives in its own 1-byte-per-face section read BEFORE
   184→   184→   the index deltas. Resolved by adding a triangle-orientation section
   185→   185→   (1 byte per triangle) at body position 5.5 (between vertex labels and
   186→   186→   triangle colours) — the same role as 377's `faceOrientationsOffset`.
   187→   187→   The body section order comment in the source makes this explicit.
   188→   188→
   189→   189→3. **`priority` byte.** 377 has a `priority` byte (255 = per-face priority
   190→   190→   array; otherwise shared priority value). The OSRS spec body section
   191→   191→   list has no priority section, but dropping the byte entirely would lose
   192→   192→   information needed for the 377 backward-compat path. Kept the byte in
   193→   193→   the OSRS trailer at the same trailer position as 377 (between
   194→   194→   `hasFaceTextures` and `hasFaceAlphas`).
   195→   195→
   196→   196→4. **Skin section encoding.** Spec body sections 13/14 say "only if
   197→   197→   `version >= 1` AND skinCount > 0" but don't say how each skin entry is
   198→   198→   encoded. Chose u8 per entry (matching the modern RuneLite decoder).
   199→   199→   If a future cache uses smart-encoded skin entries, this is the single
   200→   200→   line to change.
   201→   201→
   202→   202→5. **Detect heuristic.** `detect()` sniffs by reading the version byte at
   203→   203→   `data.length - 23`, then validates vertexCount/triangleCount/dataLen
   204→   204→   fields against the blob size. Falls back to `'legacy377'` on any
   205→   205→   sanity failure — the legacy decoder will then either succeed or report
   206→   206→   its own error. The safe default prevents false-positive OSRS
   207→   207→   classification of malformed 377 blobs.
   208→   208→
   209→   209→### `toLegacy377()` transform — limitations
   210→   210→
   211→   211→The transform emits a faithful 377 `.ob2` blob that the existing Java client
   212→   212→can render. The following OSRS-only data is **dropped**:
   213→   213→
   214→   214→| OSRS field             | 377 equivalent                | Impact |
   215→   215→|------------------------|-------------------------------|--------|
   216→   216→| `triangleMaterial`     | none                          | Faces with material textures fall back to flat colour |
   217→   217→| `triangleSkin`         | none                          | Rig-based triangle skinning lost |
   218→   218→| `vertexSkin`           | none                          | Rig-based vertex skinning lost |
   219→   219→| `maxDepth`             | none                          | Bounding-box hint lost (client recomputes) |
   220→   220→
   221→   221→The dependency tracer (task 6) MUST flag models that have non-zero
   222→   222→`triangleSkin`/`vertexSkin` arrays, because animations targeting those rigs
   223→   223→will not play correctly when the model is rendered as a 377 blob. The pilot
   224→   224→NPC (Tormented Demon) selection should prefer models with `triangleSkinCount
   225→   225→== 0 && vertexSkinCount == 0` if any are available.
   226→   226→
   227→   227→Round-trip verified by smoke test:
   228→   228→  OSRS bytes → `OsrsModel.decode()` → `toLegacy377()` → `Model.unpack()`
   229→   229→  → `Model.fromId()` — all vertex positions, triangle indices, colours,
   230→   230→  infos, alphas, tags, vertex labels, and per-face/shared priorities
   231→   231→  match exactly. Textured UV vertex indices also survive (377 has the
   232→   232→  same field shape — 3 u16 vertex indices per textured triangle).
   233→   233→

   328→   328→
   329→   329→### Key API decisions
   330→   330→1. **`OsrsAnimFrame` is a sibling class, not `extends AnimFrame`.**
   331→   331→   The 377 `AnimFrame` is heavily tied to its 4-section `unpack()` static
   332→   332→   loader and the `AnimFrame.instances[]` / `AnimFrame.order[]` registries
   333→   333→   managed by `OnDemand.cache`. Subclassing would inherit those statics
   334→   334→   verbatim and route OSRS frames into the live 377 registry — exactly
   335→   335→   what the modular pipeline wants to avoid. A sibling class with matching
   336→   336→   field shape gives polymorphism without unwanted registry coupling.
   337→   337→   This mirrors the sibling-class pattern used by `OsrsModel` (task 5-a).
   338→   338→
   339→   339→2. **`OsrsSeqType` is a sibling class extending `ConfigType` directly**, not
   340→   340→   `extends SeqType`. Same reasoning as above — `SeqType` has private
   341→   341→   static `configs[]` / `configNames` and bound `load/parse/get` methods
   342→   342→   that would route OSRS seqs into the shared 377 registry.
   343→   343→
   344→   344→3. **`OsrsAnimFrame.decode()` takes `baseId` as a parameter** even though
   345→   345→   the trailing 2 bytes of the blob also encode it. We cross-check and
   346→   346→   warn on mismatch but trust the parameter — the caller is expected to
   347→   347→   have resolved the base ID through the cache index, and the trailing
   348→   348→   bytes are a redundant integrity check (sometimes zero in OSRS caches
   349→   349→   that store the base reference in a separate index file).
   350→   350→
   351→   351→4. **`frameId` defaults to 0** in `toLegacy377()` because the OSRS blob
   352→   352→   doesn't carry a frame ID. The modular packer is expected to assign a
   353→   353→   real ID (and rewrite it into the head section if needed) before the
   354→   354→   blob reaches the live client.
   355→   355→
   356→   356→5. **`OsrsSeqType.decode()` throws on codes 16+** (sub-frame references).
   357→   357→   The task spec describes them only as "16+ for sub-frame references"
   358→   358→   without a layout. Throwing lets the dep tracer (task 6) flag any seq
   359→   359→   that uses them, rather than silently dropping data.
   360→   360→
   361→   361→### Format ambiguities resolved
   362→   362→
   363→   363→1. **"For groups with no flag, the previous group's value carries forward
   364→   364→   (or 0 for translate, 128 for scale — same default-value rule as 377)"**
   365→   365→   — interpreted as: use the SAME default-value rule as the 377 decoder
   366→   366→   (0 for OP_TRANSLATE/OP_ROTATE/OP_ALPHA, 128 for OP_SCALE). The
   367→   367→   "carry forward" phrasing in the spec was a description of how the
   368→   368→   default value doesn't change between adjacent same-bone-type groups,
   369→   369→   not a request to track a per-axis running last-value. The synthetic
   370→   370→   OP_BASE pull-forward walk from `AnimFrame.unpack()` is preserved
   371→   371→   unchanged — it emits a zero-value entry for the most recent OP_BASE
   372→   372→   bone when a non-OP_BASE bone has a non-zero flag.
   373→   373→
   374→   374→2. **OSRS body layout is INTERLEAVED, not sectioned.** The 377 decoder
   375→   375→   reads flags from a separate `tran1` section and smart values from a
   376→   376→   separate `tran2` section. The OSRS spec describes "For each group i:
   377→   377→   read typeFlag, then read axis values for set bits" — i.e., the flag
   378→   378→   byte and the smart values are read sequentially from a single
   379→   379→   section. This matches RuneLite's `FrameDefinition` decoder. Verified
   380→   380→   by round-trip smoke test below.
   381→   381→
   382→   382→3. **Optional interpolation tail.** RuneLite's `FrameDefinition` reads
   383→   383→   (after the delay byte) an optional `interpolationType: u8`, then
   384→   384→   (if 4 more bytes are present) `leftHandItem: u16` and
   385→   385→   `rightHandItem: u16`. The task spec calls these `leftHandWeight` /
   386→   386→   `rightHandWeight` — I followed the spec's naming but kept RuneLite's
   387→   387→   read layout (1 byte if any remain, then 4 bytes if at least 5 remain
   388→   388→   total). If a future cache uses a different layout, this is the
   389→   389→   single block to revisit.
   390→   390→
   391→   391→4. **Opcode 12 (`field790` in the Java client).** The 377 TS
   392→   392→   `SeqType.decode()` does NOT handle code 12 (it throws). The Java
   393→   393→   client does, reading a `u32`. The OSRS spec doesn't mention code 12
   394→   394→   explicitly, but since OSRS is a superset of the 377 client opcode
   395→   395→   set, code 12 should be handled. I added it as `field12: u32` with a
   396→   396→   comment documenting it as an opaque field carried over from the
   397→   397→   Java client (`field790`) — its semantic meaning isn't documented
   398→   398→   upstream.
   399→   399→
   400→   400→5. **`AnimBase.instances[]` is reused.** The OSRS skeleton format is
   401→   401→   byte-identical to 377 for pre-HD content (same `length u8`, `types
   402→   402→   u8[]`, `labels u8[][]` layout). Rather than duplicate the loader,
   403→   403→   `OsrsAnimFrame` references the existing `AnimBase.instances[]`
   404→   404→   registry. OSRS-only pipelines that don't load 377 anims must still
   405→   405→   call `AnimBase.unpack()` on OSRS base files to populate this
   406→   406→   registry before invoking `OsrsAnimFrame.decode()`. (This is the
   407→   407→   task 6 dep tracer's responsibility.)
   408→   408→
   409→   409→### `toLegacy377()` transform — limitations
   410→   410→
   411→   411→The transform emits a faithful 1-frame 377 anim blob that the existing
   412→   412→377 client can render via `AnimFrame.unpack()`. The following OSRS-only
   413→   413→data is **dropped**:
   414→   414→
   415→   415→| OSRS field             | 377 equivalent                | Impact |
   416→   416→|------------------------|-------------------------------|--------|
   417→   417→| `interpolationType`    | none                          | Blended/interpolated animations fall back to standard single-frame playback |
   418→   418→| `leftHandWeight`       | none                          | Left-hand blend weight lost |
   419→   419→| `rightHandWeight`      | none                          | Right-hand blend weight lost |
   420→   420→
   421→   421→The `frameId` field is REQUIRED by 377 (the head section stores it as a
   422→   422→`u16`) but is NOT carried in the OSRS blob. `toLegacy377()` uses
   423→   423→`this.frameId` (default 0). Callers that need a specific ID routed into
   424→   424→`AnimFrame.instances[id]` must set `frame.frameId` before calling
   425→   425→`toLegacy377()`.
   426→   426→
   427→   427→The skeleton (AnimBase) data is re-emitted byte-for-byte into the 377
   428→   428→base-data section. Since the OSRS skeleton format is identical to 377
   429→   429→for pre-HD content, this is lossless. The 377 `AnimBase.unpack()` will
   430→   430→push a DUPLICATE base entry into `AnimBase.instances[]` when re-decoding
   431→   431→the legacy blob — the dep tracer / packer should deduplicate by content
   432→   432→hash if memory matters.

   505→   505→### Summary
   506→   506→Ported the OSRS NpcConfig opcode decoder to TypeScript as `OsrsNpcType`,
   507→   507→following the LostCity conventions used by the existing 377 decoder at
   508→   508→`engine/src/cache/config/NpcType.ts` and matching the sibling-class pattern
   509→   509→established by tasks 5-a (`OsrsModel`) and 5-b (`OsrsAnimFrame` /
   510→   510→`OsrsSeqType`).
   511→   511→
   512→   512→The OSRS NPC opcode set is a strict superset of 377: every 377 opcode
   513→   513→decodes the same way, plus OSRS adds codes 23, 24, 25, 26, 27, 28, 29,
   514→   514→118, 119, 121, 122, 123, 124, 125, 126, 127, 128, 129, and 130-134.
   515→   515→This decoder handles BOTH sets so it can read an OSRS-format `npc.dat`
   516→   516→directly. LostCity server-side codes 200-214 are read-and-discarded
   517→   517→(with debug log) so this decoder can also parse a LostCity npc.dat
   518→   518→without crashing.
   519→   519→
   520→   520→### Public API
   521→   521→```ts
   522→   522→class OsrsNpcType extends ConfigType {
   523→   523→    // Static registry (mirror of OsrsSeqType's pattern):
   524→   524→    static configNames: Map<string, number>;
   525→   525→    static configs: OsrsNpcType[];

   527→   527→    static getId(name): number;
   528→   528→    static getByName(name): OsrsNpcType | null;
   529→   529→    static get count(): number;
   530→   530→
   531→   531→    // 377 shared fields (mirror of NpcType.ts):
   532→   532→    name, desc, size, models, heads, hasanim, readyanim, walkanim,
   533→   533→    walkanim_b, walkanim_r, walkanim_l, hasalpha, recol_s, recol_d,
   534→   534→    op (size-10 lazy array), resizex, resizey, resizez, minimap,
   535→   535→    vislevel, resizeh, resizev, alwaysontop, ambient, contrast,
   536→   536→    headicon, turnspeed, multivarbit, multivarp, multinpc, active,
   537→   537→    category, stats[6], params
   538→   538→
   539→   539→    // OSRS-only fields:
   540→   540→    runanim, runanim_l, runanim_r, runanim_b           // code 24
   541→   541→    crawlanim, crawlanim_l, crawlanim_r, crawlanim_b, // codes 25/26
   542→   542→        crawlanim_b2                                    // (5th short of code 26)
   543→   543→    recol_s_tex, recol_d_tex                           // codes 28/29
   544→   544→    sizeH, sizeW                                       // codes 118/119
   545→   545→    mapFunction, mapScene                              // code 121

   550→   550→    osrsExtensions: Int8Array(5)                       // codes 130-134
   551→   551→

   553→   553→    decode(code: number, dat: Packet): void;

   558→   558→
   559→   559→### Key API decisions
   560→   560→1. **`OsrsNpcType` extends `ConfigType` directly, NOT `NpcType`.** Same
   561→   561→   reasoning as task 5-b's `OsrsSeqType`: subclassing `NpcType` would
   562→   562→   inherit its private static `configs[]` / `configNames` and bound
   563→   563→   `load/parse/get` methods, routing OSRS NPC configs into the shared
   564→   564→   377 registry. A sibling class with a parallel registry keeps OSRS
   565→   565→   configs isolated so the modular variant selector (task 9) can pick
   566→   566→   which one to serve.
   567→   567→
   568→   568→2. **`op` array is lazily sized up to 10 entries.** Legacy opcodes 30-34
   569→   569→   populate `op[0..4]` (op1..op5). OSRS opcode 27 reads a u8 index and
   570→   570→   can populate any index (typically 5..9 for op6..op10). The array is
   571→   571→   allocated on first use and grown dynamically if opcode 27's index
   572→   572→   exceeds the current length. This avoids pre-allocating for the
   573→   573→   common case (5 ops) while still supporting the extended set.
   574→   574→
   575→   575→3. **`toLegacy377NpcConfig()` emits raw asset IDs, NOT pack-file
   576→   576→   names.** This deliberately decouples the engine class from
   577→   577→   `#tools/pack/PackFile.js` lookups (SeqPack, NpcPack, ModelPack) —
   578→   578→   mirroring the existing `NpcType` class which also doesn't import
   579→   579→   PackFile. Task 8 (content-folder writer) is expected to post-process
   580→   580→   the emitted lines to rename IDs to debugnames via the pack files.
   581→   581→   The format `key=<int>` survives the round-trip into the content
   582→   582→   folder because LostCity's config parser accepts both names and
   583→   583→   integer literals.
   584→   584→
   585→   585→4. **OSRS-only fields are emitted as `param=osrs_<field>,<value>`
   586→   586→   entries.** This preserves them through the content-folder round-trip
   587→   587→   even though the 377 client doesn't understand them. When task 9
   588→   588→   (variant registry) registers these as well-known param IDs, the
   589→   589→   re-packing path will preserve them as opaque param values.
   590→   590→
   591→   591→5. **LostCity server-side codes 200-214 are read-and-discarded, not
   592→   592→   stored.** The spec describes these as "IGNORE them (skip silently,
   593→   593→   log debug)" — interpreted as: read the payload (using the known
   594→   594→   format from `NpcType.ts`) and discard the values, so this decoder
   595→   595→   can parse a LostCity npc.dat without crashing. The discarded values
   596→   596→   are NOT stored on this class because OSRS-format NPCs don't carry
   597→   597→   these fields; the modular pipeline is expected to layer server-side
   598→   598→   config on top via the content folder rather than the cache.
   599→   599→
   600→   600→6. **`extractDependencyRefs()` excludes `multinpc[]`.** Multinpc
   601→   601→   references other NPC config IDs, but those are walked via the NPC
   602→   602→   config index, not via archive IDs (1/2 for models/anims). Including
   603→   603→   them in the returned `params`/`anims` arrays would mix asset kinds.
   604→   604→   Task 6 (dep tracer) is expected to walk multinpc separately by
   605→   605→   recursing into the NPC config index.
   606→   606→
   607→   607→7. **`osrsExtensions` is a fixed 5-slot `Int8Array` indexed by
   608→   608→   `code - 130`.** Codes 130-134 are described in the spec as "store
   609→   609→   opaquely" with no semantic meaning documented upstream. A fixed
   610→   610→   array is simpler than a `Map<number, number>` and gives O(1) access
   611→   611→   for consumers that later want to interpret specific slots.
   612→   612→
   613→   613→### Opcode ambiguities resolved
   614→   614→
   615→   615→1. **Opcode 23 reads 5 shorts, NOT 6.** The task spec listed
   616→   616→   "(stand, walk, walkL, walkR, walkB, run)" — 6 names — but said
   617→   617→   "g2×5". The trailing "run" is a spec typo; standard RuneLite reads
   618→   618→   exactly 5 shorts (stand, walk, walkLeft, walkRight, walkBack).
   619→   619→   Decoded as: readyanim, walkanim, walkanim_l, walkanim_r, walkanim_b
   620→   620→   (overwriting the same fields populated by 377 codes 13/14/17).
   621→   621→
   622→   622→2. **Opcode 26 field label is "runanim code 26" but values are all
   623→   623→   crawl-related.** Treated as a spec typo — opcode 26 is decoded as a
   624→   624→   crawl-anim family extension (5 shorts: crawl, crawlL, crawlR,
   625→   625→   crawlB, crawlBack2). Overwrites the 4 crawl fields from opcode 25
   626→   626→   (last write wins) and additionally populates `crawlanim_b2`.
   627→   627→
   628→   628→3. **Opcode 17 byte order: spec vs `NpcType.ts`.** The task spec says
   629→   629→   code 17 reads (walk, back, left, right). The existing 377
   630→   630→   `NpcType.ts` reads (walk, back, right, left) — left/right swapped.
   631→   631→   The Java client NpcType.java reads in obfuscated order with unclear
   632→   632→   semantics. LostCity's own unpack tool (`unpackNpcConfig`) reads in
   633→   633→   the SPEC order (l before r), so two of three LostCity sources agree
   634→   634→   with the spec. This decoder follows the spec (l before r) for code
   635→   635→   17. Documented as a known discrepancy with `NpcType.ts`; if
   636→   636→   LostCity ever regenerates its 377 npc.dat from RL sources, the 377
   637→   637→   `NpcType.ts` should be updated to match.
   638→   638→
   639→   639→4. **Opcode 28 vs 29 (texture recolors).** Spec describes 28 as
   640→   640→   "recol_d_texture (dst)" and 29 as "recol_s/d texture (paired with
   641→   641→   28)" — both with shape "g1 count, g2×2 per pair". The semantic
   642→   642→   distinction between 28 and 29 is unclear from the spec; both are
   643→   643→   decoded identically and stored in the same `recol_s_tex` /
   644→   644→   `recol_d_tex` parallel arrays (last opcode wins, matching how
   645→   645→   legacy opcode 40 behaves on repeat). If a future cache distinguishes
   646→   646→   them semantically, the single `decodeTexRecolors()` helper is the
   647→   647→   one place to split the storage.
   648→   648→
   649→   649→5. **`desc` field uses `gjstr()`, not `gjstrraw()`.** The task spec
   650→   650→   says code 3 reads `gjstrraw`. The Java client's `gjstrraw()`
   651→   651→   returns `byte[]` (raw bytes). The existing 377 `NpcType.ts` uses
   652→   652→   `gjstr()` (which returns a string) and stores `desc` as a string.
   653→   653→   `Packet.ts` does NOT have a `gjstrraw()` method. To match the
   654→   654→   existing TS convention (and avoid touching `Packet.ts`), this
   655→   655→   decoder uses `gjstr()`. If raw-bytes round-trip fidelity becomes
   656→   656→   important, a future task should add `Packet.gjstrraw()` and store
   657→   657→   `desc` as `Uint8Array`.
   658→   658→
   659→   659→6. **Opcode 101 contrast scaling.** The Java client reads `g1b() * 5`
   660→   660→   for the effective contrast value. The existing 377 `NpcType.ts`
   661→   661→   stores the raw `g1b()` byte (no scaling). This decoder matches the
   662→   662→   377 TS convention (store raw byte; consumer scales). Documented in
   663→   663→   a code comment.
   664→   664→
   665→   665→7. **Opcode 249 (params) inherited unchanged.** Uses
   666→   666→   `ParamHelper.decodeParams(dat)` from the existing engine — same
   667→   667→   path as 377 NpcType. No OSRS-specific changes needed.
   668→   668→
   669→   669→### `extractDependencyRefs()` — what it returns

   777→   777→### Summary
   778→   778→Implements a depth-first dependency walker that records an OSRS NPC's full
   779→   779→transitive dependency graph as a `DepsMap` (JSON-serializable). The walker
   780→   780→handles every node kind listed in the task spec (`npc`, `model`, `anim`,
   781→   781→`anim-base`, `seq`, `obj`, `param`, `struct`, `script`, `texture`,
   782→   782→`particle`, `sound`), with per-kind `visitXxx` methods that decode the
   783→   783→asset via a `CacheReader` interface (abstract — backed by either a real
   784→   784→OSRS cache or a test stub) and recurse into child refs.
   785→   785→
   786→   786→Cycle detection: per-branch `path: Set<string>` of node keys currently
   787→   787→being walked. Re-entering a key records a `DepCycle` and skips recursion —
   788→   788→guarantees termination on pathological param cross-reference loops.
   789→   789→
   790→   790→Memoization: visited nodes are cached for the lifetime of the tracer
   791→   791→instance, so `traceMany()` over multiple NPCs shares work across NPCs
   792→   792→that depend on the same model/seq/etc.

   881→   881→**Task 7 implementation notes:**
   882→   882→1. The OSRS cache stores skeletons (AnimBase) in a separate index from
   883→   883→   frames. The reader MUST load bases into `AnimBase.instances[]` BEFORE
   884→   884→   returning frames that reference them — `OsrsAnimFrame.decode()`
   885→   885→   requires the base to be present (see task 5-b worklog entry).
   886→   886→2. Every method MUST return `null` on missing IDs — do NOT throw. The
   887→   887→   tracer records null returns as `missing: true` dep refs.
   888→   888→3. `getName` is optional but recommended — without it, `DepNode.name`
   889→   889→   will be `null` for every node except NPCs (which carry their own
   890→   890→   `name` field on `OsrsNpcType`) and seqs (which carry `debugname`).
   891→   891→4. For `readParam`, prefer consulting an OSRS-side param registry if
   892→   892→   one is built. Falling back to the legacy 377 `ParamType.get(id)`
   893→   893→   works for shared params but will miss OSRS-native ones.

   895→   895→### Key design decisions
   896→   896→
   897→   897→1. **`visitNode(kind, id, path)` is PUBLIC**, not private as the task
   898→   898→   spec hinted. Rationale: the standalone `walkParams` helper needs to
   899→   899→   call it (the spec explicitly specified `walkParams(params, tracer,
   900→   900→   deps)` as a standalone function taking the tracer as an argument),
   901→   901→   and future per-kind walkers (e.g. an `OsrsObjType` walker added by
   902→   902→   task 8) should hook into the same cycle-detection + memoization
   903→   903→   logic without duplicating it. The state it touches (`nodes`,
   904→   904→   `cycles`, `missing`) is still encapsulated as private fields on
   905→   905→   the tracer instance.
   906→   906→
   907→   907→2. **`walkParams` is BOTH a public method on the tracer (`tracer.walkParams(params, deps, path)`)
   908→   908→   AND a standalone exported function.** The standalone wrapper exists
   909→   909→   for parity with the task spec's `walkParams(params, tracer, deps)`
   910→   910→   signature; it adds a required `path: Set<string>` argument (not in
   911→   911→   the spec) because cycle detection depends on it. Callers should
   912→   912→   prefer the method form.
   913→   913→
   914→   914→3. **`id` is `number | string` throughout the schema.** Numeric for
   915→   915→   cache-indexed assets (npc/model/anim/seq/obj/param/struct/texture/
   916→   916→   particle/sound); string for scripts (whose "id" is their debugname,
   917→   917→   e.g. `'dragon'`). This is necessary because LostCity runescript
   918→   918→   references assets by name (e.g. `npc_anim(dragon_attack, 0)`), and
   919→   919→   the script scanner can't resolve names to numeric cache IDs without
   920→   920→   the pack files (task 8's job). String-ID refs are recorded as
   921→   921→   `missing: true` until task 8 resolves them.
   922→   922→
   923→   923→4. **`transformedFrom` is always `null` in the tracer's output.** It
   924→   924→   will be populated by task 8 (content-folder writer) when it
   925→   925→   actually generates the 377-transformed twin of an OSRS node. The
   926→   926→   field exists in the schema now so task 8 doesn't need a schema
   927→   927→   migration.
   928→   928→
   929→   929→5. **`skins` flag on model nodes.** Per task 5-a's next-action hook,
   930→   930→   models with non-empty `triangleSkin`/`vertexSkin` arrays lose
   931→   931→   rig-skinning data when transformed to 377. The tracer records this
   932→   932→   as `skins: boolean` on model nodes so the pilot can prefer
   933→   933→   `skins === false` models when alternatives exist.
   934→   934→
   935→   935→6. **`secondaryRoots` for batched traces.** `traceMany()` keeps the
   936→   936→   single-`root` schema shape (backward compat with single-NPC
   937→   937→   traces) by making the first NPC the root and listing any
   938→   938→   additional NPCs in `secondaryRoots: DepRef[]`. The field is
   939→   939→   omitted for single-NPC traces (the common case).
   940→   940→
   941→   941→7. **`ObjType` deep walk is stubbed.** Per task spec: "for now, treat
   942→   942→   obj as a partial node and log 'TODO: deep obj walk not
   943→   943→   implemented'". The `visitObj` method records the obj as a leaf
   944→   944→   with `note: 'TODO: deep obj walk not implemented (OsrsObjType not
   945→   945→   ported)'`. When task 8 ports the obj decoder, this method should
   946→   946→   walk `obj.model`, `obj.manwear`, `obj.womanwear`, `obj.manhead`,
   947→   947→   `obj.womanhead`, `obj.countobj[]`, `obj.recol_s/d`, and
   948→   948→   `obj.params` (via `walkParams`).
   949→   949→
   950→   950→8. **`ParamType` lookup falls back to the legacy 377 registry.** When
   951→   951→   `CacheReader.readParam(id)` returns null (the common case until
   952→   952→   `OsrsParamType` is ported), the tracer tries `ParamType.get(id)`
   953→   953→   (the legacy 377 static registry). If both miss, the param is
   954→   954→   recorded with `note: 'param type unknown — value not walked
   955→   955→   (OsrsParamType not ported)'` and its value is NOT walked (the
   956→   956→   importer will bring the param's value verbatim, but the tracer
   957→   957→   can't tell what asset kind it points at).
   958→   958→
   959→   959→9. **Script scanning is best-effort regex.** LostCity runescript
   960→   960→   references assets by debugname in call sites like
   961→   961→   `npc_anim(dragon_attack, 0)`, `spotanim_npc(firebreath_attack, 92, 0)`,
   962→   962→   `inv_total(worn, antidragonbreathshield)`, `sound_synth(dragonbreath, 0, 30)`,
   963→   963→   `npc_param(attack_sound)`. The tracer scans for these patterns and
   964→   964→   records the discovered names as string-ID dep refs (all marked
   965→   965→   `missing: true` until task 8 resolves them via pack files). The
   966→   966→   scan is intentionally conservative — false positives would be
   967→   967→   worse than false negatives here (the importer would try to fetch
   968→   968→   non-existent assets).
   969→   969→
   970→   970→10. **Script discovery for an NPC uses TWO heuristics:**
   971→   971→    (a) look for `content/scripts/npc/scripts/<npcName>.rs2` (primary
   972→   972→    combat script convention); (b) scan all other `.rs2` files for
   973→   973→    `[<trigger>,<npcName>]` dispatch directives (e.g.
   974→   974→    `[ai_applayer2,babydragon] @dragon_ai_applayer2(false);`) and
   975→   975→    register each matching file as a separate script dep with
   976→   976→    `via: 'dispatch:<trigger>'`. The dispatch-scan is O(files × lines)
   977→   977→    but runs only once per NPC and only over the `scripts/npc/scripts/`
   978→   978→    subdir.

  1154→  1154→### JS5 archive → content mapping assumed
  1155→  1155→
  1156→  1156→The OSRS cache uses this layout (post-rev ~150 era, stable across modern
  1157→  1157→dumps). The reader hard-codes these constants in
  1158→  1158→`OsrsCacheAssetReader.ts`:
  1159→  1159→
  1160→  1160→| Archive | Contents | Used by |
  1161→  1161→|---------|----------|---------|
  1162→  1162→| 0       | Animations (frames AND skeletons, disjoint file IDs) | `readAnim` + `readAnimBase` |
  1163→  1163→| 1       | Models (one `.ob2` per file ID) | `readModel` |
  1164→  1164→| 2       | Music — older OSRS dumps use this for config (fallback) | `ensureConfigChildren` fallback |
  1165→  1165→| 5       | Textures (older) — fallback | `readTexture` fallback |
  1166→  1166→| 9       | Configs (multi-child bundle: `npc.dat`, `seq.dat`, etc.) | `readNpc`/`readSeq`/etc. (primary) |
  1167→  1167→| 11      | Textures (modern) | `readTexture` primary |
  1168→  1168→| 25      | Audio (synth sound effects) | `readSound` (stub — not yet read) |
  1169→  1169→

  1171→  1171→archive 9 by default, with a fallback to archive 2 for older cache
  1172→  1172→dumps. Child names are resolved via the master index's per-child name
  1173→  1173→hashes (using the same `genHash()` algorithm as `Jagfile.ts`). If the
  1174→  1174→master index doesn't have name hashes, a sequential fallback is used.
  1175→  1175→

  1177→  1177→

  1186→  1186→### Limitations / stubs
  1187→  1187→
  1188→  1188→1. **`readObj`** returns a STUB `ObjType(id)` with only the ID populated
  1189→  1189→   — the legacy `ObjType.decode()` calls `printFatalError` (which calls
  1190→  1190→   `process.exit(1)`) on unrecognized opcodes, and OSRS obj configs use
  1191→  1191→   opcodes the 377 decoder doesn't know. The dep tracer treats objs as
  1192→  1192→   leaves anyway (Task 6 worklog item 7). Swap to a real decode once
  1193→  1193→   `OsrsObjType` is ported (Task 8+).
  1194→  1194→
  1195→  1195→2. **`readParam` / `readStruct`** return null — the legacy decoders
  1196→  1196→   `throw` on unrecognized opcodes (OSRS uses new codes). The dep
  1197→  1197→   tracer falls back to the legacy 377 `ParamType.get(id)` /
  1198→  1198→   `StructType.get(id)` static registries when these return null,
  1199→  1199→   which is the documented behavior in the Task 6 worklog.
  1200→  1200→
  1201→  1201→3. **`readParticle` / `readSound`** return null — OSRS particle systems
  1202→  1202→   and sound formats are complex and undocumented. The dep tracer
  1203→  1203→   records these as missing leaves, which is fine for the pilot.
  1204→  1204→
  1205→  1205→4. **`readAnimBase`** assumes skeletons live in archive 0 (the same
  1206→  1206→   archive as anim frames). This is correct for modern OSRS but may
  1207→  1207→   differ in some cache revisions. If the base isn't found at
  1208→  1208→   archive 0 file `id`, the method returns null.
  1209→  1209→
  1210→  1210→5. **Config bundle child-name resolution** uses the master index's
  1211→  1211→   per-child name hashes. If the master index was written with
  1212→  1212→   `named = false`, falls back to sequential child-ID assignment
  1213→  1213→   based on the standard OSRS child ordering
  1214→  1214→   (`npc.dat`, `npc.idx`, `seq.dat`, `seq.idx`, ...). This is
  1215→  1215→   best-effort and may need adjustment for unusual cache dumps.
  1216→  1216→
  1217→  1217→6. **Master index reading** assumes the master index is stored as a
  1218→  1218→   single compressed container at archive 255, file 0 (read via the
  1219→  1219→   normal sector-walking protocol). The sector's archiveType byte is
  1220→  1220→   NOT validated (only `id` and `part` are), to remain compatible
  1221→  1221→   with both the `archive + 1` convention (used for normal archives)
  1222→  1222→   and the `archiveType = 255` convention (sometimes used for the
  1223→  1223→   master index). This matches OpenRS2's behavior.

  1384→  1384→### WriterOptions design
  1385→  1385→
  1386→  1386→- **`dryRun`** (default `false`): when true, the writer walks the deps map,
  1387→  1387→  allocates IDs in memory, and produces an `ImportResult` showing what
  1388→  1388→  WOULD be written — but does NOT touch the filesystem or any pack files.
  1389→  1389→  Every `WrittenEntry.path` is set to the sentinel string `'<dryrun>'`.
  1390→  1390→  Useful for previewing an import before committing it.
  1391→  1391→
  1392→  1392→- **`namePrefix`** (default `'osrs_'`): prepended to every sanitized
  1393→  1393→  debugname. Combined with the sanitized OSRS name/ID to form the final
  1394→  1394→  debugname (e.g. `osrs_tormented_demon`, `osrs_model_45000`).
  1395→  1395→
  1396→  1396→- **`groupFile`** (default `'osrs_imports'`): subfolder under
  1397→  1397→  `content/models/` where binary assets (.ob2, .anim, .base) are written,
  1398→  1398→  AND the basename of the `.npc` / `.seq` config files. Multiple NPCs
  1399→  1399→  imported in one batch share the same group file (their `[debugname]`
  1400→  1400→  blocks are appended to the same `.npc` / `.seq` file).
  1401→  1401→
  1402→  1402→- **`overwrite`** (default `false`): when false (the safe default), the
  1403→  1403→  writer NEVER overwrites existing files or pack entries — collisions
  1404→  1404→  trigger a fresh debugname with a `_2`, `_3` suffix instead. When true,
  1405→  1405→  the writer REPLACES existing `[debugname]` blocks in config files and
  1406→  1406→  overwrites binary files with the same path. NOT recommended for normal
  1407→  1407→  use — intended for re-importing after a sanitization rule change.
  1408→  1408→
  1409→  1409→### Sanitization rules
  1410→  1410→
  1411→  1411→The `sanitizeName(rawName, packType)` helper applies the following
  1412→  1412→transforms in order:
  1413→  1413→
  1414→  1414→1. **Lowercase** the entire string.
  1415→  1415→2. **Replace every char NOT in `[a-z0-9]` with `_`** — spaces, hyphens,
  1416→  1416→   apostrophes, etc. all become underscores.
  1417→  1417→3. **Collapse runs of multiple `_`** into a single `_` (so `"a  b"` →
  1418→  1418→   `"a_b"`, not `"a__b"`).
  1419→  1419→4. **Trim leading/trailing `_`** (so `"_foo_"` → `"foo"`).
  1420→  1420→5. **Prefix** with `namePrefix` (default `osrs_`).
  1421→  1421→6. **Dedupe** against the target pack file AND a session-reserved set
  1422→  1422→   (in-memory, populated by prior `sanitizeName()` calls in the same
  1423→  1423→   writer lifetime). If the candidate name is already taken, append
  1424→  1424→   `_2`, `_3`, `_4`, ... until unique.
  1425→  1425→
  1426→  1426→Examples:
  1427→  1427→- `"Tormented demon"` → `"osrs_tormented_demon"`
  1428→  1428→- `"td_stand"` → `"osrs_td_stand"`
  1429→  1429→- `"model_45000"` → `"osrs_model_45000"`
  1430→  1430→- `"anim_2005"` (after `osrs_anim_2005` already exists in `anim.pack`)
  1431→  1431→  → `"osrs_anim_2005_2"`
  1432→  1432→
  1433→  1433→The session-reserved set is critical for batch imports where two OSRS
  1434→  1434→assets sanitize to the same candidate name in the same run — without
  1435→  1435→it, the second asset would silently overwrite the first's pack entry.
  1436→  1436→
  1437→  1437→### Idempotency approach
  1438→  1438→
  1439→  1439→The writer is idempotent at three layers:
  1440→  1440→
  1441→  1441→1. **Deps-map layer** (primary): every OSRS `DepNode` has a
  1442→  1442→   `transformedFrom` field (set by Task 8 per the schema). On each run,
  1443→  1443→   the writer's `bootstrapResolverFromDepsMap()` walks the input deps
  1444→  1444→   map and pre-populates `NameResolver` from any node with
  1445→  1445→   `transformedFrom !== null` (looking up the debugname from the
  1446→  1446→   corresponding pack file). Then `processNode()` checks the field
  1447→  1447→   directly: if set, the node is recorded as `skipped` with reason
  1448→  1448→   `already_imported (newId=N)` and NOT re-processed.
  1449→  1449→
  1450→  1450→2. **NameResolver layer** (cross-NPC): if two NPCs in the same batch
  1451→  1451→   reference the same OSRS asset (e.g. both use OSRS model 45000), the
  1452→  1452→   first NPC's `processNode()` call registers the asset in
  1453→  1453→   `NameResolver`. When the second NPC's `processNode()` runs for the
  1454→  1454→   same OSRS ID, it finds the existing entry in NameResolver, records
  1455→  1455→   it as `skipped` with reason `shared_with_prior_npc`, and updates the
  1456→  1456→   deps map's `transformedFrom` field to point at the existing new ID.
  1457→  1457→
  1458→  1458→3. **Config-file layer** (defensive): `appendConfigBlock()` scans the
  1459→  1459→   target `.npc` / `.seq` config file for an existing `[<debugname>]`
  1460→  1460→   header before appending. If found and `overwrite=false`, the append
  1461→  1461→   is skipped (with a warning). If found and `overwrite=true`, the
  1462→  1462→   existing block is stripped via `stripConfigBlock()` before the new
  1463→  1463→   block is appended. This protects against the case where a prior run
  1464→  1464→   wrote the config file but failed before persisting the updated
  1465→  1465→   deps.json (so the next run's deps-map layer doesn't know about the
  1466→  1466→   prior import).
  1467→  1467→
  1468→  1468→The CLI (`Import.ts`) persists the updated deps.json (with
  1469→  1469→`transformedFrom` filled in) to `content/deps/<npcName>.deps.json`
  1470→  1470→after every successful non-dry-run import. Subsequent runs pick this
  1471→  1471→file up via `--deps=<path>` (or auto-trace + apply the persisted
  1472→  1472→`transformedFrom` from a prior run via the `--out=` round-trip).
  1473→  1473→
  1474→  1474→### Pack-file ID allocation strategy
  1475→  1475→
  1476→  1476→**Decision: `max(existing IDs) + 1` for every pack type.**
  1477→  1477→
  1478→  1478→The schema supports reserved ranges per pack type (e.g. "models
  1479→  1479→50000-59999 reserved for OSRS imports") but no ranges are reserved
  1480→  1480→today. The `allocateId(packType)` helper loads the pack file, finds
  1481→  1481→the maximum existing ID, and returns `max + 1` (or `0` for an empty
  1482→  1482→pack, matching LostCity's convention where every existing pack starts
  1483→  1483→at ID 0).
  1484→  1484→
  1485→  1485→Rationale:
  1486→  1486→- Simplest strategy that preserves backwards-compat (existing IDs are
  1487→  1487→  never reused or overwritten).
  1488→  1488→- The LostCity content folder doesn't have a "free ID range" convention
  1489→  1489→  — IDs are assigned sequentially as new assets are added.
  1490→  1490→- If a future task adds reserved ranges, `allocateId()` is the single
  1491→  1491→  function to change (per-pack-type strategy can be added without
  1492→  1492→  touching call sites).
  1493→  1493→
  1494→  1494→**Anims are registered in BOTH `anim.pack` AND `animset.pack`** (with
  1495→  1495→the same ID and name). This matches the existing LostCity convention
  1496→  1496→where both packs share the same `id=name` mapping for every anim file:
  1497→  1497→- `anim.pack` (AnimPack) is consulted by `SeqConfig.parseSeqConfig()`
  1498→  1498→  when resolving `frameN=<animname>` refs in `.seq` config files.
  1499→  1499→- `animset.pack` (AnimSetPack) is consulted by
  1500→  1500→  `engine/tools/pack/graphics/pack.ts` when mapping `.anim` files to
  1501→  1501→  archive 2 file IDs during cache packing.
  1502→  1502→
  1503→  1503→Registering in only one would break either the seq config parser or
  1504→  1504→the cache packer. The writer registers in both atomically (within the
  1505→  1505→same `updatePackFile()` call sequence, with both packs sharing the
  1506→  1506→same dirty/save batch).
  1507→  1507→
  1508→  1508→### NPC config line rewriting
  1509→  1509→
  1510→  1510→`OsrsNpcType.toLegacy377NpcConfig()` emits raw integer asset IDs in
  1511→  1511→its output (e.g. `model1=45000`, `readyanim=9200`,
  1512→  1512→`param=osrs_runanim,9102`). The writer's `rewriteNpcConfigLine()`
  1513→  1513→post-processes each line to rewrite those IDs to their newly-assigned
  1514→  1514→debugnames via `NameResolver`:
  1515→  1515→
  1516→  1516→| Line shape                              | Rewrite                                          |
  1517→  1517→|-----------------------------------------|--------------------------------------------------|
  1518→  1518→| `modelN=<id>`                           | `modelN=<modelDebugname>`                        |
  1519→  1519→| `headN=<id>`                            | `headN=<modelDebugname>`                         |
  1520→  1520→| `readyanim=<id>`                        | `readyanim=<seqDebugname>`                       |
  1521→  1521→| `walkanim=<id>[,<id>,<id>,<id>]`        | per-element rewrite to seq debugnames            |
  1522→  1522→| `param=osrs_runanim,<id>`               | `param=osrs_runanim,<seqDebugname>`              |
  1523→  1523→| `param=osrs_runanim_{l,r,b},<id>`       | `param=osrs_runanim_{l,r,b},<seqDebugname>`      |
  1524→  1524→| `param=osrs_crawlanim,<id>`             | `param=osrs_crawlanim,<seqDebugname>`            |
  1525→  1525→| `param=osrs_crawlanim_{l,r,b,b2},<id>`  | `param=osrs_crawlanim_{l,r,b,b2},<seqDebugname>` |
  1526→  1526→| `param=osrs_walkanim_b,<id>`            | `param=osrs_walkanim_b,<seqDebugname>`           |
  1527→  1527→
  1528→  1528→Lines whose OSRS ID can't be resolved via `NameResolver` (e.g. the
  1529→  1529→tracer didn't walk that model) fall back to the `model_<id>` /
  1530→  1530→`seq_<id>` / `anim_<id>` literal form — the LostCity config parser
  1531→  1531→accepts both forms (it tries `getByName(value)` first, falling back to
  1532→  1532→`parseInt(value)` if the name lookup returns -1).
  1533→  1533→
  1534→  1534→The header line (`[<osrs_debugname>]`) is also replaced with
  1535→  1535→`[<sanitized_debugname>]` so the pack entry matches the config block
  1536→  1536→header. (The OSRS emitter uses the OSRS debugname or `npc_<id>` as
  1537→  1537→the header — we override it with our `osrs_<name>` sanitized form.)
  1538→  1538→
  1539→  1539→### Seq config emission
  1540→  1540→
  1541→  1541→`writeSeq()` emits an ini-style `[debugname]` block per seq, mirroring
  1542→  1542→the format of `content/scripts/areas/area_kalphite/configs/kalphite.seq`

  1544→  1544→
  1545→  1545→```

  1547→  1547→frame1=osrs_anim_2005
  1548→  1548→delay1=2
  1549→  1549→param=osrs_replayFrameDelay,5
  1550→  1550→
  1551→  1551→```
  1552→  1552→
  1553→  1553→The 377 shared fields (loops, walkmerge, stretches, priority,
  1554→  1554→replaceheldleft, replaceheldright, maxloops, preanim_move,
  1555→  1555→postanim_move, duplicatebehavior) are emitted using the same key names
  1556→  1556→as `SeqConfig.parseSeqConfig()` expects. OSRS-only fields (field12,

  1558→  1558→`param=osrs_<field>,<value>` entries, matching the convention used by

  1698→  1698→### Resolution precedence (the 4-step rule)
  1699→  1699→Applied top-to-bottom (first hit wins):
  1700→  1700→
  1701→  1701→1. **Per-NPC player override** (`state.npcOverrides.get(npcId)`)
  1702→  1702→   - Honored UNCONDITIONALLY — the resolver does NOT check whether the chosen variant is actually available. The rationale: the override represents the player's explicit preference; `resolveNpcConfigForPlayer` on the registry is responsible for gracefully falling back to whatever's actually loadable (e.g. a player who forced `'osrs'` on an NPC with no OSRS import will see the legacy model via the fallback, but the resolver still reports `'osrs'` so the UI can flag the mismatch).
  1703→  1703→
  1704→  1704→2. **Per-region player override** (`state.regionOverrides.get(regionId)`)
  1705→  1705→   - STUB: today the registry always passes `regionId = -1` (which never matches any entry), so this step is effectively a no-op. The spawn-side wiring that would populate "current player region" is deferred to a later task. The `resolveNpcVariantForRegion(npcId, playerId, regionId)` method exists on the registry so the spawn-site migration is a one-line change once the region lookup is wired.
  1706→  1706→
  1707→  1707→3. **Player era preset** (`state.eraPreset`)
  1708→  1708→   - Respects availability (unlike steps 1 and 2). The `resolveEraPreset` helper translates:
  1709→  1709→     - `'05era'`   → `'legacy377'` (or `'osrs'` if the NPC has no legacy twin — i.e. brand-new OSRS content).
  1710→  1710→     - `'07era'`   → `'osrs'` if available, else `'legacy377'`. (Today equivalent to `'allOSRS'` because we don't track per-NPC "year added" — see the `EraPreset` docstring in `PlayerVariantState.ts`.)
  1711→  1711→     - `'allOSRS'` → `'osrs'` if available, else `'legacy377'`.
  1712→  1712→     - `'mixed'`   → same as `'05era'` (the override map is checked first; this is the fallback).
  1713→  1713→
  1714→  1714→4. **Server default** (`'05era'` → `'legacy377'`)
  1715→  1715→   - Defensive fallback — should never be reached because `PlayerVariantState` always has a valid `eraPreset`.
  1716→  1716→
  1717→  1717→**Backwards-compat guarantee:** a freshly-created state (default `eraPreset = '05era'`, empty override maps) selects `'legacy377'` for every NPC that has a legacy twin — i.e. the existing `NpcType.get(id)` path is preserved unchanged for any NPC that hasn't been imported. Only NPCs that were imported from OSRS via Task 8 (and registered in `content/deps/variants.json`) are eligible for the `'osrs'` variant — and only for players who explicitly chose an OSRS-era preset or set a per-NPC `'osrs'` override.

  1743→  1743→### variants.json schema (the on-disk index)
  1744→  1744→```json
  1745→  1745→{
  1746→  1746→  "version": 1,
  1747→  1747→  "variants": [
  1748→  1748→    {
  1749→  1749→      "legacyNpcId":     number,         // -1 if brand-new OSRS NPC (no legacy twin)
  1750→  1750→      "osrsNpcId":       number,         // the new 377 pack-file ID assigned by Task 8
  1751→  1751→      "osrsDebugname":   string,         // sanitized debugname (with `osrs_` prefix)
  1752→  1752→      "legacyDebugname": string | null,  // null if legacyNpcId === -1
  1753→  1753→      "depMapPath":      string,         // relative path: content/deps/<name>.deps.json
  1754→  1754→      "importedAt":      string          // ISO timestamp
  1755→  1755→    }
  1756→  1756→  ]
  1757→  1757→}
  1758→  1758→```
  1759→  1759→
  1760→  1760→For NPCs that have NO legacy equivalent (e.g. Tormented Demon), `legacyNpcId === -1` and `legacyDebugname === null`. `getAvailableVariants(osrsNpcId)` returns `['osrs']` only — the player's era preset and any `'legacy377'` override are honored unconditionally per the precedence rule, but `resolveNpcConfigForPlayer` falls back to the imported NpcType since there's no legacy to load.
  1761→  1761→
  1762→  1762→For NPCs with a legacy twin (e.g. Kalphite Queen, which exists in both 377 and OSRS), `legacyNpcId` is the existing 377 ID. `getAvailableVariants(legacyNpcId)` returns `['legacy377', 'osrs']` — the player's choice applies.
  1763→  1763→
  1764→  1764→### Integration pattern with existing NpcType
  1765→  1765→
  1766→  1766→The variant registry is NOT a wrapper around `NpcType`. It's a side-channel: when an NPC is being spawned for a specific player, the spawn code calls `VariantRegistry.resolveNpcVariant(npcId, playerId)` to decide which config ID to use. The two paths are:
  1767→  1767→
  1768→  1768→- `'legacy377'` → use the existing `NpcType.get(npcId)` path UNCHANGED. This is the default for every NPC that hasn't been imported.
  1769→  1769→- `'osrs'`       → use `NpcType.get(osrsImportedNpcId)` — the imported NPC's new 377 pack-file ID. The imported NPC config has its own `models[]` array pointing at the OSRS-imported models' new 377 IDs — so no per-model swap is needed at the spawn site.
  1770→  1770→
  1771→  1771→A single helper does the lookup:
  1772→  1772→```ts
  1773→  1773→VariantRegistry.getInstance().resolveNpcConfigForPlayer(npcId, playerId): NpcType
  1774→  1774→```
  1775→  1775→
  1776→  1776→Returns the legacy NpcType (`NpcType.get(npcId)`) or the OSRS-imported NpcType (`NpcType.get(osrsImportedNpcId)`). Future engine work can migrate NPC spawn sites one at a time to call this helper instead of `NpcType.get(npcId)` directly. Until a spawn site is migrated, the existing `NpcType.get(npcId)` path continues to work unchanged — the variant system is invisible to it. The registry does NOT modify `NpcType.configs[]` or any other existing static registry; it only consults them (for the `legacyModelOwner` reverse index used by `resolveModelVariant`).
  1777→  1777→
  1778→  1778→The spawn-site migration is intentionally deferred — touching every `NpcType.get(...)` call in the codebase (7 files: `World.ts`, `entity/Npc.ts`, `entity/Player.ts`, `GameMap.ts`, `script/ScriptIterators.ts`, `script/handlers/NpcOps.ts`, `script/handlers/PlayerOps.ts`) is too invasive for one task. The helper is in place; the migration can happen one spawn site at a time.
  1779→  1779→
  1780→  1780→### Per-player state caching
  1781→  1781→
  1782→  1782→`resolveNpcVariant(npcId, playerId)` is SYNC (the spawn path is hot). The per-player `PlayerVariantState` is loaded asynchronously by `VariantPersistence.loadPlayerVariantState(playerId)` at login time and stashed in the registry's `playerStateCache` Map. The login handler is expected to call:
  1783→  1783→```ts
  1784→  1784→const state = await loadPlayerVariantState(playerId);
  1785→  1785→VariantRegistry.getInstance().setPlayerState(playerId, state);
  1786→  1786→```
  1787→  1787→On logout: `VariantRegistry.getInstance().clearPlayerState(playerId)`.
  1788→  1788→
  1789→  1789→If `resolveNpcVariant` is called for a playerId that's not in the cache (e.g. before the login handler has warmed it), the registry returns the server default (`'legacy377'` for everything with a legacy twin, `'osrs'` for brand-new OSRS NPCs). A warning is logged so the developer knows the cache wasn't warmed — but the engine doesn't crash.
