    32→### Engine side (TypeScript, `engine-ts`)
    33→- **Cache reader**: `engine/src/io/FileStream.ts` — 255-archive file store

    68→### Format diff: 377 vs OSRS (key for the port)
    69→The OSRS model format is a **strict superset** of the 377 format:
    70→- Header trailer is 18 bytes in 377, **23 bytes** in OSRS (extra 5 bytes:
    71→  `version`, `textureUVCount`, `triangleSkinCount`, `vertexSkinCount`,
    72→  `maxDepth`).
    73→- OSRS adds: per-vertex skin data, per-triangle skin data, textured UV coords,
    74→  `version` byte (used as a feature gate).
    75→- Old models decode fine under OSRS rules by treating the new fields as
    76→  zero-length when their flag bytes are 0.
    77→- Animations changed more substantially: OSRS anim frames use a different
    78→  layout (no shared `head` block — each frame is self-contained).
    79→
    80→### Pack-list format (hand-maintained today)
    81→Plain text, one entry per line: `0=hans\n1=man\n...`. Generated/validated by
    82→`PackFile.save()`. We will extend, not replace.

   132→## Task 5-a — Port OSRS ModelDefinition decoder to TS
   133→**Agent:** OsrsModelPorter
   134→**Status:** done
   135→**Files created** (both under `engine/src/cache/osrs/`):

   138→
   139→### Summary
   140→Ported RuneLite's `ModelDefinition.java` to TypeScript as `OsrsModel`, following
   141→the LostCity conventions used by the existing 377 decoder at
   142→`engine/src/cache/graphics/Model.ts`. Output uses `Packet` from `#/io/Packet.js`
   143→for all byte reading, `printWarning` from `#/util/Logger.js` for warnings, and
   144→the same 4-space / single-quote / semi style as the rest of the engine.
   145→
   146→### Public API
   147→```ts
   148→class OsrsModel {
   149→  static decode(id: number, data: Uint8Array | null): OsrsModel | null;
   150→  static detect(data: Uint8Array): 'osrs' | 'legacy377';
   151→  toLegacy377(): Uint8Array;
   152→  // plus public decoded fields (typed arrays) and header counts
   153→}
   154→
   155→class OsrsModelMetadata {
   156→  // all trailer fields + computed section offsets
   157→}
   158→```
   159→
   160→The class is **API-compatible in spirit** with the existing 377 `Model` class
   161→(sibling `Metadata` class + static `decode()`), but the OSRS decoder is
   162→one-shot (no separate `unpack`/`fromId` split) — it is intended for pipeline
   163→use (decode → transform → pack), not for runtime client caching. The 377
   164→decoder's two-stage split was an optimisation for the live game server; the
   165→OSRS pipeline doesn't need it.
   166→
   167→### Spec ambiguities resolved
   168→
   169→1. **Trailer byte count.** The task spec listed 12 trailer fields (17 bytes)
   170→   but asserted the trailer is 23 bytes. Resolved by combining the spec list
   171→   with the worklog note ("extra 5 bytes: `version`, `textureUVCount`,
   172→   `triangleSkinCount`, `vertexSkinCount`, `maxDepth`") and keeping every 377
   173→   field (including `priority`, which the spec omitted). Final layout = 17
   174→   fields, 23 bytes total. The 5 NEW bytes are: `version`,
   175→   `hasFaceTextures`, `triangleSkinCount`, `vertexSkinCount`, `maxDepth`
   176→   (all u8). `textureUVCount` from the worklog is interpreted as the same
   177→   field as `triangleTextureCount` (the count of textured triangles that
   178→   carry UVs).
   179→
   180→2. **Triangle orientation/type byte.** The spec lists section 11
   181→   ("triangle vertex indices") as using the "same 1/2/3/4 type encoding as
   182→   377" but doesn't list a separate orientation section. In 377 the
   183→   orientation byte lives in its own 1-byte-per-face section read BEFORE
   184→   the index deltas. Resolved by adding a triangle-orientation section
   185→   (1 byte per triangle) at body position 5.5 (between vertex labels and
   186→   triangle colours) — the same role as 377's `faceOrientationsOffset`.
   187→   The body section order comment in the source makes this explicit.
   188→
   189→3. **`priority` byte.** 377 has a `priority` byte (255 = per-face priority
   190→   array; otherwise shared priority value). The OSRS spec body section
   191→   list has no priority section, but dropping the byte entirely would lose
   192→   information needed for the 377 backward-compat path. Kept the byte in
   193→   the OSRS trailer at the same trailer position as 377 (between
   194→   `hasFaceTextures` and `hasFaceAlphas`).
   195→
   196→4. **Skin section encoding.** Spec body sections 13/14 say "only if
   197→   `version >= 1` AND skinCount > 0" but don't say how each skin entry is
   198→   encoded. Chose u8 per entry (matching the modern RuneLite decoder).
   199→   If a future cache uses smart-encoded skin entries, this is the single
   200→   line to change.
   201→
   202→5. **Detect heuristic.** `detect()` sniffs by reading the version byte at
   203→   `data.length - 23`, then validates vertexCount/triangleCount/dataLen
   204→   fields against the blob size. Falls back to `'legacy377'` on any
   205→   sanity failure — the legacy decoder will then either succeed or report
   206→   its own error. The safe default prevents false-positive OSRS
   207→   classification of malformed 377 blobs.
   208→
   209→### `toLegacy377()` transform — limitations
   210→
   211→The transform emits a faithful 377 `.ob2` blob that the existing Java client
   212→can render. The following OSRS-only data is **dropped**:
   213→
   214→| OSRS field             | 377 equivalent                | Impact |
   215→|------------------------|-------------------------------|--------|
   216→| `triangleMaterial`     | none                          | Faces with material textures fall back to flat colour |
   217→| `triangleSkin`         | none                          | Rig-based triangle skinning lost |
   218→| `vertexSkin`           | none                          | Rig-based vertex skinning lost |
   219→| `maxDepth`             | none                          | Bounding-box hint lost (client recomputes) |
   220→
   221→The dependency tracer (task 6) MUST flag models that have non-zero
   222→`triangleSkin`/`vertexSkin` arrays, because animations targeting those rigs
   223→will not play correctly when the model is rendered as a 377 blob. The pilot
   224→NPC (Tormented Demon) selection should prefer models with `triangleSkinCount
   225→== 0 && vertexSkinCount == 0` if any are available.
   226→
   227→Round-trip verified by smoke test:
   228→  OSRS bytes → `OsrsModel.decode()` → `toLegacy377()` → `Model.unpack()`
   229→  → `Model.fromId()` — all vertex positions, triangle indices, colours,
   230→  infos, alphas, tags, vertex labels, and per-face/shared priorities
   231→  match exactly. Textured UV vertex indices also survive (377 has the
   232→  same field shape — 3 u16 vertex indices per textured triangle).
   233→

   240→### Local helper

   271→### Summary
   272→Ported RuneLite's `FrameDefinition.java` (anim frames) and the OSRS-extended
   273→`SequenceDefinition` opcode set to TypeScript, following the LostCity
   274→conventions used by the existing 377 decoders at
   275→`engine/src/cache/graphics/AnimFrame.ts` and
   276→`engine/src/cache/config/SeqType.ts`. Output uses `Packet` from

   287→### Public API
   288→```ts
   289→class OsrsAnimFrame {
   290→    // 377-shape fields (for polymorphism with AnimFrame):
   291→    delay: number;
   292→    base: number;
   293→    length: number;
   294→    groups: Int32Array;
   295→    x: Int32Array;
   296→    y: Int32Array;
   297→    z: Int32Array;
   298→
   299→    // OSRS-only extras:
   300→    frameId: number;           // assigned by caller before toLegacy377()
   301→    frameLength: number;       // original group count
   302→    flags: Int32Array;         // per-group typeFlag bytes
   303→    interpolationType: number; // 0 if absent
   304→    leftHandWeight: number;    // -1 if absent
   305→    rightHandWeight: number;   // -1 if absent
   306→

   308→    toLegacy377(): Uint8Array;
   309→}
   310→
   311→class OsrsSeqType extends ConfigType {
   312→    // 377-shape fields (mirror of SeqType):

   315→    postanim_move, duplicatebehavior, duration
   316→
   317→    // OSRS-only extras:
   318→    field12: number;           // code 12 (carried from Java client's field790)
   319→    leftHandItem: number;      // code 13
   320→    rightHandItem: number;     // code 14
   321→    replayFrameDelay: number;  // code 15
   322→
   323→    decode(code: number, dat: Packet): void;
   324→    postDecode(): void;
   325→    // Plus static configs[] / get / getId / getByName / count (mirror of SeqType)
   326→}
   327→```
   328→
   329→### Key API decisions
   330→1. **`OsrsAnimFrame` is a sibling class, not `extends AnimFrame`.**
   331→   The 377 `AnimFrame` is heavily tied to its 4-section `unpack()` static
   332→   loader and the `AnimFrame.instances[]` / `AnimFrame.order[]` registries
   333→   managed by `OnDemand.cache`. Subclassing would inherit those statics
   334→   verbatim and route OSRS frames into the live 377 registry — exactly
   335→   what the modular pipeline wants to avoid. A sibling class with matching
   336→   field shape gives polymorphism without unwanted registry coupling.
   337→   This mirrors the sibling-class pattern used by `OsrsModel` (task 5-a).
   338→
   339→2. **`OsrsSeqType` is a sibling class extending `ConfigType` directly**, not
   340→   `extends SeqType`. Same reasoning as above — `SeqType` has private
   341→   static `configs[]` / `configNames` and bound `load/parse/get` methods
   342→   that would route OSRS seqs into the shared 377 registry.
   343→
   344→3. **`OsrsAnimFrame.decode()` takes `baseId` as a parameter** even though
   345→   the trailing 2 bytes of the blob also encode it. We cross-check and
   346→   warn on mismatch but trust the parameter — the caller is expected to
   347→   have resolved the base ID through the cache index, and the trailing
   348→   bytes are a redundant integrity check (sometimes zero in OSRS caches
   349→   that store the base reference in a separate index file).
   350→
   351→4. **`frameId` defaults to 0** in `toLegacy377()` because the OSRS blob
   352→   doesn't carry a frame ID. The modular packer is expected to assign a
   353→   real ID (and rewrite it into the head section if needed) before the
   354→   blob reaches the live client.
   355→
   356→5. **`OsrsSeqType.decode()` throws on codes 16+** (sub-frame references).
   357→   The task spec describes them only as "16+ for sub-frame references"
   358→   without a layout. Throwing lets the dep tracer (task 6) flag any seq
   359→   that uses them, rather than silently dropping data.
   360→
   361→### Format ambiguities resolved
   362→
   363→1. **"For groups with no flag, the previous group's value carries forward
   364→   (or 0 for translate, 128 for scale — same default-value rule as 377)"**
   365→   — interpreted as: use the SAME default-value rule as the 377 decoder
   366→   (0 for OP_TRANSLATE/OP_ROTATE/OP_ALPHA, 128 for OP_SCALE). The
   367→   "carry forward" phrasing in the spec was a description of how the
   368→   default value doesn't change between adjacent same-bone-type groups,
   369→   not a request to track a per-axis running last-value. The synthetic
   370→   OP_BASE pull-forward walk from `AnimFrame.unpack()` is preserved
   371→   unchanged — it emits a zero-value entry for the most recent OP_BASE
   372→   bone when a non-OP_BASE bone has a non-zero flag.
   373→
   374→2. **OSRS body layout is INTERLEAVED, not sectioned.** The 377 decoder
   375→   reads flags from a separate `tran1` section and smart values from a
   376→   separate `tran2` section. The OSRS spec describes "For each group i:
   377→   read typeFlag, then read axis values for set bits" — i.e., the flag
   378→   byte and the smart values are read sequentially from a single
   379→   section. This matches RuneLite's `FrameDefinition` decoder. Verified
   380→   by round-trip smoke test below.
   381→
   382→3. **Optional interpolation tail.** RuneLite's `FrameDefinition` reads
   383→   (after the delay byte) an optional `interpolationType: u8`, then
   384→   (if 4 more bytes are present) `leftHandItem: u16` and
   385→   `rightHandItem: u16`. The task spec calls these `leftHandWeight` /
   386→   `rightHandWeight` — I followed the spec's naming but kept RuneLite's
   387→   read layout (1 byte if any remain, then 4 bytes if at least 5 remain
   388→   total). If a future cache uses a different layout, this is the
   389→   single block to revisit.
   390→
   391→4. **Opcode 12 (`field790` in the Java client).** The 377 TS
   392→   `SeqType.decode()` does NOT handle code 12 (it throws). The Java
   393→   client does, reading a `u32`. The OSRS spec doesn't mention code 12
   394→   explicitly, but since OSRS is a superset of the 377 client opcode
   395→   set, code 12 should be handled. I added it as `field12: u32` with a
   396→   comment documenting it as an opaque field carried over from the
   397→   Java client (`field790`) — its semantic meaning isn't documented
   398→   upstream.
   399→
   400→5. **`AnimBase.instances[]` is reused.** The OSRS skeleton format is
   401→   byte-identical to 377 for pre-HD content (same `length u8`, `types
   402→   u8[]`, `labels u8[][]` layout). Rather than duplicate the loader,
   403→   `OsrsAnimFrame` references the existing `AnimBase.instances[]`
   404→   registry. OSRS-only pipelines that don't load 377 anims must still
   405→   call `AnimBase.unpack()` on OSRS base files to populate this
   406→   registry before invoking `OsrsAnimFrame.decode()`. (This is the
   407→   task 6 dep tracer's responsibility.)
   408→
   409→### `toLegacy377()` transform — limitations
   410→
   411→The transform emits a faithful 1-frame 377 anim blob that the existing
   412→377 client can render via `AnimFrame.unpack()`. The following OSRS-only
   413→data is **dropped**:
   414→
   415→| OSRS field             | 377 equivalent                | Impact |
   416→|------------------------|-------------------------------|--------|
   417→| `interpolationType`    | none                          | Blended/interpolated animations fall back to standard single-frame playback |
   418→| `leftHandWeight`       | none                          | Left-hand blend weight lost |
   419→| `rightHandWeight`      | none                          | Right-hand blend weight lost |
   420→
   421→The `frameId` field is REQUIRED by 377 (the head section stores it as a
   422→`u16`) but is NOT carried in the OSRS blob. `toLegacy377()` uses
   423→`this.frameId` (default 0). Callers that need a specific ID routed into
   424→`AnimFrame.instances[id]` must set `frame.frameId` before calling
   425→`toLegacy377()`.
   426→
   427→The skeleton (AnimBase) data is re-emitted byte-for-byte into the 377
   428→base-data section. Since the OSRS skeleton format is identical to 377
   429→for pre-HD content, this is lossless. The 377 `AnimBase.unpack()` will
   430→push a DUPLICATE base entry into `AnimBase.instances[]` when re-decoding
   431→the legacy blob — the dep tracer / packer should deduplicate by content
   432→hash if memory matters.

   505→### Summary
   506→Ported the OSRS NpcConfig opcode decoder to TypeScript as `OsrsNpcType`,
   507→following the LostCity conventions used by the existing 377 decoder at
   508→`engine/src/cache/config/NpcType.ts` and matching the sibling-class pattern
   509→established by tasks 5-a (`OsrsModel`) and 5-b (`OsrsAnimFrame` /
   510→`OsrsSeqType`).
   511→
   512→The OSRS NPC opcode set is a strict superset of 377: every 377 opcode
   513→decodes the same way, plus OSRS adds codes 23, 24, 25, 26, 27, 28, 29,
   514→118, 119, 121, 122, 123, 124, 125, 126, 127, 128, 129, and 130-134.
   515→This decoder handles BOTH sets so it can read an OSRS-format `npc.dat`
   516→directly. LostCity server-side codes 200-214 are read-and-discarded
   517→(with debug log) so this decoder can also parse a LostCity npc.dat
   518→without crashing.
   519→
   520→### Public API
   521→```ts
   522→class OsrsNpcType extends ConfigType {
   523→    // Static registry (mirror of OsrsSeqType's pattern):
   524→    static configNames: Map<string, number>;
   525→    static configs: OsrsNpcType[];
   526→    static get(id): OsrsNpcType;
   527→    static getId(name): number;
   528→    static getByName(name): OsrsNpcType | null;
   529→    static get count(): number;
   530→
   531→    // 377 shared fields (mirror of NpcType.ts):
   532→    name, desc, size, models, heads, hasanim, readyanim, walkanim,
   533→    walkanim_b, walkanim_r, walkanim_l, hasalpha, recol_s, recol_d,
   534→    op (size-10 lazy array), resizex, resizey, resizez, minimap,
   535→    vislevel, resizeh, resizev, alwaysontop, ambient, contrast,
   536→    headicon, turnspeed, multivarbit, multivarp, multinpc, active,
   537→    category, stats[6], params
   538→
   539→    // OSRS-only fields:
   540→    runanim, runanim_l, runanim_r, runanim_b           // code 24
   541→    crawlanim, crawlanim_l, crawlanim_r, crawlanim_b, // codes 25/26
   542→        crawlanim_b2                                    // (5th short of code 26)
   543→    recol_s_tex, recol_d_tex                           // codes 28/29
   544→    sizeH, sizeW                                       // codes 118/119
   545→    mapFunction, mapScene                              // code 121
   546→    heightmap                                          // code 122
   547→    hoverText                                          // code 123
   548→    flipX, flipY                                       // codes 124/125
   549→    forceRenderPriority, clipOnMinimap, clip, interactable  // codes 126-129
   550→    osrsExtensions: Int8Array(5)                       // codes 130-134
   551→
   552→    // Methods:
   553→    decode(code: number, dat: Packet): void;
   554→    toLegacy377NpcConfig(): string[];
   555→    extractDependencyRefs(): { models: number[]; heads: number[]; anims: number[]; params: number[] };
   556→}
   557→```
   558→
   559→### Key API decisions
   560→1. **`OsrsNpcType` extends `ConfigType` directly, NOT `NpcType`.** Same
   561→   reasoning as task 5-b's `OsrsSeqType`: subclassing `NpcType` would
   562→   inherit its private static `configs[]` / `configNames` and bound
   563→   `load/parse/get` methods, routing OSRS NPC configs into the shared
   564→   377 registry. A sibling class with a parallel registry keeps OSRS
   565→   configs isolated so the modular variant selector (task 9) can pick
   566→   which one to serve.
   567→
   568→2. **`op` array is lazily sized up to 10 entries.** Legacy opcodes 30-34
   569→   populate `op[0..4]` (op1..op5). OSRS opcode 27 reads a u8 index and
   570→   can populate any index (typically 5..9 for op6..op10). The array is
   571→   allocated on first use and grown dynamically if opcode 27's index
   572→   exceeds the current length. This avoids pre-allocating for the
   573→   common case (5 ops) while still supporting the extended set.
   574→
   575→3. **`toLegacy377NpcConfig()` emits raw asset IDs, NOT pack-file
   576→   names.** This deliberately decouples the engine class from
   577→   `#tools/pack/PackFile.js` lookups (SeqPack, NpcPack, ModelPack) —
   578→   mirroring the existing `NpcType` class which also doesn't import
   579→   PackFile. Task 8 (content-folder writer) is expected to post-process
   580→   the emitted lines to rename IDs to debugnames via the pack files.
   581→   The format `key=<int>` survives the round-trip into the content
   582→   folder because LostCity's config parser accepts both names and
   583→   integer literals.
   584→
   585→4. **OSRS-only fields are emitted as `param=osrs_<field>,<value>`
   586→   entries.** This preserves them through the content-folder round-trip
   587→   even though the 377 client doesn't understand them. When task 9
   588→   (variant registry) registers these as well-known param IDs, the
   589→   re-packing path will preserve them as opaque param values.
   590→
   591→5. **LostCity server-side codes 200-214 are read-and-discarded, not
   592→   stored.** The spec describes these as "IGNORE them (skip silently,
   593→   log debug)" — interpreted as: read the payload (using the known
   594→   format from `NpcType.ts`) and discard the values, so this decoder
   595→   can parse a LostCity npc.dat without crashing. The discarded values
   596→   are NOT stored on this class because OSRS-format NPCs don't carry
   597→   these fields; the modular pipeline is expected to layer server-side
   598→   config on top via the content folder rather than the cache.
   599→
   600→6. **`extractDependencyRefs()` excludes `multinpc[]`.** Multinpc
   601→   references other NPC config IDs, but those are walked via the NPC
   602→   config index, not via archive IDs (1/2 for models/anims). Including
   603→   them in the returned `params`/`anims` arrays would mix asset kinds.
   604→   Task 6 (dep tracer) is expected to walk multinpc separately by
   605→   recursing into the NPC config index.
   606→
   607→7. **`osrsExtensions` is a fixed 5-slot `Int8Array` indexed by
   608→   `code - 130`.** Codes 130-134 are described in the spec as "store
   609→   opaquely" with no semantic meaning documented upstream. A fixed
   610→   array is simpler than a `Map<number, number>` and gives O(1) access
   611→   for consumers that later want to interpret specific slots.
   612→
   613→### Opcode ambiguities resolved
   614→
   615→1. **Opcode 23 reads 5 shorts, NOT 6.** The task spec listed
   616→   "(stand, walk, walkL, walkR, walkB, run)" — 6 names — but said
   617→   "g2×5". The trailing "run" is a spec typo; standard RuneLite reads
   618→   exactly 5 shorts (stand, walk, walkLeft, walkRight, walkBack).
   619→   Decoded as: readyanim, walkanim, walkanim_l, walkanim_r, walkanim_b
   620→   (overwriting the same fields populated by 377 codes 13/14/17).
   621→
   622→2. **Opcode 26 field label is "runanim code 26" but values are all
   623→   crawl-related.** Treated as a spec typo — opcode 26 is decoded as a
   624→   crawl-anim family extension (5 shorts: crawl, crawlL, crawlR,
   625→   crawlB, crawlBack2). Overwrites the 4 crawl fields from opcode 25
   626→   (last write wins) and additionally populates `crawlanim_b2`.
   627→
   628→3. **Opcode 17 byte order: spec vs `NpcType.ts`.** The task spec says
   629→   code 17 reads (walk, back, left, right). The existing 377
   630→   `NpcType.ts` reads (walk, back, right, left) — left/right swapped.
   631→   The Java client NpcType.java reads in obfuscated order with unclear
   632→   semantics. LostCity's own unpack tool (`unpackNpcConfig`) reads in
   633→   the SPEC order (l before r), so two of three LostCity sources agree
   634→   with the spec. This decoder follows the spec (l before r) for code
   635→   17. Documented as a known discrepancy with `NpcType.ts`; if
   636→   LostCity ever regenerates its 377 npc.dat from RL sources, the 377
   637→   `NpcType.ts` should be updated to match.
   638→
   639→4. **Opcode 28 vs 29 (texture recolors).** Spec describes 28 as
   640→   "recol_d_texture (dst)" and 29 as "recol_s/d texture (paired with
   641→   28)" — both with shape "g1 count, g2×2 per pair". The semantic
   642→   distinction between 28 and 29 is unclear from the spec; both are
   643→   decoded identically and stored in the same `recol_s_tex` /
   644→   `recol_d_tex` parallel arrays (last opcode wins, matching how
   645→   legacy opcode 40 behaves on repeat). If a future cache distinguishes
   646→   them semantically, the single `decodeTexRecolors()` helper is the
   647→   one place to split the storage.
   648→
   649→5. **`desc` field uses `gjstr()`, not `gjstrraw()`.** The task spec
   650→   says code 3 reads `gjstrraw`. The Java client's `gjstrraw()`
   651→   returns `byte[]` (raw bytes). The existing 377 `NpcType.ts` uses
   652→   `gjstr()` (which returns a string) and stores `desc` as a string.
   653→   `Packet.ts` does NOT have a `gjstrraw()` method. To match the
   654→   existing TS convention (and avoid touching `Packet.ts`), this
   655→   decoder uses `gjstr()`. If raw-bytes round-trip fidelity becomes
   656→   important, a future task should add `Packet.gjstrraw()` and store
   657→   `desc` as `Uint8Array`.
   658→
   659→6. **Opcode 101 contrast scaling.** The Java client reads `g1b() * 5`
   660→   for the effective contrast value. The existing 377 `NpcType.ts`
   661→   stores the raw `g1b()` byte (no scaling). This decoder matches the
   662→   377 TS convention (store raw byte; consumer scales). Documented in
   663→   a code comment.
   664→
   665→7. **Opcode 249 (params) inherited unchanged.** Uses
   666→   `ParamHelper.decodeParams(dat)` from the existing engine — same
   667→   path as 377 NpcType. No OSRS-specific changes needed.
   668→
   669→### `extractDependencyRefs()` — what it returns

   674→    heads: number[];    // archive-1 IDs from `heads[]` (code 60)
   675→                        // — tracked separately for head-icon render path
